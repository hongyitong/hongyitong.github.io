Dive into Claude Code: The Design Space of Today’s and Future AI Agent Systems

深入解析 Claude Code：当下与未来 AI Agent 系统的设计空间

Claude Code is an agentic coding tool that can run shell commands, edit files, and call external services on behalf of the user.

Claude Code 是一款智能体（agentic）代码编写工具，能够代表用户运行 Shell 命令、编辑文件并调用外部服务。

This study describes its comprehensive architecture by analyzing the publicly available TypeScript source code and further comparing it with OpenClaw, an independent open-source AI agent system that answers many of the same design questions from a different deployment context.

本研究通过分析其公开的 TypeScript 源代码，详尽地阐述了其整体架构，并进一步将其与 OpenClaw 进行了对比。OpenClaw 是一个独立的开源 AI Agent 系统，它在不同的部署语境下回答了许多相同的设计问题。

Our analysis identifies five human values, philosophies, and needs that motivate the architecture (human decision authority, safety and security, reliable execution, capability amplification, and contextual adaptability) and traces them through thirteen design principles to specific implementation choices.

我们的分析确立了驱动该架构的五个人类价值、哲学与需求（人类决策自主权、安全与保障、可靠执行、能力放大以及上下文适应性），并通过 13 项设计原则将其追溯至具体的实现选择。

The core of the system is a simple while-loop that calls the model, runs tools, and repeats.

该系统的核心是一个简单的 while 循环，用于调用模型、运行工具并不断重复此过程。

Most of the code, however, lives in the systems around this loop: a permission system with seven modes and an ML-based classifier, a five-layer compaction pipeline for context management, four extensibility mechanisms (MCP, plugins, skills, and hooks), a subagent delegation and orchestration mechanism, and append-oriented session storage.

然而，绝大部分代码都存在于围绕该循环的外围系统中：一个包含七种模式且基于机器学习分类器的权限系统、一个用于上下文管理的五层压缩流水线、四种扩展机制（MCP、插件、技能和钩子）、一个子智能体（subagent）委派与编排机制，以及面向追加（append-oriented） expedition 的会话存储。

A comparison with OpenClaw, a multi-channel personal assistant gateway, shows that the same recurring design questions produce different architectural answers when the deployment context changes: from per-action safety evaluation to perimeter-level access control, from a single CLI loop to an embedded runtime within a gateway control plane, and from context-window extensions to gateway-wide capability registration.

与多通道个人助手网关 OpenClaw 的对比表明，当部署语境发生变化时，同样反复出现的设计问题会产生不同的架构答案：从针对单次动作的安全评估转变为边界级别的访问控制，从单一的 CLI 循环转变为网关控制平面的嵌入式运行时，以及从上下文窗口扩展转变为全网关范围的能力注册。

We finally identify six open design directions for future agent systems, grounded in recent empirical, architectural, and policy

我们最后结合近期的实证研究、架构演进和政策趋势，指出了未来智能体系统值得探索的六个开放性设计方向。

1 Introduction

1 引言

AI-assisted software development has evolved from autocomplete-style tools such as GitHub Copilot (Chen et al., 2021), through IDE-integrated assistants like Cursor (Cursor, 2026), to fully agentic systems that autonomously plan multi-step modifications, execute shell commands, read and write files, and iterate on their own outputs.

AI 辅助软件开发已经从 GitHub Copilot 等自动补全风格的工具（Chen et al., 2021），演进到像 Cursor 这样集成了 IDE 的助手（Cursor, 2026），再到如今能够自主规划多步修改、执行 Shell 命令、读写文件并对其自身输出进行迭代的完全智能体（agentic）系统。

Claude Code (Anthropic, 2026a) is an agentic coding tool released by Anthropic (Anthropic, 2026c). Its official documentation describes an “agentic loop” that plans and executes actions toward accomplishing a goal and can call tools, evaluate results, and continue until the task is done.

Claude Code（Anthropic, 2026a）是 Anthropic 公司发布的一款智能体编码工具（Anthropic, 2026c）。其官方文档描述了一个“智能体循环”（agentic loop），该循环围绕实现某一目标来规划并执行动作，能够调用工具、评估结果，并持续运行直到任务完成。

This shift from suggestion to autonomous action introduces architectural requirements that have no counterpart in completion-based tools. These requirements define a design space, a set of recurring questions spanning topics such as safety, context management, extensibility, and delegation that every coding agent must navigate.

这种从“提供建议”到“自主行动”的转变，引入了在基于补全的工具中不曾存在的架构需求。这些需求定义了一个设计空间，即每一个编码智能体都必须面对的一系列周期性问题，涵盖了安全性、上下文管理、可扩展性和委派等主题。

This study uses source-level analysis of Claude Code to show how one production system answers these questions. Despite growing adoption, Anthropic publishes user-facing documentation for Claude Code but not detailed architectural descriptions. This study uses source code analysis to describe architectural design decisions.

本研究通过对 Claude Code 的源码级分析，展示了一个生产级系统是如何回答这些问题的。尽管其采用率日益增长，但 Anthropic 仅发布了面向用户的 Claude Code 文档，并未提供详细的架构描述。本研究利用源代码分析来阐述其架构设计决策。

Anthropic’s internal survey of 132 engineers and researchers (Huang et al., 2025) reports that about 27% of Claude Code-assisted tasks were work that would not have been attempted without the tool, suggesting that the architecture enables qualitatively new workflows rather than merely accelerating existing ones.

Anthropic 对 132 名工程师和研究人员进行的内部调查（Huang et al., 2025）显示，在由 Claude Code 辅助完成的任务中，约有 27% 的工作如果离开了该工具根本不会被尝试。这表明该架构开启了性质全新的工作流，而不仅仅是加速了现有的工作流。

In this work, we first identify five human values/philosophies and thirteen design principles that motivate the architecture (Section 2), then organize the analysis in three parts:

在这项工作中，我们首先确立了驱动该架构的五个人类价值/哲学以及十三项设计原则（第 2 节），然后将分析分为三个部分：

  1. Design-space analysis. We identify recurring design questions (where reasoning lives, how the iteration loop is structured, what safety posture to adopt, how the extension surface is partitioned, how context is managed, how work is delegated across subagents, and how sessions persist) and analyze Claude Code’s answers through a 7-component high-level structure and a 5-layer subsystem architecture, tracing each choice to specific source files (Section 3). The analysis aims to build a deep understanding of the system mechanism, with the goal of informing the design of better and more powerful agent systems.

    设计空间分析。我们识别了反复出现的设计问题（推理存在于何处、迭代循环如何构建、采取何种安全姿态、扩展面如何划分、上下文如何管理、工作如何在子智能体之间委派，以及会话如何持久化），并通过一个包含 7 个组件的高层结构和一个 5 层的子系统架构来分析 Claude Code 的解答，同时将每个选择追溯到具体的源文件（第 3 节）。该分析旨在建立对系统机制的深入理解，以期为设计更好、更强大的智能体系统提供参考。

  2. Architectural contrast with OpenClaw. Beyond analyzing Claude Code itself, we also compare its design philosophy with that of open-source agent system OpenClaw (Steinberger and OpenClaw Contributors, 2026), a multi-channel personal assistant gateway, across six design dimensions to show how the same recurring questions produce different answers under different deployment contexts (Section 10), in order to highlight both the common principles and the key differences between commercial and open-source software. This comparison helps reveal how deployment setting, product goals, safety requirements, and user assumptions shape architectural choices in different ways. By examining where these systems converge and where they diverge, our study aims to provide useful guidance and practical insights for the design of future, more capable agent systems.

    与 OpenClaw 的架构对比。除了分析 Claude Code 本身之外，我们还在六个设计维度上，将其设计哲学与开源智能体系统 OpenClaw（Steinberger 及其开源贡献者, 2026，一个多通道个人助手网关）进行了对比，以展示在不同的部署语境下，相同的周期性问题如何产生不同的答案（第 10 节），从而突出商业软件与开源软件之间的共同原则和关键差异。这种对比有助于揭示部署环境、产品目标、安全需求和用户假设如何以不同方式塑造架构选择。通过检验这些系统的趋同点与分歧点，我们的研究旨在为未来更高效的智能体系统设计提供有用的指导和切实的见解。

  3. Open directions for future agent systems. Building on the design-space analysis and the OpenClaw contrast, Section 12 identifies six open directions spanning the observability-evaluation gap, cross-session persistence, harness boundary evolution, horizon scaling, governance, and the evaluative lens, each drawing on empirical, architectural, and policy literature. Used as an evaluative lens, our study also reveals an open question: while the Claude Code agent system substantially amplifies the short-term capabilities of programmers and end users, it offers limited mechanisms that explicitly support long-term human improvement, deeper understanding, and sustained codebase coherence.

    未来智能体系统的开放方向。基于设计空间分析以及与 OpenClaw 的对比，第 12 节指出了跨越可观测性与评估差距、跨会话持久化、治理边界演进、视野扩展、治理以及评估视角的六个开放方向，每个方向都借鉴了实证、架构和政策文献。作为一种评估视角，我们的研究还揭示了一个开放性问题：尽管 Claude Code 智能体系统显著放大了程序员和终端用户的短期能力，但它在显式支持人类的长期进步、加深理解以及维持代码库的持续一致性方面，提供的机制非常有限。

The core agent loop is a while-true cycle with state management. The surrounding subsystems for safety, extensibility, context management, delegation, and persistence make up the bulk of the implementation. Source-level analysis allows us to identify design choices, subsystem boundaries, and implementation trade-offs directly from the system itself rather than inferring them solely from product descriptions.

核心智能体循环是一个带有状态管理的 while-true 循环。 而围绕其展开的用于安全性、可扩展性、上下文管理、委派和持久化的子系统则构成了实现的主要部分。源码级的分析使我们能够直接从系统本身识别设计选择、子系统边界和实现权衡，而不是仅仅从产品描述中进行推断。

Running example. To keep the architecture concrete, we trace the task “Fix the failing test in auth.test.ts” through Sections 3 to 9. This example illustrates how a seemingly simple user request activates multiple architectural layers, including tool invocation, permission checks, context selection, iterative repair, delegation, and session persistence.

运行示例。为了使架构更具体，我们将在第 3 节至第 9 节中追踪“修复 auth.test.ts 中失败的测试”这一任务。该示例展示了一个看似简单的用户请求如何激活多个架构层，包括工具调用、权限检查、上下文选择、迭代修复、委派和会话持久化。

Sections 4 to 9 each analyze a major subsystem’s design choices. Section 10 contrasts the analysis with OpenClaw, Section 11 provides discussion, and Section 12 surveys open questions for future agent systems. Sections 13 and 14 then cover related work and conclusions. Section B describes the evidence base and methodology.

论文结构。第 2 节确立了驱动该架构的人类价值和设计原则。第 3 节介绍了高层架构及其回答的设计问题。第 4 节至第 9 节分别分析了各个主要子系统的设计选择。第 10 节将该分析与 OpenClaw 进行对比，第 11 节展开讨论，第 12 节概述了未来智能体系统的开放性问题。第 13 节和第 14 节随后介绍了相关工作和结论。附录 B 阐述了证据基础和研究方法。

2 Design Philosophies, Design Principles and Architectural Motivations

2 设计哲学、设计原则与架构动机

Production coding agents are built by humans, for humans, and the architectural decisions they embed reflect what their creators believe matters. This section identifies the human values that motivate Claude Code’s design, traces them through recurring design principles, and frames the design-space questions that organize the analysis in Sections 3 to 9.

生产级编码智能体是由人类构建并为人类服务的，它们所嵌入的架构决策反映了其创造者所认为重要的事情。本节确立了驱动 Claude Code 设计的人类价值，通过反复出现的设计原则对其进行追溯，并构建了组织第 3 节至第 9 节分析的设计空间问题。

Anthropic’s framework for safe agents states a central tension: “Agents must be able to work autonomously; their independent operation is exactly what makes them valuable. But humans should retain control over how their goals are pursued” (Anthropic, 2025a).

Anthropic 的安全智能体框架阐明了一个核心冲突：“智能体必须能够自主工作；其独立运行正是其价值所在。但人类应当保持对如何追求其目标的控制权”（Anthropic, 2025a）。

Claude’s Constitution resolves this not through rigid decision procedures but by cultivating “good judgment and sound values that can be applied contextually” (Anthropic, 2026b). These commitments, together with empirical findings about how developers actually use the tool (Huang et al., 2025; McCain et al., 2026), point to five human values that shape the architecture.

Claude 的宪章（Constitution）解决这一冲突的方式不是通过死板的决策程序，而是通过培养“能够结合语境应用的良好判断力和合理价值观”（Anthropic, 2026b）。这些承诺，结合关于开发者实际如何使用该工具的实证研究结果（Huang et al., 2025; McCain et al., 2026），指向了塑造该架构的五个人类价值。

2.1 Five Values and Philosophies

2.1 五个价值与哲学

Human Decision Authority. The human retains ultimate decision authority over what the system does, organized through a principal hierarchy (Anthropic, then operators, then users) that formalizes who holds authority over what (Anthropic, 2026b).

人类决策自主权。人类对系统所做的事情保留最终的决策自主权，这是通过一个主体层级架构（Anthropic，接着是运营商，然后是用户）来组织的，该架构形式化了谁对什么拥有权威（Anthropic, 2026b）。

The system is designed so that humans can exercise informed control: they can observe actions in real time, approve or reject proposed operations, interrupt compatible in-progress operations, and audit after the fact.

该系统旨在让人类能够行使知情控制：他们可以实时观察动作、批准或拒绝提议的操作、中断处于进行中的兼容操作，并在事后进行审计。

When Anthropic found that users approve 93% of permission prompts (Hughes, 2026), the response was not to add more warnings but to restructure the problem: defined boundaries (sandboxing, auto-mode classifiers) within which the agent can work freely, rather than per-action approvals that users stop reviewing once habituated (Dworken and Weller-Davies, 2025).

当 Anthropic 发现用户批准了 93% 的权限提示时（Hughes, 2026），其应对措施并不是增加更多警告，而是重新构建该问题：划定清晰的边界（沙箱、自动模式分类器）让智能体在其中自由工作，而不是采用用户一旦习惯后便不再审阅的逐项动作批准机制（Dworken and Weller-Davies, 2025）。

Safety, Security, and Privacy. The system protects humans, their code, their data, and their infrastructure from harm, even when the human is inattentive or makes mistakes. This is distinct from Human Decision Authority: where authority is about the human’s power to choose, safety is about the system’s obligation to protect even when that power lapses.

安全、保障与隐私。即使在人类疏忽或犯错时，系统也会保护人类、他们的代码、数据以及基础设施免受伤害。这与“人类决策自主权”不同：自主权关系到人类的选择权力，而安全性则关系到系统在即使该权力失效时也必须予以保护的义务。

Anthropic’s safe-agents framework separately identifies securing agent interactions and protecting privacy across extended interactions as core commitments (Anthropic, 2025a). The auto-mode threat model (Hughes, 2026) explicitly targets four risk categories: overeager behavior, honest mistakes, prompt injection, and model misalignment.

Anthropic 的安全智能体框架分别将保障智能体交互安全和在长期交互中保护隐私确定为核心承诺（Anthropic, 2025a）。自动模式（auto-mode）的威胁模型（Hughes, 2026）明确针对四类风险：过度积极的行为、无心之失、提示词注入以及模型失调。

Reliable Execution. The agent does what the human actually meant, stays coherent over time, and supports verifying its work before declaring success.

可靠执行。智能体做人类真正想表达的事情，在长时间内保持连贯，并支持在宣布成功之前对其工作进行验证。

This value spans both single-turn correctness (did it interpret the request faithfully?) and long-horizon dependability (does it remain coherent across context window boundaries, session resumption, and multi-agent delegation?).

这一价值既涵盖了单轮正确性（它是否忠实地理解了请求？），也涵盖了长时程的可靠性（它在跨越上下文窗口边界、会话恢复和多智能体委派时是否能保持连贯？）。

Anthropic’s product documentation (Anthropic, 2026d) describes a three-phase loop that the agent repeats until the task is complete: gather context, take action, and verify results.

Anthropic 的产品文档（Anthropic, 2026d）描述了一个由三个阶段组成的循环，智能体会重复该循环直到任务完成：收集上下文、采取行动以及验证结果。

The agent design guidance (Schluntz and Zhang, 2024) further emphasizes that “ground truth from the environment” at each step assesses progress.

智能体设计指南（Schluntz and Zhang, 2024）进一步强调，每一步中“来自环境的真实情况（ground truth）”都会评估进展。

The harness-design guidance (Rajasekaran, 2026) likewise notes that “agents tend to respond by confidently praising the work,” even when quality is mediocre, motivating separation of generation from evaluation.

测试框架（harness）设计指南（Rajasekaran, 2026）同样指出，“智能体往往倾向于通过自信地赞美工作来进行回应”，即使工作质量极其平庸，这也促成了生成与评估的分离。

Capability Amplification. The system materially increases what the human can accomplish per unit of effort and cost. Anthropic’s internal survey (Huang et al., 2025), discussed in Section 1, suggests that the architecture enables qualitatively new workflows, not merely faster existing ones: approximately 27% of tasks represented work that would not otherwise have been attempted.

**能力放大。系统实质性地增加了人类在每单位努力和成本下所能成就的事情。**正如第 1 节中讨论的 Anthropic 内部调查（Huang et al., 2025）所表明的，该架构开启了性质全新的工作流，而不仅仅是加速了现有的工作流：大约 27% 的任务代表了若非使用该工具否则根本不会尝试的工作。

The system is described by its creators as “a Unix utility rather than a traditional product,” built from the smallest building blocks that are “useful, understandable, and extensible” (Cherny and Wu, 2025).

该系统被其创造者描述为“一个 Unix 工具，而非传统的产品”，它由“有用、可理解且可扩展”的最微小构建块打造而成（Cherny and Wu, 2025）。

The architecture invests in deterministic infrastructure (context management, tool routing, recovery) rather than decision scaffolding (explicit planners or state graphs), on the premise that increasingly capable models benefit more from a rich operational environment than from frameworks that constrain their choices.

该架构投资于确定性的基础设施（上下文管理、工具路由、恢复），而不是决策脚手架（显式规划器或状态图），其前提是：能力日益增强的模型从丰富的运行环境中获益，要远多于从限制其选择的框架中获益。

Contextual Adaptability. The system fits the user’s specific context (their project, tools, conventions, and skill level) and the relationship improves over time.

上下文适应性。系统契合用户的特定语境（他们的项目、工具、惯例和技能水平），并且这种关系会随着时间的推移而改善。

The extension architecture (CLAUDE.md, skills, MCP, hooks, plugins) provides configurability at multiple levels of context cost (Sections 6 and 7). Longitudinal data (McCain et al., 2026) shows that the human-agent relationship evolves: auto-approve rates increase from approximately 20% at fewer than 50 sessions to over 40% by 750 sessions.

扩展架构（CLAUDE.md、技能、MCP、钩子、插件）在多个上下文成本级别上提供了可配置性（第 6 节和第 7 节）。纵向数据（McCain et al., 2026）表明，人类与智能体之间的关系是不断演进的：自动批准率从会话少于 50 次时的约 20%，提高到会话达到 750 次时的 40% 以上。

This pattern, described as autonomy that is “co-constructed by the model, the user, and the product,” means the system is designed for trust trajectories rather than fixed trust states. MCP’s donation to the Linux Foundation’s Agentic AI Foundation (The Linux Foundation, 2025) reflects the ecosystem dimension of this value.

这种被描述为由“模型、用户和产品共同构建”的自主性模式，意味着该系统是为信任轨迹（trust trajectories）而非固定的信任状态而设计的。MCP 向 Linux 基金会的智能体 AI 基金会（Agentic AI Foundation）的捐赠（The Linux Foundation, 2025）反映了这一价值的生态系统维度。

2.3 From Values to Architecture

Each value traces through its principles to specific architectural decisions:

  • Safety, Security, and Privacy motivates defense in depth, deny-first defaults, reversibility-weighted assessment, externalized policy, and isolated subagent boundaries (Sections 5 and 8).

    安全、保障与隐私：驱动了纵深防御、默认拒绝设置、基于可逆性的加权评估、外部化策略以及隔离的子智能体边界（第 5 和第 8 节）。

  • Reliable Execution motivates context-as-scarce-resource, append-only durable state, graceful recovery, isolated subagent boundaries, and defense in depth (Sections 4 and 7 to 9).

    能力放大：驱动了极简脚手架、可组合的扩展性、基于可逆性的加权风险、上下文管理以及优雅恢复（第 4 至 6 节）。

  • Contextual Adaptability motivates transparent file-based memory, composable extensibility, the graduated trust spectrum, and externalized programmable policy (Sections 5 to 7).

    上下文适应性：驱动了基于文件的透明记忆、可组合的扩展性、渐进式信任光谱以及外部化可编程策略（第 5 至 7 节）。

These mappings also reveal what the architecture does not do: it does not impose explicit planning graphs on the model’s reasoning, does not provide a single unified extension mechanism, and does not restore all session-scoped trust-related state across resume. 这些映射同样揭示了该架构所没有做的事情：它没有在模型的推理上强加显式的规划图，没有提供单一的统一扩展机制，也没有在恢复会话时还原所有会话范围内的信任相关状态。这些特征的缺失与上述原则集是一致的。

2.4 An Evaluative Lens: Long-term Capability Preservation

2.4 评估视角：长期能力的维持

The five values above describe what the architecture is designed to serve. This paper also applies a sixth concern, whether the architecture preserves long-term human capability, as an evaluative lens.

上述五个价值描述了该架构的设计服务对象。本文还引入了第六个关注点——该架构是否能维持人类的长期能力——并将其作为一种评估视角。

This concern is real: Anthropic’s own study of 132 engineers and researchers (Huang et al., 2025) documents a “paradox of supervision” in which overreliance on AI risks atrophying the skills needed to supervise it, and independent research (Shen and Tamkin, 2026) finds that developers in AI-assisted conditions score 17% lower on comprehension tests.

这种担忧是切实的：Anthropic 自身对 132 名工程师和研究人员的研究（Huang et al., 2025）记录了一种 “监督悖论”，即过度依赖 AI 会带来使监督 AI 所需技能发生萎缩的风险；而独立研究（Shen and Tamkin, 2026）也发现，在 AI 辅助环境下的开发者在理解力测试中的得分要低 17%。

However, this concern is not prominently reflected as a design driver in the architecture or in Anthropic’s stated design values.

然而，在现有的架构或 Anthropic 阐明的设计价值中，这一关注点并未作为主要的设计驱动力得到显著体现。

We therefore treat it not as a co-equal value but as a cross-cutting concern: a question applied across all five values in Section 11, asking whether short-term amplification comes at the cost of long-term human understanding, codebase coherence, and the developer pipeline.

因此，我们不将其视为一个平等的价值，而是作为一个横切关注点（cross-cutting concern）：在第 11 节中将其应用于所有五个价值中，借此探讨短期的能力放大是否会以牺牲人类的长期理解、代码库一致性以及开发人员的培养梯队为代价。

3 Architecture Overview

3 架构概述

Building a production coding agent requires answering several recurring design questions: where should reasoning live, how many execution engines are needed, what safety posture to adopt, and what resource to treat as the binding constraint. Claude Code’s architecture can be read as one set of answers to these questions.

构建一个生产级的编码智能体需要回答几个周期性出现的设计问题：推理应当存在于何处、需要多少个执行引擎、采取何种安全姿态，以及将何种资源视为绑定的核心限制。Claude Code 的架构可以被解读为对这些问题的一套解答。

At the implementation level, the system has seven components connected by a main data flow: a user submits a prompt through one of several interfaces, which feeds into a shared agent loop.

在实现层面上，该系统包含由一条主数据流连接的七个组件：用户通过几种接口之一提交提示词，该提示词会被输入到一个共享的智能体循环中。

The agent loop assembles context, calls the Claude model, receives responses that may include tool-use requests, routes those requests through a permission system, and dispatches approved actions to concrete tools that interact with the execution environment.

该智能体循环负责组装上下文、调用 Claude 模型、接收可能包含工具使用请求的响应、将这些请求路由通过权限系统，并将批准的动作分发给与执行环境交互的具体工具。

Throughout this process, state and persistence mechanisms record the conversation transcript, manage session identity, and support resume, fork, and rewind operations.

在整个过程中，状态与持久化机制会记录对话文本、管理会话身份，并支持恢复（resume）、分叉（fork）和倒带（rewind）操作。

3.1 Design Questions and Running Example

3.1 设计问题与运行示例

The description is organized around four design questions that recur across production coding agents, each grounding one or more of the design principles identified in Table 1. Each question is introduced here with Claude Code’s answer, a note on plausible alternatives, and then demonstrated progressively through Sections 4 to 9.

接下来的阐述将围绕生产级编码智能体中反复出现的四个设计问题展开，每个问题都构成了表 1 中所列的一项或多项设计原则的基础。这里在引入每个问题时，均附带了 Claude Code 的解答、对可能的可选方案的说明，并在第 4 节至第 9 节中进行渐进式的演示。

Where does reasoning live? In Claude Code, the model reasons about what to do; the harness is responsible for executing actions. The model emits tool_use blocks as part of its response, and the harness parses them, checks permissions, dispatches them to tool implementations, and collects results (query.ts).

推理存在于何处？在 Claude Code 中，模型负责推理要做什么；运行框架（harness）则负责执行动作。模型在其响应中输出 tool_use（工具使用）块，运行框架对其进行解析、检查权限、将其分发给具体的工具实现并收集结果（query.ts）。

The model never directly accesses the filesystem, runs shell commands, or makes network requests. This separation has a security consequence: because reasoning and enforcement occupy separate code paths, a compromised or adversarially manipulated model cannot override the sandboxing, permission checks, or deny-first rules implemented in the harness.

模型绝不直接访问文件系统、运行 Shell 命令或发起网络请求。这种分离带来了一个安全上的结果：由于推理和执行占据了不同的代码路径，一个被攻破或遭受对抗性操纵的模型无法推翻运行框架中实现的沙箱、权限检查或默认拒绝规则。

The model’s only interface to the outside world is the structured tool_use protocol, which the harness validates before execution. Community analysis of the extracted source estimates that only about 1.6% of Claude Code’s codebase constitutes AI decision logic, with the remaining 98.4% being operational infrastructure, a ratio that illustrates how thin the core agent reasoning layer is.

模型与外部世界的唯一接口就是结构化的 tool_use 协议，运行框架在执行前会对其进行验证。社区对提取出的源码进行分析估计，Claude Code 的代码库中只有大约 1.6% 构成了 AI 决策逻辑，其余 98.4% 全是运行基础设施，这一比例生动地说明了核心智能体推理层是多么地薄。

Alternative designs invest more heavily in scaffolding-side reasoning: Devin maintains explicit planning and task-tracking structures, while LangGraph (LangChain, Inc., 2024) routes control flow through developer-defined state graphs.

可选的设计方案在脚手架侧的推理上投入更深：Devin 维持着显式的规划和任务追踪结构，而 LangGraph（LangChain, Inc., 2024）则通过开发者定义的状态图来路由控制流。

How many execution engines? Claude Code uses a single queryLoop() function that executes regardless of whether the user is interacting through an interactive terminal, a headless CLI invocation, the Agent SDK, or an IDE integration (query.ts). Only the rendering and user-interaction layer varies.

需要多少个执行引擎？Claude Code 使用单个 queryLoop() 函数，无论用户是通过交互式终端、无头（headless）CLI 调用、Agent SDK 还是 IDE 集成进行交互，该函数都会执行（query.ts）。改变的仅仅是渲染和用户交互层。

Other systems use mode-specific engines: for example, an IDE integration may follow a different code path than a CLI tool, trading uniformity for surface-specific optimization.

其他系统则使用特定模式的引擎：例如，IDE 集成可能会走一条与 CLI 工具不同的代码路径，以此牺牲统一性来换取针对特定界面的优化。

What is the default safety posture? Claude Code’s default safety posture is deny-first with human escalation: deny rules override ask rules override allow rules, and unrecognized actions are escalated to the user rather than allowed silently (permissions.ts).

默认的安全姿态是什么？Claude Code 的默认安全姿态是默认拒绝与人类升级确认：拒绝规则（deny）覆盖询问规则（ask），询问规则覆盖允许规则（allow），并且未识别的动作会升级提交给用户确认，而不是默默允许（permissions.ts）。

Multiple independent safety layers (permission rules, PreToolUse hooks, the auto-mode classifier when enabled, and optional shell sandboxing) apply in parallel, so any one can block an action (Section 5). This combines the deny-first with human escalation and defense in depth with layered mechanisms principles from Table 1.

多个独立的安全层（权限规则、PreToolUse 钩子、启用时的自动模式分类器以及可选的 Shell 沙箱）并行适用，因此任何一层都可以阻断动作（第 5 节）。这结合了表 1 中的“默认拒绝与人类升级确认”以及“具备分层机制的纵深防御”原则。

Alternative approaches shift the trust boundary elsewhere: SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b) rely on container-based isolation to contain arbitrary execution, while Aider (Gauthier, 2024) uses git-based rollback as its primary safety net.

可选的方法将信任边界移到了别处：SWE-Agent 和 OpenHands（Yang et al., 2024; Wang et al., 2024b）依赖基于容器的隔离来遏制任意执行，而 Aider（Gauthier, 2024）则将基于 Git 的回滚作为其主要的安全网。

What is the binding resource constraint? In Claude Code, the context window (200K for older models, 1M for the Claude 4.6 series) is the binding resource constraint. Five distinct context-reduction strategies execute before every model call (query.ts), and several other subsystem decisions (lazy loading of instructions, deferred tool schemas, summary-only subagent returns) exist to limit context consumption (Section 7).

绑定的核心资源限制是什么？在 Claude Code 中，上下文窗口（旧模型为 200K，Claude 4.6 系列为 1M）是绑定的核心资源限制。在每次模型调用之前，都会执行五种不同的上下文裁剪策略（query.ts），并且还存在若干其他子系统决策（延迟加载指令、推迟工具 Schema 加载、子智能体仅返回摘要）以限制上下文消耗（第 7 节）。

The five-layer pipeline exists because no single compaction strategy addresses all types of context pressure. Budget reduction targets individual tool outputs that overflow size limits. Snip handles temporal depth. Microcompact reacts to cache overhead. Context collapse manages very long histories. Auto-compact performs semantic compression as a last resort.

之所以存在这个五层流水线，是因为没有任何单一的压缩策略能解决所有类型的上下文压力。预算削减（Budget reduction）针对的是超出大小限制的单个工具输出。裁剪（Snip）处理时间深度的文件。微压缩（Microcompact）应对缓存开销。上下文坍缩（Context collapse）管理极长的历史记录。自动压缩（Auto-compact）则作为最后手段进行语义压缩。

Each layer operates at a different cost-benefit tradeoff, and earlier, cheaper layers run before costlier ones. Alternative architectures treat other resources as the primary bottleneck, for instance compute budget (limiting the number of model calls or tool invocations) or working memory (maintaining an explicit scratchpad rather than relying on the conversation history).

每一层都在不同的成本效益权衡下运行，而且较早、较便宜的层会在较昂贵的层之前运行。可选的架构将其他资源视为主要瓶颈，例如计算预算（限制模型调用或工具调用的次数）或工作记忆（维持一个显式的便签本（scratchpad），而不是依赖对话历史记录）。

Subsequent sections trace the request through the query loop, permission gate, tool pool, context window, subagent delegation, and session persistence.

运行示例。为了使这些原则切实落地，我们将一个单一的任务贯穿第 3 节至第 9 节：“修复 auth.test.ts 中失败的测试”。在本节中，用户通过 Claude Code 的其中一个接口提交了提示词。后续章节将追踪该请求通过查询循环、权限闸门、工具池、上下文窗口、子智能体委派以及会话持久化的全过程。

    接口：交互式 CLI、无头 CLI（claude -p）、Agent SDK 以及 IDE/桌面/浏览器。所有界面都向同一个循环提供输入。

  3. Agent loop: The iterative cycle of model call, tool dispatch, and result collection, implemented as the queryLoop() async generator in query.ts.

    智能体循环：模型调用、工具分发和结果收集的迭代循环，在 query.ts 中实现为 queryLoop() 异步生成器。

  4. Permission system: Deny-first rule evaluation (permissions.ts), the auto-mode ML classifier, and hook-based interception (types/hooks.ts).

    权限系统：默认拒绝规则评估（permissions.ts）、自动模式机器学习分类器以及基于钩子（hook）的拦截（types/hooks.ts）。

  5. Tools: Up to 54 built-in tools (19 unconditional, 35 conditional on feature flags and user type) assembled via assembleToolPool() (tools.ts), merged with MCP-provided tools. Plugins contribute indirectly through MCP servers and the skill/command registry.

    工具：通过 assembleToolPool()（tools.ts）组装的多达 54 个内置工具（19 个无条件启用，35 个取决于特性标志和用户类型），并与 MCP（模型上下文协议）提供的工具进行合并。插件通过 MCP 服务器以及技能/命令注册表间接做出贡献。

  6. State & persistence: Mostly append-only JSONL session transcripts (sessionStorage.ts), global prompt history (history.ts), and subagent sidechain files.

    状态与持久化：主要为只读追加的 JSONL 会话历史文本（sessionStorage.ts）、全局提示词历史（history.ts）以及子智能体侧链（sidechain）文件。

  7. Execution environment: Shell execution with optional sandboxing (shouldUseSandbox.ts), filesystem operations, web fetching, MCP server connections, and remote execution.

    执行环境：带有可选沙箱的 Shell 执行（shouldUseSandbox.ts）、文件系统操作、网络获取、MCP 服务器连接以及远程执行。

The data flow follows a left-to-right spine: the user submits a request through an interface, which enters the agent loop. The loop proposes actions to the permission system; approved actions reach tools, which interact with the execution environment and return tool_result messages back to the loop. State and persistence sit alongside the loop, recording transcripts and loading prior session data.

数据流遵循从左到右的主干：用户通过接口提交请求，请求进入智能体循环。该循环向权限系统提议动作用于评估；获得批准的动作将到达工具，工具与执行环境交互并将 tool_result（工具结果）消息返回给循环。状态与持久化并存于循环一侧，负责记录文本并加载先前的会话数据。

The application entry point main() in main.tsx initializes security settings (including NoDefaultCurrentDirectoryInExePath to prevent Windows PATH hijacking), registers signal handlers for graceful shutdown, and dispatches to the appropriate execution mode.

main.tsx 中的应用程序入口点 main() 用于初始化安全设置（包括 NoDefaultCurrentDirectoryInExePath 以防止 Windows PATH 劫持）、注册用于优雅停机的信号处理程序，并分发到相应的执行模式。

Context assembly is a memoized state loader, not a routing hub: getSystemContext() (context.ts) computes session-level system context including git status, and getUserContext() (context.ts) loads the CLAUDE.md hierarchy and current date. 状态层（上下文组装、运行时状态、持久化、记忆、侧链）。上下文组装是一个带有记忆化（memoized）的状态加载器，而不是路由中心：getSystemContext()（context.ts）计算会话级的系统上下文（包括 git 状态），而 getUserContext()（context.ts）加载 CLAUDE.md 层级结构和当前日期。两者都会被缓存以供复用：系统上下文会被追加到系统提示词中，而用户上下文则作为一条用户上下文消息添加。

Q: claude.md不是系统级的吗?

要理解这段话，我们需要区分 “内容的作用域（它是管什么的）” 和 “代码的实现/注入方式（它是怎么塞给大模型的）”。

你直觉上认为 CLAUDE.md 是“系统级”的，是因为从用户角度看，它里面写的是项目的全局规范、编码准则，用来规范模型的整体行为，这非常符合我们对“系统提示词（System Prompt）”的认知。

但在这段技术分析中，作者是从底层代码架构和 Token 经济学的角度来解释的。我们可以从以下三个维度来拆解为什么它被归类为“用户上下文消息”：

  1. 注入位置的不同（System Prompt vs. User Message）

在对接大模型（如 Claude）的 API 时，输入通常分为 system 字段和 messages 列表。

  • getSystemContext()（会话级系统上下文）：

    它处理的是真正意义上的底层环境信息，比如当前的 git status（有哪些文件被修改了、当前在哪个分支）。这些信息是系统底层状态的反映，因此它被直接追加到 API 的 System Prompt 中，作为模型思考的底层背景。

  • getUserContext()（包含 CLAUDE.md）：

    代码将 CLAUDE.md 的内容以及当前日期，打包成了一条特殊的 User Message（用户消息），塞进了对话历史（messages）的头部或特定位置。

  1. 为什么架构上要把 CLAUDE.md 做成用户消息？

这种设计看似违反直觉，但在工程实践（特别是构建 Agent 护栏）中非常高明：

  • 权重的微调与控制：很多 LLM 在实际推理时，对指令的遵循程度会受到其所在位置的影响。将项目特异性的规范（CLAUDE.md）作为用户上下文输入，有时能让模型更明确地感知到“这是用户在这个项目中立下的规矩”。
  • 多层级结构（Hierarchy）的动态组装：原文提到了 CLAUDE.md 层级结构。Claude Code 支持在多级目录下放置不同的 CLAUDE.md（例如根目录有一个，某个子模块目录下还有一个）。当 Agent 切换工作目录时，getUserContext() 可能会动态、延迟加载（lazy-loaded）这些文件。
  • 与动态机制解耦：真正的系统提示词一般保持高度静态，以便最大化利用提示词缓存（Prompt Caching）来降低 Token 成本并提高响应速度。如果把经常变动的、或者需要根据路径动态组合的 CLAUDE.md 强行揉进系统提示词，可能会频繁导致缓存失效。
  1. 总结

我们可以用一个简单的结构对比来看清它在底层运行时（Runtime）的真实样貌：

Plaintext

所以，CLAUDE.md 在内容功能上确实扮演着系统规范的角色，但在 Claude Code 的代码实现里，它是作为“用户侧的上下文数据”被读取并作为消息注入到对话中的。 这段话揭示的正是这种“功能定义”与“底层工程实现”之间的错位。

The src/state/ directory manages runtime application state. Session transcripts are stored as mostly append-only JSONL files at project-specific paths (sessionStorage.ts). The CLAUDE.md + memory subsystem provides a four-level instruction hierarchy (claudemd.ts) from managed settings to directory-specific files, plus auto-memory entries that Claude writes during conversations (Section 7.2). Sidechain transcripts (sessionStorage.ts:247) store each subagent’s conversation in a separate file, preventing subagent content from inflating the parent context (Section 8.3). Global prompt history is maintained in history.jsonl (history.ts). Resume and fork operations reconstruct session state from transcripts (conversationRecovery.ts).

src/state/ 目录负责管理运行时应用程序的状态。会话历史文本作为主要为只读追加的 JSONL 文件存储在特定于项目的路径中（sessionStorage.ts）。CLAUDE.md + 记忆子系统提供了一个从托管设置到特定目录文件的四级指令层级结构（claudemd.ts），外加 Claude 在对话期间写入的自动记忆条目（第 7.2 节）。侧链历史文本（sessionStorage.ts:247）将每个子智能体的对话存储在独立的文件中，防止子智能体的内容撑大父上下文（第 8.3 节）。全局提示词历史维护在 history.jsonl（history.ts）中。恢复（resume）和分叉（fork）操作通过历史文本重构会话状态（conversationRecovery.ts）。

Backend layer (execution backends, external resources). Shell command execution with optional sandboxing (BashTool.tsx, PowerShellTool.tsx), remote execution support (src/remote/), MCP server connections across multiple transport variants including stdio, SSE, HTTP, WebSocket, SDK, and IDE-specific adapters (services/mcp/client.ts), and 42 tool subdirectories in src/tools/ implement concrete tool logic.

后端层（执行后端、外部资源）。带有可选沙箱的 Shell 命令执行（BashTool.tsx, PowerShellTool.tsx）、远程执行支持（src/remote/）、跨越多种传输变体（包括 stdio、SSE、HTTP、WebSocket、SDK 和特定于 IDE 的适配器）的 MCP 服务器连接（services/mcp/client.ts），以及 src/tools/ 中的 42 个工具子目录，共同实现了具体的工具逻辑。

3.4 QueryEngine: A Clarification

3.4 QueryEngine：澄清

The class documentation at QueryEngine.ts states: “QueryEngine owns the query lifecycle and session state for a conversation. It extracts the core logic from ask() into a standalone class that can be used by both the headless/SDK path and (in a future phase) the REPL.” The class is a conversation wrapper for non-interactive surfaces, not the engine itself.

QueryEngine.ts 中的类文档指出：“QueryEngine 拥有对话的查询生命周期和会话状态。它将 ask() 中的核心逻辑提取到一个独立的类中，该类既可用于无头/SDK 路径，也可用于（在未来阶段）REPL。”该类是针对非交互式界面的对话包装器，而不是引擎本身。

Its constructor accepts a QueryEngineConfig with initial messages, an abort controller, a file-state cache, and other per-conversation state. Its submitMessage() method is an async generator that orchestrates a single turn. The shared query path lives in query() (query.ts), which wraps an internal queryLoop(); QueryEngine delegates to query().

其构造函数接受一个包含初始消息、中止控制器、文件状态缓存以及其他单次对话状态的 QueryEngineConfig。其 submitMessage() 方法是一个编排单轮交互的异步生成器。共享的查询路径存在于 query()（query.ts）中，该函数包装了一个内部的 queryLoop()；QueryEngine 将任务委派给 query()。

This distinction matters architecturally: the interactive CLI also calls query(), bypassing QueryEngine entirely. The shared code path is the loop function, not the engine class.

这种区分在架构上非常重要：交互式 CLI 同样调用 query()，从而完全绕过了 QueryEngine。共享的代码路径是循环函数，而不是引擎类。

3.5 Permission and Safety Layers

3.5 权限与安全层

The safety-by-default principle is implemented through seven independent layers. A request must pass through all applicable layers, and any single layer can block it:

默认安全原则是通过七个独立的层来实现的。一个请求必须通过所有适用的层，任何单一的层都可以阻断它：

  1. Tool pre-filtering (tools.ts): Blanket-denied tools are removed from the model’s view before any call, preventing the model from attempting to invoke them.

    工具预过滤（tools.ts）：在进行任何调用之前，被通盘拒绝的工具会从模型的视野中移除，从而防止模型尝试调用它们。

  2. Deny-first rule evaluation (permissions.ts): Deny rules always take precedence over allow rules, even when the allow rule is more specific.

    默认拒绝规则评估（permissions.ts）：拒绝规则（deny）总是优先于允许规则（allow），即使允许规则更加具体。

  3. Permission mode constraints (types/permissions.ts): The active mode determines baseline handling for requests matching no explicit rule.

    权限模式约束（types/permissions.ts）：激活的模式决定了对未匹配到显式规则的请求的基线处理方式。

  4. Auto-mode classifier: An ML-based classifier evaluates tool safety, potentially denying requests the rule system would allow.

    自动模式分类器：一个基于机器学习的分类器评估工具的安全性，有可能拒绝规则系统原本会允许的请求。

  5. Shell sandboxing (shouldUseSandbox.ts): Approved shell commands may still execute inside a sandbox restricting filesystem and network access.

    Shell 沙箱化（shouldUseSandbox.ts）：获批的 Shell 命令仍可能在一个限制文件系统和网络访问的沙箱内执行。

  6. Not restoring permissions on resume (conversationRecovery.ts): Session-scoped permissions are not restored on resume or fork.

    恢复会话时不还原权限（conversationRecovery.ts）：在恢复或分叉会话时，属于先前会话范围的权限不会被还原。

  7. Hook-based interception (types/hooks.ts): PreToolUse hooks can modify permission decisions; PermissionRequest hooks can resolve decisions asynchronously alongside the user dialog (or before it, in coordinator mode).

    基于钩子的拦截（types/hooks.ts）：PreToolUse 钩子可以修改权限决策；PermissionRequest 钩子可以伴随用户对话异步解析决策（或者在协调器模式下，在对话之前解析）。

These layers are described in detail in Section 5.

这些层将在第 5 节中详细描述。

3.6 Context as Bottleneck: Beyond Compaction

3.6 上下文作为瓶颈：压缩之外

Beyond the five-layer compaction pipeline (detailed in Section 7), several other subsystem decisions reflect the context-as-bottleneck constraint:

除了五层压缩流水线（详见第 7 节）之外，其他若干子系统的决策也反映了“视上下文为瓶颈”的约束：

  • CLAUDE.md lazy loading: The base CLAUDE.md hierarchy is loaded at session start, but additional nested-directory instruction files and conditional rules are loaded only when the agent reads files in those directories, preventing unused instructions from consuming context.

    CLAUDE.md 延迟加载：基础的 CLAUDE.md 层级结构在会话开始时加载，但只有当智能体读取对应目录中的文件时，才会加载额外的嵌套目录指令文件和条件规则，以此防止未使用的指令消耗上下文。

  • Deferred tool schemas: When ToolSearch is enabled, some tools include only their names in the initial context; full schemas are loaded on demand.

    推迟工具 Schema 加载：当启用 ToolSearch 时，某些工具在初始上下文中仅包含其名称；完整的 Schema 会根据需求延迟加载。

  • Subagent summary-only return: Subagents return only summary text to the parent, not their full conversation history (Section 8).

    子智能体仅返回摘要：子智能体仅向父智能体返回摘要文本，而不返回其完整的对话历史记录（第 8 节）。

  • Per-tool-result budget: Individual tool results are capped at a configurable size, preventing a single verbose output from consuming disproportionate context.

    单个工具结果的预算限制：单个工具的结果被限制在可配置的大小内，防止单一冗长的输出消耗不成比例的上下文。

4 Turn Execution: The Agentic Query Loop

4 轮次执行：智能体查询循环

When the user submits “Fix the failing test in auth.test.ts,” the input enters a reactive loop, one of several possible orchestration patterns for coding agents. This section examines Claude Code’s choice of a simple while-loop architecture and traces one turn of that loop end-to-end, illustrating three design principles from Table 1: minimal scaffolding with maximal operational harness, context as scarce resource with progressive management, and graceful recovery and resilience.

当用户提交“修复 auth.test.ts 中失败的测试”时，该输入会进入一个响应式循环（reactive loop），这是编码智能体几种可能的编排模式之一。本节将审视 Claude Code 对简单 while 循环架构的选择，并端到端地追踪该循环的一个轮次，以此阐述表 1 中的三项设计原则：极简脚手架与极高运行框架、视上下文为稀缺资源并进行渐进式管理，以及优雅恢复与韧性。

4.1 The Query Pipeline

4.1 查询流水线

Each turn follows a fixed sequence (Figure 2, query.ts):

每个轮次都遵循一个固定的序列（图 2，query.ts）：

  1. Settings resolution. The queryLoop() function destructures immutable parameters including the system prompt, user context, permission callback, and model configuration.

    设置解析：queryLoop() 函数解构不可变参数，包括系统提示词、用户上下文、权限回调和模型配置。

  2. Mutable state initialization. A single State object stores all mutable state across iterations, including messages, tool context, compaction tracking, and recovery counters. The loop’s seven continue points (the “continue sites”) each overwrite this object in one whole-object assignment rather than mutating fields individually.

    可变状态初始化：一个单一的 State 对象存储了跨迭代的所有可变状态，包括消息、工具上下文、压缩追踪和恢复计数器。循环的七个 continue 点（即“continue 站点”）各自通过一次整包对象赋值来重写该对象，而不是单独修改各个字段。

  3. Context assembly. The function getMessagesAfterCompactBoundary() retrieves messages from the last compact boundary forward, ensuring that compacted content is represented by its summary rather than the original messages.

    上下文组装：getMessagesAfterCompactBoundary() 函数检索自上一个压缩边界以来的消息，确保被压缩的内容由其摘要代替，而非原始消息。

  4. Pre-model context shapers. Five shapers execute sequentially (Section 4.3).

    模型前置上下文整形器：五个整形器按顺序执行（第 4.3 节）。

  5. Model call. A for await loop over deps.callModel() streams the model’s response, passing assembled messages (with user context prepended), the full system prompt, thinking configuration, the available tool set, an abort signal, the current model specification, and additional options including fast-mode settings, effort value, and fallback model.

    模型调用：一个遍历 deps.callModel() 的 for await 循环流式传输模型的响应，传入组装好的消息（前置了用户上下文）、完整的系统提示词、思维配置（thinking configuration）、可用工具集、中止信号、当前模型规范，以及包括快速模式设置、思考烈度值（effort value）和备用模型在内的附加选项。

  6. Tool-use dispatch. If the response contains tool_use blocks, they flow to the tool orchestration layer (Section 4.2).

    工具使用分发：如果响应包含 tool_use 块，它们将流向工具编排层（第 4.2 节）。

  7. Permission gate. Each tool request passes through the permission system (Section 5).

    权限闸门：每个工具请求都必须通过权限系统（第 5 节）。

  8. Tool execution and result collection. Tool results are added to the conversation as tool_result messages, and the loop continues.

    工具执行与结果收集：工具结果作为 tool_result 消息添加到对话中，然后循环继续。

  9. Stop condition. If the response contains no tool_use blocks (text only), the turn is complete.

    停止条件：如果响应不包含 tool_use 块（仅有文本），则该轮次完成。

The queryLoop() function is defined as an AsyncGenerator, yielding StreamEvent, RequestStartEvent, Message, TombstoneMessage, and ToolUseSummaryMessage events as it progresses. This generator-based design enables streaming output to the UI layer while maintaining a single synchronous control flow within the loop.

queryLoop() 函数被定义为一个 AsyncGenerator（异步生成器），在其推进过程中会产出（yield）StreamEvent、RequestStartEvent、Message、TombstoneMessage 和 ToolUseSummaryMessage 事件。这种基于生成器的设计使得能够向 UI 层流式传输输出，同时在循环内保持单一的同步控制流。

Claude Code’s reactive loop follows the ReAct pattern (Yao et al., 2022): the model generates reasoning and tool invocations, the harness executes actions, and results feed the next iteration. Alternative orchestration patterns include explicit graph-based routing (LangGraph) (LangChain, Inc., 2024), where control flow is defined as a state machine with typed edges, and tree-search methods (Zhou et al., 2023) that explore multiple action trajectories before committing.

Claude Code 的响应式循环遵循 ReAct 模式（Yao et al., 2022）：模型生成推理和工具调用，运行框架执行动作，结果再反馈给下一次迭代。可选的编排模式包括显式的基于图的路由（LangGraph）（LangChain, Inc., 2024）——其控制流被定义为具有类型化边缘的状态机，以及在提交前探索多种动作轨迹的树搜索方法（Zhou et al., 2023）。

Anthropic’s own documentation (Schluntz and Zhang, 2024) identifies five composable workflow patterns (prompt chaining, routing, parallelization, orchestrator-workers, and evaluator-optimizer) of which Claude Code primarily uses the orchestrator-workers pattern for subagent delegation (Section 8) while keeping the core loop reactive. The reactive design trades search completeness for simplicity and latency: each turn commits to one action sequence without backtracking.

Anthropic 自己的文档（Schluntz and Zhang, 2024）确定了五种可组合的工作流模式（提示词链式组合、路由、并行化、编排器-工人以及评估器-优化器），其中 Claude Code 核心保持响应式循环，但在子智能体委派中主要使用编排器-工人模式（第 8 节）。这种响应式设计用搜索的完整性换取了简单性和低延迟：每个轮次都会投入到某一个动作序列中，而不进行回溯。

4.2 Tool Dispatch and Streaming Execution

4.2 工具分发与流式执行

When the model response contains tool_use blocks, the system chooses between two execution paths. The primary path uses StreamingToolExecutor, which begins executing tools as they stream in from the model response, reducing latency for multi-tool responses. The fallback path uses runTools() in toolOrchestration.ts, which iterates over partitions produced by partitionToolCalls(). Both paths classify tools as concurrent-safe or exclusive. Read-only operations can execute in parallel, while state-modifying operations like shell commands are serialized.

当模型响应包含 tool_use 块时，系统会在两条执行路径之间进行选择。主要路径使用 StreamingToolExecutor，它在工具从模型响应中流式传回时就立即开始执行它们，从而降低了多工具响应的延迟。备用路径使用 toolOrchestration.ts 中的 runTools()，它对 partitionToolCalls() 产生的分区进行迭代。两条路径都会将工具分类为并发安全（concurrent-safe）或排他性（exclusive）。只读操作可以并行执行，而像 Shell 命令这样修改状态的操作则被串行化。

The StreamingToolExecutor (StreamingToolExecutor.ts) manages concurrent execution with two coordination mechanisms:

StreamingToolExecutor.ts 通过两种协调机制来管理并发执行：

  • Sibling abort controller. Fires when any Bash tool errors, immediately terminating other in-flight subprocesses rather than letting them run to completion.

    兄弟中止控制器（Sibling abort controller）：当任何 Bash 工具报错时触发，立即终止其他正在执行的子进程，而不是让它们运行到结束。

  • Progress-available signal. Wakes up the getRemainingResults() consumer when new output is ready.

    进度可用信号（Progress-available signal）：当新输出准备就绪时，唤醒 getRemainingResults() 消费者。

Results are buffered and emitted in the order tools were received, so output order stays the same even when tools run in parallel. This is important because the model expects tool results in the same order as its tool-use requests. This concurrent-read, serial-write execution model occupies a middle ground between fully serial dispatch and more aggressive speculative approaches such as PASTE (Sui et al., 2026), which speculatively pre-executes predicted future tool calls while the model is still generating, hiding tool latency through speculation.

结果会被缓冲并按照接收到工具的顺序发出，因此即使工具并行运行，输出顺序也保持不变。这非常重要，因为模型期望工具结果的顺序与其工具使用请求的顺序一致。这种“并发读、串行写”的执行模型介于完全串行分发与更激进的推测性方法（例如 PASTE（Sui et al., 2026））之间。PASTE 在模型仍在生成时，就推测性地预执行预测的未来工具调用，通过推测来隐藏工具延迟。

The tool result collection phase iterates over updates from either the streaming executor or the synchronous runTools() generator. Each update may carry a tool result, an attachment, or a progress event. A special check detects hook_stopped_continuation attachments: if a PostToolUse hook signals that the turn should not continue, a shouldPreventContinuation flag is set. Results are normalized for the Anthropic API via normalizeMessagesForAPI(), filtering to keep only user-type messages.

工具结果收集阶段会对来自流式执行器或同步 runTools() 生成器的更新进行迭代。每个更新可能携带工具结果、附件或进度事件。一项特殊的检查用于检测 hook_stopped_continuation 附件：如果 PostToolUse 钩子发出信号表示该轮次不应继续，则会设置 shouldPreventContinuation 标志。结果通过 normalizeMessagesForAPI() 针对 Anthropic API 进行规范化，过滤以仅保留用户类型的消息。

4.3 Pre-Model Context Shapers

4.3 模型前置上下文整形器

Five context shapers execute sequentially in query.ts before every model call, each operating on the messagesForQuery array. The five shapers run in sequence, with earlier steps applying lighter reductions before later steps apply broader compaction.

在每次模型调用之前，五个上下文整形器会在 query.ts 中按顺序执行，每个整形器都对 messagesForQuery 数组进行操作。这五个整形器依次运行，较早的步骤先进行较轻量级的削减，随后较晚的步骤再应用更大范围的压缩。

Budget reduction. (applyToolResultBudget()). Enforces per-message size limits on tool results, replacing oversized outputs with content references. Exempt tools (those where maxResultSizeChars is not finite) retain their full output. Content replacements are persisted for agent and session query sources to enable reconstruction on resume. Budget reduction runs before microcompact because microcompact operates purely by tool_use_id and never inspects content; the two compose cleanly.

预算削减（applyToolResultBudget()）。对工具结果强制执行单条消息大小限制，将超大输出替换为内容引用。豁免工具（即 maxResultSizeChars 为无穷大的工具）保留其完整输出。内容替换会被持久化到智能体和会话查询源中，以便在恢复会话时进行重构。预算削减在微压缩之前运行，因为微压缩纯粹依据 tool_use_id 运行，绝不检查内容；两者可以干净地组合。

Snip. (snipCompactIfNeeded(), gated by HISTORY_SNIP). A lightweight trim that removes older history segments, returning {messages, tokensFreed, boundaryMessage}. The snipTokensFreed value is plumbed to auto-compact because the main token counter derives context size from the usage field on the most recent assistant message, and that message survives snip with its pre-snip input_tokens still attached; snip’s savings are therefore invisible to the counter unless passed through explicitly.

裁剪（snipCompactIfNeeded()，受 HISTORY_SNIP 门控）。一种轻量级的修剪，用于移除较旧的历史片段，并返回 {messages, tokensFreed, boundaryMessage}。snipTokensFreed 的值会被传递给自动压缩，因为主标元（token）计数器是从最近一次助手消息的 usage 字段派生上下文大小的，而该消息在裁剪后依然存在，且仍附带有裁剪前的 input_tokens；因此，除非显式传递，否则裁剪节省的开销对计数器是不可见的。

Microcompact. Fine-grained compression that always runs a time-based path and optionally a cache-aware path (gated by CACHED_MICROCOMPACT). When the cached path is enabled, boundary messages are deferred until after the API response so they can use actual cache_deleted_input_tokens rather than estimates. Returns {messages, compactionInfo} where compactionInfo may include pendingCacheEdits.

微压缩（Microcompact）。细粒度的压缩，总是运行一条基于时间的路径，并可选地运行一条感知缓存的路径（受 CACHED_MICROCOMPACT 门控）。当启用缓存路径时，边界消息会被推迟到 API 响应之后，以便它们可以使用实际的 cache_deleted_input_tokens 而不是估计值。返回 {messages, compactionInfo}，其中 compactionInfo 可能包含 pendingCacheEdits。

Context collapse. Gated by CONTEXT_COLLAPSE. A read-time projection over the conversation history. The source comments explain: “Nothing is yielded; the collapsed view is a read-time projection over the REPL’s full history. Summary messages live in the collapse store, not the REPL array. This is what makes collapses persist across turns.” Unlike the other shapers, context collapse does not mutate the REPL’s stored history; it replaces the messagesForQuery array with a projected view via applyCollapsesIfNeeded(), so the model sees the collapsed version while the full history remains available for reconstruction.

上下文坍缩（Context collapse）。受 CONTEXT_COLLAPSE 门控。对话历史记录上的一种读时投影（read-time projection）。源码注释解释道：“不产出（yield）任何内容；坍缩视图是 REPL 完整历史记录上的读时投影。摘要消息存在于坍缩存储中，而不是 REPL 数组中。这就是使坍缩能够跨轮次持久存在的原因。”与其他整形器不同，上下文坍缩不会改变 REPL 存储的历史记录；它通过 applyCollapsesIfNeeded() 将 messagesForQuery 数组替换为一个投影视图，从而使模型看到坍缩后的版本，而完整的历史记录仍可用于重构。

Auto-compact. The fifth shaper, triggering a full model-generated summary via compactConversation() in compact.ts. This function executes PreCompact hooks, creates a summary request using getCompactPrompt(), and calls the model to produce a compressed summary. The result feeds into buildPostCompactMessages() (compact.ts). Auto-compact fires only when the context still exceeds the pressure threshold after all four previous shapers have run.

自动压缩（Auto-compact）。第五个整形器，通过 compact.ts 中的 compactConversation() 触发由模型生成的完整摘要。该函数执行 PreCompact 钩子，使用 getCompactPrompt() 创建摘要请求，并调用模型以生成压缩摘要。结果会被输入到 buildPostCompactMessages()（compact.ts）中。只有在前面四个整形器运行后上下文仍然超过压力阈值时，自动压缩才会触发。

4.4 Recovery Mechanisms

4.4 恢复机制

The query loop implements several recovery mechanisms for edge cases:

查询循环针对边缘情况实现了几种恢复机制：

  • Max output tokens escalation: When the response hits the output token cap, the system can retry with an escalated limit, subject to a GrowthBook flag and the absence of an existing override or environment-variable cap. Up to three recovery attempts are allowed per turn (MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3).

    最大输出标元阶梯式提升（Max output tokens escalation）：当响应达到输出标元上限时，系统可以尝试提升上限并重试，这取决于 GrowthBook 标志以及是否存在现有的覆盖设置或环境变量上限。每轮最多允许尝试三次恢复（MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3）。

  • Reactive compaction (gated by REACTIVE_COMPACT): When the context is near capacity, reactive compact summarizes just enough to free space. The hasAttemptedReactiveCompact flag ensures this fires at most once per turn.

    反应式压缩（受 REACTIVE_COMPACT 门控）：当上下文接近容量极限时，反应式压缩会恰到好处地生成摘要以释放空间。hasAttemptedReactiveCompact 标志确保该操作每轮最多触发一次。

  • Prompt-too-long handling: If the API returns a prompt_too_long error, the loop first attempts context-collapse overflow recovery and reactive compaction. Only after these fail does it terminate with reason: ’prompt_too_long’.

    提示词过长处理：如果 API 返回 prompt_too_long 错误，循环首先会尝试上下文坍缩溢出恢复和反应式压缩。只有在这些尝试都失败后，循环才会终止，原因为：‘prompt_too_long’。

  • Streaming fallback: The onStreamingFallback callback handles streaming API issues, allowing the loop to retry with a different strategy.

    流式传输回退：onStreamingFallback 回调用于处理流式 API 问题，允许循环使用不同的策略进行重试。

  • Fallback model: The fallbackModel parameter enables switching to an alternative model if the primary model fails.

    备用模型：fallbackModel 参数允许在主模型失败时切换到备用模型。

4.5 Stop Conditions

4.5 停止条件

Multiple conditions can terminate the loop:

多种条件都可以终止该循环：

  1. No tool use: The model produces only text content (the primary stop condition).

    无工具使用：模型仅产生文本内容（主要停止条件）。

  2. Max turns: The configurable maxTurns limit is reached.

    最大轮次：达到了可配置的 maxTurns 限制。

  3. Context overflow: The API returns prompt_too_long.

    上下文溢出：API 返回 prompt_too_long。

  4. Hook intervention: A PostToolUse hook sets hook_stopped_continuation.

    钩子干预：PostToolUse 钩子设置了 hook_stopped_continuation。

  5. Explicit abort: The abortController signal fires.

    显式中止：abortController 信号触发。

The turn pipeline determines how tool requests are orchestrated and recovered. The next section examines the gate that determines whether each request is permitted to execute at all.

轮次流水线决定了工具请求是如何被编排和恢复的。下一节将检查决定每个请求究竟是否被允许执行的闸门。

5 Tool Authorization and Control Boundaries

5 工具授权与控制边界

Production coding agents adopt different safety architectures: layered policy enforcement, OS-level sandboxing, or version-control-based rollback. Claude Code combines the first two, implementing four design principles from Table 1: deny-first with human escalation, graduated trust spectrum, defense in depth with layered mechanisms, and reversibility-weighted risk assessment.

生产级编码智能体采用了不同的安全架构：分层策略执行、操作系统级沙箱，或基于版本控制的回滚。Claude Code 结合了前两者，实现了表 1 中的四项设计原则：默认拒绝与人类升级确认、渐进式信任光谱、具备分层机制的纵深防御，以及基于可逆性的加权风险评估。

When Claude decides to execute a tool (for example, running npm test via BashTool to reproduce the auth test failure), the request enters the permission pipeline shown in Figure 4. Every tool invocation passes through the permission system, and the default behavior is to deny or ask rather than allow silently.

当 Claude 决定执行某个工具时（例如，通过 BashTool 运行 npm test 以复现 auth 测试失败），该请求会进入图 4 所示的权限流水线。每次工具调用都会通过权限系统，其默认行为是拒绝（deny）或询问（ask），而不是默默允许（allow）。

This default is motivated by a documented behavioral pattern: Anthropic’s auto-mode analysis (Hughes, 2026) found that users approve approximately 93% of permission prompts, indicating that approval fatigue renders interactive confirmation behaviorally unreliable as a sole safety mechanism. Because users habitually approve without careful review, the system must maintain safety independently of human vigilance.

这种默认设置源于一个已被记录的行为模式：Anthropic 的自动模式分析（Hughes, 2026）发现，用户批准了大约 93% 的权限提示，这表明审批疲劳使得交互式确认在行为习惯上无法单独作为可靠的安全机制。由于用户习惯性地在未仔细审阅的情况下予以批准，系统必须保持独立于人类警惕性的安全性。

This motivates the architectural commitment to deny-first evaluation, blanket-deny pre-filtering, and sandboxing as independent layers that operate regardless of user attentiveness.

这就促成了架构上的承诺，即将默认拒绝评估、通盘拒绝预过滤以及沙箱化作为独立的层，无论用户是否专注，这些层都会运行。

5.1 Permission Modes and Rule Evaluation

5.1 权限模式与规则评估

Seven permission modes exist across the type definitions (5 external modes at types/permissions.ts; auto added conditionally; bubble in the type union):

在类型定义中存在七种权限模式（types/permissions.ts 中有 5 种外部模式；满足条件时会加入 auto 模式；bubble 模式存在于类型联合体中）：

  1. plan: The model must create a plan; execution proceeds only after user approval.

    plan（规划）：模型必须创建一个规划；只有在用户批准后才能继续执行。

  2. default: Standard interactive use. Most operations require user approval.

    default（默认）：标准的交互式使用。大多数操作都需要用户批准。

  3. acceptEdits: Edits within the working directory and certain filesystem shell commands (mkdir, rmdir, touch, rm, mv, cp, sed) are auto-approved; other shell commands require approval.

    acceptEdits（接受编辑）：工作目录内的编辑以及特定的文件系统 Shell 命令（mkdir, rmdir, touch, rm, mv, cp, sed）会被自动批准；其他 Shell 命令则需要批准。

  4. auto: An ML-based classifier evaluates requests that do not pass fast-path checks (gated by TRANSCRIPT_CLASSIFIER).

    auto（自动）：一个基于机器学习的分类器会对未通过快速路径（fast-path）检查的请求进行评估（受 TRANSCRIPT_CLASSIFIER 门控）。

  5. dontAsk: No prompting, but deny rules are still enforced.

    dontAsk（不询问）：不进行提示，但依然执行拒绝规则。

  6. bypassPermissions: Skips most permission prompts, but safety-critical checks and bypass-immune rules still apply.

    bypassPermissions（绕过权限）：跳过大多数权限提示，但关键安全的检查和免疫绕过的规则依然适用。

  7. bubble: Internal-only mode for subagent permission escalation to the parent terminal.

    bubble（气泡/冒泡）：内部专用模式，用于子智能体向父终端进行权限升级。

The five externally visible modes (acceptEdits, bypassPermissions, default, dontAsk, plan) are defined in the EXTERNAL_PERMISSION_MODES array. The auto mode is conditionally included only when the TRANSCRIPT_CLASSIFIER feature flag is active. The bubble mode exists in the type union but not in either mode array; it is used internally for subagent permission escalation (Section 8).

这五个外部可见的模式（acceptEdits, bypassPermissions, default, dontAsk, plan）定义在 EXTERNAL_PERMISSION_MODES 数组中。只有当 TRANSCRIPT_CLASSIFIER 特性标志处于激活状态时，才会条件式地包含 auto 模式。bubble 模式存在于类型联合体（type union）中，但不存在于任何一个模式数组中；它在内部用于子智能体的权限升级（第 8 节）。

Permission rules are evaluated in deny-first order (permissions.ts). The toolMatchesRule() function checks deny rules first: a deny rule always takes precedence over an allow rule, even when the allow rule is more specific. A broad deny (“deny all shell commands”) cannot be overridden by a narrow allow (“allow npm test”).

权限规则按照默认拒绝的顺序进行评估（permissions.ts）。toolMatchesRule() 函数首先检查拒绝规则：拒绝规则总是优先于允许规则，即使允许规则更加具体。宽泛的拒绝（“拒绝所有 Shell 命令”）不能被狭义的允许（“允许 npm test”）所覆盖。

The rule system supports tool-level matching (by tool name) and content-level matching (matching specific tool input patterns, such as Bash(prefix:npm)).

该规则系统支持工具级的匹配（通过工具名称）和内容级的匹配（匹配特定的工具输入模式，例如 Bash(prefix:npm)）。

The seven modes span a graduated autonomy spectrum, from plan (user approves all plans before execution) through default and acceptEdits to bypassPermissions (minimal prompting). This gradient reflects a recurring design tension: as autonomy increases, the system must shift from interactive approval to automated safety checks.

这七种模式跨越了一个渐进式的自主权光谱，从 plan（用户在执行前批准所有规划），通过 default 和 acceptEdits，再到 bypassPermissions（极少提示）。这种梯度反映了一个反复出现的设计张力：随着自主权的增加，系统必须从交互式批准转向自动化安全检查。

Other agent systems resolve this tension differently. SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b) use Docker container isolation, sandboxing the agent’s entire execution environment rather than evaluating individual tool invocations. Aider (Gauthier, 2024) relies on Git as a safety net, making all changes reversible through version control.

其他智能体系统以不同的方式解决这种张力。SWE-Agent 和 OpenHands（Yang et al., 2024; Wang et al., 2024b）使用 Docker 容器隔离，将智能体的整个执行环境沙箱化，而不是评估单个工具的调用。Aider（Gauthier, 2024）依赖 Git 作为安全网，通过版本控制使所有更改都可逆。

Claude Code’s approach layers multiple policy-enforcement mechanisms on top of optional container sandboxing, trading simplicity for fine-grained control over individual actions.

Claude Code 的方法则是在可选的容器沙箱之上叠加了多个策略执行机制，牺牲了简单性以换取对单个动作的细粒度控制。

5.2 The Authorization Pipeline

5.2 授权流水线

The full authorization pipeline proceeds through several stages:

完整的授权流水线包含以下几个阶段：

Pre-filtering. Before any tool request reaches runtime evaluation, filterToolsByDenyRules() (tools.ts) strips blanket-denied tools from the model’s view entirely at tool pool assembly time.

预过滤。在任何工具请求进入运行时评估之前，tools.ts 中的 filterToolsByDenyRules() 会在工具池组装阶段，将通盘拒绝（blanket-denied）的工具完全从模型的视野中剥离。

The documentation states: “Uses the same matcher as the runtime permission check, so MCP server-prefix rules like mcp__server strip all tools from that server before the model sees them.” This prevents the model from attempting to invoke forbidden tools, so the model does not waste calls on them.

文档指出：“使用与运行时权限检查相同的匹配器，因此像 mcp__server 这样的 MCP 服务器前缀规则会在模型看到它们之前，剥离该服务器的所有工具。”这可以防止模型尝试调用被禁止的工具，从而避免模型在其上浪费调用次数。

PreToolUse hook. Registered hooks fire as part of the permission pipeline. A PreToolUse hook can return a permissionDecision to deny or ask, or an updatedInput that modifies the tool’s input parameters (types/hooks.ts).

PreToolUse 钩子。已注册的钩子作为权限流水线的一部分触发。PreToolUse 钩子可以返回一个拒绝（deny）或询问（ask）的 permissionDecision（权限决策），或者返回一个修改工具输入参数的 updatedInput（types/hooks.ts）。

这段话描述的是 Claude Code 内部的一套“多重安全护栏（Multi-layered Guardrails）”和“异步并发控制”机制。

简单来说，它的核心意思是：即使某个前端拦截或局部判断（钩子）觉得这个操作“没问题（allow）”，系统也不会立刻执行，后面还有更严格的“终审判决”和用户确认。

我们可以把这段话拆解为两部分来理解：权限的“非充分条件”逻辑，以及两种不同运行模式下的执行顺序。

  1. 核心逻辑：允许（Allow）只是拿到了入场券，不是免死金牌

    “钩子返回允许（allow）并不会绕过后续基于规则的拒绝或安全检查。”

在安全架构设计中，这叫“默认拒绝（Deny-by-default）”与“最小特权原则”。

  • 钩子（Hook，比如 useCanUseTool）：是代码中用来快速判断模型当前能不能用某个工具的一个前置函数。
  • 逻辑关系：如果钩子返回了 deny（拒绝），那这个动作直接被卡死；但如果钩子返回了 allow（允许），系统并不会盲目信任它，而是继续把它送去过更严格的安检——比如 ML 分类器检查、敏感文件黑名单规则、危险 Shell 命令强匹配 等。只有全部通过，动作才合法。

  1. 交互式路径（Interactive Path）：先斩后奏的视觉优化

    “在交互式路径中，用户对话框会首先排队，而钩子异步运行”

当用户坐在电脑前看着 CLI 终端进行交互时，用户体验（响应速度）非常重要。

  • 机制：当大模型想要执行一个需要你同意的敏感操作时（比如修改文件），系统会同时做两件事：
    1. 立刻在终端上准备好弹出让用户确认的对话框（排队等待显示）。
    2. 在后台异步（同时）运行各种复杂的权限和安全钩子计算。
  • 目的：这样用户就不需要坐在屏幕前，干等着系统跑完一堆晦涩的安全规则代码才弹出提示。对话框先出来，能极大地降低用户的视觉卡顿感（降低感知延迟）。

  1. 后台智能体路径（Coordinator/Background Path）：安全至上的严格串行

    “在协调器（coordinator）以及类似的后台智能体路径中，则会等待自动化检查完成后再显示对话框。”

当 Claude Code 开启了子智能体（Subagent）或者在后台自动串联执行复杂任务时，情况发生了变化——这时候绝对的安全和减少不必要的人类打扰成了第一优先级。

  • 机制：系统会变成严格的“串行”模式。大模型提出请求 $\rightarrow$ 系统在后台把所有自动化安全检查（静态规则、分类器等）全部跑完 $\rightarrow$ 如果自动化检查直接判定违规，直接在后台拦截并报错 $\rightarrow$ 只有当自动化检查觉得可以放行、但又必须由人类兜底时，才会把对话框抛给用户。
  • 目的：避免后台的子智能体频繁弹出一些根本通不过自动化安检的“垃圾确认请求”来轰炸用户，做到“不该问的绝不问，必须问的洗干净了再问”。

总结

这段话揭示了 Claude Code 作为一个生产级 Agent 的工程成熟度。它不是简单地用一个 if-else 去判断安全，而是将“安全规则”与“用户交互”解耦成多道异步防线，并在前端（人机交互）和后端（智能体协同）采取了不同的并发策略，兼顾了体验与绝对的控制权。

Q：如果系统可能拒绝，那钩子还有什么用？

这是一个非常经典的安全架构（Security Architecture）设计问题。你可能会觉得：“既然通过了钩子也可能被后面的规则拒绝，那为什么不直接让后面的规则去查，要钩子干嘛？这不是多此一举吗？”

其实不然。在软件工程（特别是涉及大模型工具调用的高风险系统）中，这种设计不仅有用，而且是必不可少的。钩子的核心价值可以总结为以下四个方面：

  1. 快速战术截断（Fail-Fast）：降低整体计算成本

大模型的安全检查是非常“贵”且“慢”的。例如，调用一个 ML（机器学习）分类器去判断大模型的意图是否安全，需要消耗大量的算力、Token 和时间。

  • 如果没有钩子：大模型每发起一次工具调用，系统都要把请求发送给 ML 分类器和复杂的黑名单规则库进行全局扫描。
  • 有了钩子（作为前哨）：钩子可以执行非常轻量、局部的静态检查。
    • 举个例子：用户当前处于 read-only（只读权限模式）。大模型突然尝试调用 writeFile（写文件工具）。
    • 钩子的作用：它不需要去惊动复杂的 AI 分类器，直接在最前端返回 deny。这个动作在 1 毫秒内就被“卡死”了。

这就是 Fail-Fast（快速失败） 原则：用最低的成本，把最明显的违规行为挡在最外面。

  1. 纵深防御（Defense in Depth）：没有单点失效风险

在安全领域，没有任何单一的检查机制是 100% 完美的。

  • 规则库和黑名单可能会因为路径匹配的漏洞（比如绕过相对路径 ../）被攻破。
  • ML 分类器可能会遭遇提示词注入（Prompt Injection），被大模型“忽悠”从而给出错误的放行判断。

钩子和后续检查的关系，就像是机场安检：

  • 第一道防线（钩子）：在航站楼门口检查你的身份证和机票（合不合法，有没有基本权限）。
  • 第二道防线（ML/黑名单等）：让你过 X 光机和金属探测门（检查你有没有携带危险物品）。

机票合法（钩子 allow）不代表你没有携带违禁品（后续检查可能 deny）。 如果取消了第一道防线（钩子），直接把所有人推给 X 光机，安检口就会彻底瘫痪；而如果没有第二道防线，只看机票，恐怖分子就会混入飞机。两者的结合实现了纵深防御，任何一层失效，还有另一层兜底。

  1. 提供上下文组装与动态干预的“生命周期钩子”

在软件架构中，Hook（钩子）往往不仅仅是一个“布尔值过滤器”，它还承担着状态注入和阶段性干预的职责。

根据本文前述章节的逻辑，Claude Code 的钩子（如 types/hooks.ts 中的 27 个事件）是在工具执行的各个关键节点触发的：

  • 在检查时：钩子可以用来动态获取当前的最新状态（例如当前的临时环境变量、当前的路径作用域规则）。
  • 在放行时：如果钩子决定 allow，它同时可以在上下文中打上一个“前置标记”，告诉后续的系统：“这个调用是经过某某插件或某某模式授权的，可以进入下一阶段安检。”

它是一个事件驱动（Event-driven）的节点，让开发者可以自由地在“工具正要执行”、“工具执行中”、“工具执行完”等时间点插入自定义的代码。

  1. 权限的“权限范围划分（Scoping）”

在架构中，allow 往往不是一个简单的“可以”，而是一个“在什么范围内可以”。

  • 子智能体（Subagent）隔离：当主智能体派生出一个子智能体去执行某项子任务时，子智能体拥有自己独立的上下文。
  • 钩子的细粒度控制：钩子可以针对这个子智能体返回：“在当前这个子会话的隔离作用域内，我允许（allow）它看这个文件。”
  • 后续安全检查的全局控制：虽然钩子在局部作用域放行了，但全局的安全护栏（比如全局禁用某些敏感系统命令）依然有效。

总结

钩子返回 allow，并不是给大模型下发了“最终通行证”，而是代表“局部政治审查合格，可以送交中央进行技术复核”。

它存在的目的不是为了做“终审判决”，而是为了低成本地拦截明显违规、组装动态上下文、并作为纵深防御的第一道哨所。两者的配合，才让整个 Agent 系统的确定性工具箱（Harness）变得既高效又安全。

A hook allow does not bypass subsequent rule-based denies or safety checks. In the interactive path, the user dialog is queued first and hooks run asynchronously; coordinator and similar background-agent paths await automated checks before showing the dialog.

钩子返回允许（allow）并不会绕过后续基于规则的拒绝或安全检查。在交互式路径中，用户对话框会首先排队，而钩子异步运行；在协调器（coordinator）以及类似的后台智能体路径中，则会等待自动化检查完成后再显示对话框。

Rule evaluation. The deny-first rule engine evaluates the request. MCP tools are matched by their fully qualified mcp__server__tool name, and server-level rules match all tools from that server.

规则评估。默认拒绝的规则引擎对请求进行评估。MCP 工具通过其完全限定的 mcp__server__tool 名称进行匹配，而服务器级别的规则会匹配该服务器中的所有工具。

Permission handler. The handler in useCanUseTool.tsx branches into one of four paths based on runtime context:

权限处理程序。useCanUseTool.tsx 中的处理程序根据运行时上下文分支到以下四条路径之一：

  1. Coordinator: For multi-agent coordination mode. Attempts automated resolution (classifier, hooks, rules) before falling back to user interaction.

    Coordinator（协调器）：用于多智能体协调模式。在回退到用户交互之前，尝试自动化解析（分类器、钩子、规则）。

  2. Swarm worker: Handles worker agents in a multi-agent swarm with their own resolution logic.

    Swarm worker（集群工人）：处理多智能体集群中的工人智能体，它们拥有自己的解析逻辑。

  3. Speculative classifier: When BASH_CLASSIFIER is enabled and the tool is BashTool, a speculative classifier races a pre-started classification result against a timeout. If the classifier returns with high confidence, the tool is approved instantly without user interaction.

    Speculative classifier（推测性分类器）：当启用 BASH_CLASSIFIER 且工具为 BashTool 时，推测性分类器会将预先启动的分类结果与超时时间进行竞态比对。如果分类器以高置信度返回结果，工具将立即获得批准，无需用户交互。

  4. Interactive: The fallback path. Presents the standard user approval dialog through the terminal UI.

    Interactive（交互式）：回退路径。通过终端 UI 展现标准的用户审批对话框。

In coordinator and some background paths, automated resolution is attempted before user interaction. In the standard interactive path, the dialog can appear first while hooks or classifier checks continue in parallel.

在协调器和某些后台路径中，会在用户交互之前尝试自动化解析。在标准的交互式路径中，对话框可以首先出现，同时钩子或分类器检查并行持续进行。

When the classifier or a deny rule blocks an action, the system treats the denial as a routing signal rather than a hard stop: the model receives the denial reason, revises its approach, and attempts a safer alternative in the next loop iteration.

当分类器或拒绝规则阻断某项动作时，系统会将该拒绝视为一种路由信号，而不是硬性终止：模型会接收到拒绝原因，修正其方法，并在下一次循环迭代中尝试更安全的替代方案。

Claude Code 的安全层不仅是一个“绊脚石”，也是一个“指路牌”。它把大模型撞墙的失败经历，变成了一段有价值的提示词（Prompt），塞回给大模型让它自己‘开窍’，从而在安全不妥协的前提下，把任务继续做下去。

The PermissionDenied hook event (Section 6) enables external code to observe and respond to these denials programmatically. This recovery-oriented design means that permission enforcement shapes the agent’s behavior rather than simply halting it.

PermissionDenied 钩子事件（第 6 节）使外部代码能够以编程方式观察并响应这些拒绝。这种面向恢复的设计意味着权限执行能够塑造智能体的行为，而不是简单地终止它。

5.3 Auto-Mode Classifier and Hook Lifecycle

5.3 自动模式分类器与钩子生命周期

The auto-mode classifier (yoloClassifier.ts) participates in permission decisions when enabled. When TRANSCRIPT_CLASSIFIER is enabled, the classifier loads three prompt resources:

启用时，自动模式分类器（yoloClassifier.ts）会参与权限决策。当 TRANSCRIPT_CLASSIFIER 启用时，分类器会加载三种提示词资源：

  • A base system prompt.

    基础系统提示词。

  • An external permissions template.

    外部权限模板。

  • For Anthropic-internal users, a separate internal template.

    针对 Anthropic 内部用户，一个单独的内部模板。

The classifier evaluates the proposed tool invocation against the conversation transcript and the permission template, producing an allow, deny, or request for manual approval. The function isUsingExternalPermissions() checks USER_TYPE and a forceExternalPermissions config flag to select the appropriate template.

分类器结合对话历史文本和权限模板来评估提议的工具调用，从而产生允许（allow）、拒绝（deny）或请求人工审批（request for manual approval）的结果。isUsingExternalPermissions() 函数会检查 USER_TYPE 和 forceExternalPermissions 配置标志，以此选择合适的模板。

Of the 27 hook events defined in the source (coreTypes.ts), five participate directly in the permission flow, each with a specific Zod-validated output schema (types/hooks.ts):

在源码中定义的 27 种钩子事件（coreTypes.ts）中，有 5 种直接参与权限流，每种都带有经 Zod 验证的特定输出 Schema（types/hooks.ts）：

  • PreToolUse: Can return permissionDecision (deny or ask, but allow does not bypass subsequent checks), permissionDecisionReason, and updatedInput (modify parameters).

    PreToolUse：可以返回 permissionDecision（拒绝或询问，但允许并不会绕过后续检查）、permissionDecisionReason 以及 updatedInput（修改参数）。

  • PostToolUse: Can inject additionalContext and, for MCP tools, return updatedMCPToolOutput to modify results before they enter the context.

    PostToolUse：可以注入 additionalContext，并且对于 MCP 工具，可以返回 updatedMCPToolOutput 以便在结果进入上下文之前对其进行修改。

  • PostToolUseFailure: Can inject additionalContext for error-specific guidance.

    PostToolUseFailure：可以针对特定错误注入 additionalContext 以提供引导。

  • PermissionDenied: Can provide retry guidance after auto-mode denials.

    PermissionDenied：可以在自动模式拒绝后提供重试引导。

  • PermissionRequest: Can return a decision of allow or deny. In coordinator and similar paths, this can resolve before the user dialog. In the standard interactive path, it can also run alongside the dialog.

    PermissionRequest：可以返回允许或拒绝的决策。在协调器及类似路径中，这可以在用户对话框弹出前完成解析。在标准交互式路径中，它也可以与对话框并行运行。

For non-MCP tools, the tool_result is emitted before the PostToolUse hook fires. For MCP tools, the result is delayed until after post hooks have run, enabling updatedMCPToolOutput to take effect.

对于非 MCP 工具，tool_result 会在 PostToolUse 钩子触发之前发出。对于 MCP 工具，结果会推迟到后置（post）钩子运行之后再发出，从而使 updatedMCPToolOutput 能够生效。

5.4 Shell Sandboxing

5.4 Shell 沙箱化

Shell sandboxing provides an additional layer of protection for Bash and PowerShell commands (shouldUseSandbox.ts). The shouldUseSandbox() function checks whether sandboxing is globally enabled, whether the invocation has opted out, and whether the command matches any exclusion patterns.

Shell 沙箱化为 Bash 和 PowerShell 命令提供了额外的保护层（shouldUseSandbox.ts）。shouldUseSandbox() 函数会检查沙箱化是否全局启用、该调用是否选择了选择性退出（opt out），以及该命令是否匹配任何排除模式。

When active, the sandbox provides filesystem and network isolation independent of the application-level permission model. A command can be permission-approved but still sandboxed, or permission-denied and never reach the sandbox check. The two systems operate on different axes: authorization versus isolation.

激活时，沙箱提供独立于应用级权限模型的文件系统和网络隔离。一个命令可以获得权限批准但仍在沙箱内执行，或者因权限被拒绝而永远不会到达沙箱检查阶段。这两个系统在不同的轴线上运行：授权与隔离。

The layered safety architecture rests on an independence assumption: if one layer fails, others catch the violation. However, several layers share common performance constraints.

分层的安全架构依赖于一个独立性假设：如果一层失效，其他层会捕获违规行为。然而，有几个层共享了相同的性能约束。

Security researchers (Adversa.ai, 2026) have documented that commands with more than 50 subcommands fall back to a single generic approval prompt instead of running per-subcommand deny-rule checks, because per-subcommand parsing caused UI freezes.

安全研究人员（Adversa.ai, 2026）记录表明，包含 50 个以上子命令的命令会降级退回到单一的通用审批提示，而不是运行针对每个子命令的拒绝规则检查，因为针对每个子命令的解析曾导致 UI 冻结。

This example demonstrates that defense-in-depth can degrade when its layers share failure modes, a structural tension between safety and performance analyzed further in Section 11.3.

该示例表明，当防范纵深内的各层共享失效模式时，纵深防御性能可能会发生退化。安全性与性能之间的这种结构性张力将在第 11.3 节中做进一步分析。

权限流水线决定了工具请求是否执行。下一节将检查是什么决定了最初存在哪些工具：组装模型动作表面的扩展性架构。

对于编码智能体而言，一个周期性出现的设计问题是如何构建扩展表面：是采用单一的统一机制、少数几种专门的机制，还是采用具有不同上下文成本的分层堆栈。

此处的分析阐明了表 1 中的两项设计原则：可组合的多机制扩展性以及外部化的可编程策略。

Returning to the running example, once Claude is trying to repair auth.test.ts and the earlier npm test request has been mediated by the permission system (Section 5), the next question is what extension-enabled action surface is available for the repair.

回到运行示例，一旦 Claude 尝试修复 auth.test.ts 并且先前的 npm test 请求已经过权限系统的调解（第 5 节），接下来的问题就是有哪些通过扩展启用的动作表面可用于修复。

When a turn begins in Claude Code, the model sees not just built-in tools like BashTool and FileReadTool, but also database query tools from an MCP server, a custom lint skill from .claude/skills/, and tools contributed by an installed plugin.

当 Claude Code 的一个轮次开始时，模型不仅能看到 BashTool 和 FileReadTool 等内置工具，还能看到来自 MCP 服务器的数据库查询工具、来自 .claude/skills/ 的自定义 lint 技能，以及由已安装插件贡献的工具。

These arrive through four mechanisms that extend the agent at different points of the loop: MCP servers provide external tool integration, plugins package and distribute bundles of components, skills inject domain-specific instructions, and hooks intercept the tool execution lifecycle.

这些工具通过四种机制引入，并在循环的不同位置对智能体进行扩展：MCP 服务器提供外部工具集成，插件打包并分发组件包，技能注入特定领域的指令，而钩子则拦截工具执行的生命周期。

Anthropic’s documentation (Anthropic, 2026d) presents a broader view that includes CLAUDE.md (Section 7) and subagents (Section 8) alongside the four mechanisms analyzed here. We treat CLAUDE.md and subagents in their own sections because they operate in different subsystems (context construction and delegation, respectively), but the context-cost ordering is architecturally significant: it reveals how each extension point trades off expressiveness against the bounded context window.

Anthropic 的文档（Anthropic, 2026d）提出了一个更广泛的视角，将 CLAUDE.md（第 7 节）和子智能体（subagents，第 8 节）与本文分析的四种机制并列。我们之所以在各自独立的章节中探讨 CLAUDE.md 和子智能体，是因为它们运行在不同的子系统（分别属于上下文构建和任务委派子系统），但这种基于上下文成本（context-cost）的排序在架构上具有重要意义：它揭示了每个扩展点是如何在表达能力（expressiveness）与受限的上下文窗口（bounded context window）之间进行权衡的。

6.1 Four Extension Mechanisms

6.1 四种扩展机制

The mechanisms are implemented in distinct source directories (Figure 5) and serve different integration patterns:

这些机制在不同的源目录中实现（图 5），并服务于不同的集成模式：

MCP servers. The Model Context Protocol is the primary external tool integration path. MCP servers are configured from multiple scopes: project, user, local, and enterprise, with additional plugin and claude.ai servers merged at runtime (services/mcp/config.ts).

MCP 服务器。模型上下文协议（Model Context Protocol）是主要的外部工具集成路径。MCP 服务器可从多个作用域进行配置：项目、用户、本地和企业，并在运行时合并额外的插件和 claude.ai 服务器（services/mcp/config.ts）。

The MCP client (services/mcp/client.ts) supports multiple transport types: stdio, SSE, HTTP, WebSocket, SDK, plus IDE-specific variants (sse-ide, ws-ide) and an internal claudeai-proxy. Each connected server contributes tool definitions as MCPTool objects. Dedicated built-in tools ListMcpResourcesTool and ReadMcpResourceTool provide access to MCP resources.

MCP 客户端（services/mcp/client.ts）支持多种传输类型：stdio、SSE、HTTP、WebSocket、SDK，外加特定于 IDE 的变体（sse-ide、ws-ide）以及一个内部的 claudeai-proxy。每个连接的服务器都会贡献工具定义作为 MCPTool 对象。专用的内置工具 ListMcpResourcesTool 和 ReadMcpResourceTool 提供了对 MCP 资源的访问。

Plugins. Plugins serve a dual role: they are both a packaging format and a distribution mechanism. The PluginManifestSchema (utils/plugins/schemas.ts) accepts ten component types: commands, agents, skills, hooks, MCP servers, LSP servers, output styles, channels, settings, and user configuration.

插件。插件扮演着双重角色：它们既是一种打包格式，也是一种分发机制。PluginManifestSchema（utils/plugins/schemas.ts）接受十种组件类型：命令、智能体、技能、钩子、MCP 服务器、LSP 服务器、输出样式、通道、设置和用户配置。

The plugin loader (utils/plugins/pluginLoader.ts) validates manifests and routes each component to its respective registry: commands and skills surface through the SkillTool meta-tool, agents appear in definitions consumed by AgentTool, hooks merge into the hook registry, MCP and LSP servers fold into their standard configurations, and output styles modify response formatting.

插件加载器（utils/plugins/pluginLoader.ts）验证清单并将每个组件路由到其各自的注册表：命令和技能通过 SkillTool 元工具呈现，智能体出现在由 AgentTool 消耗的定义中，钩子合并到钩子注册表中，MCP 和 LSP 服务器折叠到它们各自的标准配置中，而输出样式则修改响应格式。

A single plugin package can therefore extend Claude Code across multiple component types simultaneously, making plugins the primary distribution vehicle for third-party extensions.

因此，单个插件包可以同时跨多种组件类型扩展 Claude Code，使插件成为第三方扩展的主要分发载体。

Skills. Each skill is defined by a SKILL.md file with YAML frontmatter. The parseSkillFrontmatterFields() function (loadSkillsDir.ts) parses 15+ fields including display name, description, allowed tools (granting the skill access to additional tools), argument hints, model overrides, execution context (’fork’ for isolated execution), associated agent definitions, effort levels, and shell configuration.

技能。每个技能都由一个带有 YAML 前言（frontmatter）的 SKILL.md 文件定义。parseSkillFrontmatterFields() 函数（loadSkillsDir.ts）解析 15 个以上的字段，包括显示名称、描述、允许的工具（授予技能访问额外工具的权限）、参数提示、模型覆盖、执行上下文（用于隔离执行的 ‘fork’）、关联的智能体定义、思考烈度级别和 Shell 配置。

Skills can define their own hooks, which register dynamically on invocation. Bundled skills are registered in-memory at startup. When invoked, the SkillTool meta-tool injects the skill’s instructions into the context.

技能可以定义它们自己的钩子，这些钩子在调用时动态注册。捆绑的技能在启动时在记忆中注册。当被调用时，SkillTool 元工具会将技能的指令注入到上下文中。

Hooks. The source code defines 27 hook events spanning tool authorization (PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied), session lifecycle (SessionStart, SessionEnd, Setup, Stop, StopFailure), user interaction (UserPromptSubmit, Elicitation, ElicitationResult), subagent coordination (SubagentStart, SubagentStop, TeammateIdle, TaskCreated, TaskCompleted), context management (PreCompact, PostCompact, InstructionsLoaded, ConfigChange), workspace events (CwdChanged, FileChanged, WorktreeCreate, WorktreeRemove), and notifications (coreTypes.ts, coreSchemas.ts).

钩子。源码定义了 27 种钩子事件，涵盖工具授权（PreToolUse、PostToolUse、PostToolUseFailure、PermissionRequest、PermissionDenied）、会话生命周期（SessionStart、SessionEnd、Setup、Stop、StopFailure）、用户交互（UserPromptSubmit、Elicitation、ElicitationResult）、子智能体协调（SubagentStart、SubagentStop、TeammateIdle、TaskCreated、TaskCompleted）、上下文管理（PreCompact、PostCompact、InstructionsLoaded、ConfigChange）、工作区事件（CwdChanged、FileChanged、WorktreeCreate、WorktreeRemove）以及通知（coreTypes.ts、coreSchemas.ts）。

Of these, 15 have event-specific output schemas with rich fields supporting permission decisions, context injection, input modification, MCP result transformation, and retry control (types/hooks.ts). Persisted hook commands configured via settings and plugins use four command types: shell commands (type: command), LLM prompt hooks (type: prompt), HTTP hooks (type: http), and agentic verifier hooks (type: agent) (schemas/hooks.ts).

其中，有 15 个具有特定于事件的输出 Schema，带有支持权限决策、上下文注入、输入修改、MCP 结果转换和重试控制的丰富字段（types/hooks.ts）。通过设置和插件配置的持久化钩子命令使用四种命令类型：Shell 命令（type: command）、LLM 提示词钩子（type: prompt）、HTTP 钩子（type: http）和智能体校验器钩子（type: agent）（schemas/hooks.ts）。

The runtime additionally supports non-persistable callback hooks (type: callback) used by the SDK and internal instrumentation (types/hooks.ts). Hook sources include settings.json, plugins, and managed policy at startup; skill hooks register dynamically on invocation (utils/hooks.ts). The five tool-authorization events are detailed in Section 5.3.

运行时还额外支持由 SDK 和内部检测使用的不可持久化回调钩子（type: callback）（types/hooks.ts）。钩子来源包括启动时的 settings.json、插件和托管策略；技能钩子在调用时动态注册（utils/hooks.ts）。这五个工具授权事件已在第 5.3 节中详细描述。

6.2 Tool Pool Assembly

6.2 工具池组装

The assembleToolPool() function at tools.ts is documented as “the single source of truth for combining built-in tools with MCP tools.” The assembly follows a five-step pipeline:

tools.ts 中的 assembleToolPool() 函数被记录为“合并内置工具与 MCP 工具的唯一事实来源”。其组装遵循一个五步流水线：

  1. Base tool enumeration. getAllBaseTools() (tools.ts) returns an array of up to 54 tools: 19 are always included (such as BashTool, FileReadTool, AgentTool, SkillTool), and 35 more are conditionally included based on feature flags, environment variables, and user type. Anthropic-internal users get additional internal tools. Worktree mode enables EnterWorktreeTool and ExitWorktreeTool. Agent swarms enable team tools. When embedded search tools are available in the Bun binary, dedicated GlobTool and GrepTool are omitted.

    基础工具枚举：getAllBaseTools()（tools.ts）返回一个包含多达 54 个工具的数组：其中 19 个始终包含在内（例如 BashTool、FileReadTool、AgentTool、SkillTool），另外 35 个根据特性标志、环境变量和用户类型条件式包含。Anthropic 内部用户会获得额外的内部工具。工作树（Worktree）模式会启用 EnterWorktreeTool 和 ExitWorktreeTool。智能体集群（Agent swarms）会启用团队工具。当 Bun 二进制文件中包含嵌入式搜索工具时，专用的 GlobTool 和 GrepTool 将被省略。

  2. Mode filtering. getTools() (tools.ts) applies mode-specific filtering. In CLAUDE_CODE_SIMPLE mode, only Bash, Read, and Edit are available (or REPLTool in the REPL branch; plus coordinator tools if applicable). Each tool’s isEnabled() method is called for runtime availability checks.

    模式过滤：getTools()（tools.ts）应用特定于模式的过滤。在 CLAUDE_CODE_SIMPLE 模式下，只有 Bash、Read 和 Edit 可用（或者在 REPL 分支中为 REPLTool；如果适用的话还包括协调器工具）。每个工具的 isEnabled() 方法都会被调用以进行运行时可用性检查。

  3. Deny rule pre-filtering. filterToolsByDenyRules() (tools.ts) strips blanket-denied tools from the model’s view before any call.

    拒绝规则预过滤：filterToolsByDenyRules()（tools.ts）在进行任何调用之前，将通盘拒绝的工具从模型的视野中剥离。

  4. MCP tool integration. MCP tools from appState.mcp.tools are filtered by deny rules and merged with built-in tools.

    MCP 工具集成：来自 appState.mcp.tools 的 MCP 工具经拒绝规则过滤后，与内置工具进行合并。

  5. Deduplication. Tools are deduplicated by name, with built-in tools taking precedence over MCP tools.

    去重：工具按名称进行去重，内置工具的优先级高于 MCP 工具。

Both REPL.tsx (via the useMergedTools hook) and AgentTool.tsx (when building the worker tool set) invoke this function, ensuring consistent assembly across all execution paths. At request time, deferred tools may be hidden from the model’s context until explicitly queried via ToolSearch (tools.ts).

REPL.tsx（通过 useMergedTools 钩子）和 AgentTool.tsx（在构建工人工具集时）都会调用此函数，从而确保所有执行路径上的组装保持一致。在请求时，被推迟的工具可能会隐匿于模型的上下文中，直到通过 ToolSearch（tools.ts）进行显式查询。

Agent-based extension (custom agent definitions via .claude/agents/*.md and plugin-contributed agents) is covered in Section 8, because agents differ fundamentally from the four mechanisms above: they create new, isolated context windows rather than extending the current one.

基于智能体的扩展（通过 .claude/agents/*.md 的自定义智能体定义以及插件贡献的智能体）将在第 8 节中讨论，因为智能体与上述四种机制有本质不同：它们会创建全新的、隔离的上下文窗口，而不是扩展当前的窗口。

6.3 Why Four Mechanisms?

6.3 为什么需要四种机制？

Given that each additional extension mechanism increases the surface area developers must learn, a natural question is why Claude Code uses four distinct mechanisms rather than consolidating into one or two.

鉴于每增加一种扩展机制都会增加开发人员必须学习的表面积，一个自然产生的问题是，为什么 Claude Code 使用四种不同的机制，而不是整合为一两种。

The answer lies in the observation that different kinds of extensibility impose different costs on the context window, and a single mechanism cannot span the full range from zero-context lifecycle hooks to schema-heavy tool servers without forcing unnecessary trade-offs on extension authors.

答案在于以下观察：不同类型的扩展性对上下文窗口施加了不同的成本，而单一的机制无法在不迫使扩展创作者做出无谓权衡的情况下，跨越从零上下文的生命周期钩子到包含繁重 Schema 的工具服务器的完整范围。

As Table 2 summarizes, each mechanism trades deployment complexity for a different kind of extensibility. MCP servers provide runtime tool integration (the model gains new callable tools) at the cost of server management overhead and context budget consumed by tool schemas. Skills shape how the agent thinks (not just what tools it has) at minimal context cost, since only frontmatter descriptions (not full content) stay in the prompt.

表 2 每种扩展机制所独特提供的能力。上下文成本（Context cost）是指该机制在激活时，会消耗多少受限的上下文窗口。

机制 独特能力 上下文成本 注入点
MCP 服务器 外部服务集成（多传输协议） 高（工具模式） model()：工具池
插件 多组件打包 + 分发 中（视具体情况而定） 所有三个点
技能 领域特异性指令 + 元工具调用 低（仅描述信息） assemble()：上下文注入
钩子 生命周期拦截 + 事件驱动的自动化 默认为 零 execute()：工具执行前/后

如表 2 所总结，每种机制都以部署复杂性换取了不同类型的扩展性。MCP 服务器提供运行时工具集成（模型获得新的可调用工具），代价是服务器管理开销以及工具 Schema 所消耗的上下文预算。技能以极低的上下文成本塑造智能体的思考方式（而不仅仅是它拥有什么工具），因为提示词中只保留前言描述（而非完整内容）。

Hooks provide cross-cutting lifecycle control (blocking, rewriting, or annotating tool calls) with no context footprint by default, though hooks can opt into injecting additional context. Plugins bundle any combination of the other three into distributable packages, acting as the packaging and distribution layer rather than a distinct runtime primitive.

钩子在默认情况下不占用上下文空间，提供横切生命周期的控制（阻断、重写或注释工具调用），不过钩子可以选择注入额外的上下文。插件将其他三者的任意组合打包到可分发的包中，充当打包和分发层，而不是一种独特的运行时原语。

The graduated context-cost ordering (zero for hooks, low for skills, medium for plugins, high for MCP) means that cheap extensions can scale widely without exhausting the context window, while expensive ones are reserved for cases that genuinely require new tool surfaces.

这种渐进式的上下文成本排序（钩子为零、技能较低、插件中等、MCP 较高）意味着廉价的扩展可以广泛扩展而不会耗尽上下文窗口，而昂贵的扩展则留给真正需要新工具表面的情况。

Some agent frameworks provide a single extension mechanism, typically a tool-only API where all customization arrives as additional callable tools. Others use two tiers, separating tools from configuration or instruction injection. Claude Code’s four-mechanism approach can accommodate a broader range of extension patterns, from zero-context event handlers to full external service integrations, but it increases the learning curve developers face when deciding which mechanism to use for a given integration task.

一些智能体框架提供单一的扩展机制，通常是仅限工具的 API，其中所有自定义都作为额外的可调用工具引入。其他框架则采用两层架构，将工具与配置或指令注入分离开来。Claude Code 的四机制方法可以容纳更广泛的扩展模式——从零上下文的事件处理程序到完整的外部服务集成，但这增加了开发人员在决定针对给定集成任务使用哪种机制时所面临的学习曲线。

7 Context Construction and Memory

7 上下文构建与记忆

How an agent manages its context window and persists user instructions is a central design choice, with different systems choosing between file-based transparency, database-backed retrieval, and opaque learned representations. The design choices here implement two principles from Table 1: context as scarce resource with progressive management and transparent file-based configuration and memory.

智能体如何管理其上下文窗口并持久化用户指令是一个核心的设计选择，不同的系统在基于文件的透明性、基于数据库的检索以及不透明的已学习表征之间进行选择。这里的设计选择实现了表 1 中的两项原则：视上下文为稀缺资源并进行渐进式管理，以及基于文件的透明配置与记忆。

By this point in the running example, the task has accumulated state: the original request, the npm test permission outcome, the tool pool assembled in Section 6, and any file reads or command outputs gathered so far. This section asks how that growing state is packed into Claude Code’s bounded context window before the next model call.

在运行示例的这一阶段，任务已经积累了状态：原始请求、npm test 权限评估结果、第 6 节中组装的工具池，以及目前为止收集到的所有文件读取或命令输出。本节将探究在下一次模型调用之前，如何将这些不断增长的状态打包进 Claude Code 有限的上下文窗口中。

Before the model is called, the agent loop assembles a context window from the tool pool (Section 6), CLAUDE.md files, auto memory, and conversation history. The following subsections cover the assembly order, the CLAUDE.md hierarchy, and the multi-step compaction pipeline.

在调用模型之前，智能体循环会从工具池（第 6 节）、CLAUDE.md 文件、自动记忆和对话历史记录中组装出一个上下文窗口。以下小节将涵盖组装顺序、CLAUDE.md 层级结构以及多步骤压缩流水线。

7.1 Context Window Assembly

7.1 上下文窗口组装

The context window (Figure 6) is assembled from the following sources, some at initial assembly and others injected late during the turn:

上下文窗口（图 6）由以下来源组装而成，其中部分在初始组装时加入，另一部分在轮次进行中延迟注入：

  1. System prompt, incorporating output style modifications and any --append-system-prompt flag content.

    系统提示词：结合了输出样式修改以及任何 --append-system-prompt 标志的内容。

  2. Environment info via getSystemContext() (context.ts): git status (skipped in remote mode or when git instructions are disabled) and an optional cache-breaking injection for internal builds (gated by BREAK_CACHE_COMMAND). Memoized once per session.

    环境信息：通过 getSystemContext()（context.ts）获取，包括 git status（在远程模式或禁用 git 指令时跳过）以及针对内部构建的可选的缓存破坏（cache-breaking）注入（受 BREAK_CACHE_COMMAND 门控）。每个会话仅记忆化（Memoized）一次。

  3. CLAUDE.md hierarchy via getUserContext() (context.ts): four-level instruction file hierarchy (Section 7.2). Also memoized.

    CLAUDE.md 层级结构：通过 getUserContext()（context.ts）获取的四级指令文件层级结构（第 7.2 节）。同样已被记忆化。

  4. Path-scoped rules: conditional and directory-matched rules that load lazily when the agent reads files in matching directories.

    路径作用域规则：条件式和目录匹配的规则，当智能体读取匹配目录中的文件时会延迟加载。

  5. Auto memory: contextually relevant memory entries prefetched asynchronously.

    自动记忆：异步预取的上下文相关记忆条目。

  6. Tool metadata: skill descriptions, MCP tool names, and deferred tool definitions (via ToolSearch, on demand).

    工具元数据：技能描述、MCP 工具名称以及延迟的工具定义（根据需求通过 ToolSearch 获取）。

  7. Conversation history: carried forward, subject to compaction.

    对话历史记录：向前传递，需进行压缩。

  8. Tool results: file reads, command outputs, subagent summaries.

    工具结果：文件读取、命令输出、子智能体摘要。

  9. Compact summaries: replacing older history segments.

    压缩摘要：替换较旧的历史片段。

The system prompt assembly at query.ts combines system context with the base prompt via asSystemPrompt(appendSystemContext(systemPrompt, systemContext))(). User context (CLAUDE.md and date) is prepended to the message array via prependUserContext().

query.ts 中的系统提示词组装通过 asSystemPrompt(appendSystemContext(systemPrompt, systemContext))() 将系统上下文与基础提示词合并。用户上下文（CLAUDE.md 和日期）通过 prependUserContext() 被前置到消息数组中。

This separation means CLAUDE.md content occupies a different structural position in the API request than the system prompt, potentially affecting model attention patterns.

这种分离意味着，在 API 请求中，CLAUDE.md 内容与系统提示词相比占据了不同的结构位置，这可能会影响模型的注意力分配模式。

Several context sources are injected late, after the main window is constructed: relevant-memory prefetch (query.ts), MCP instructions deltas (only new or changed server instructions), agent listing deltas, and background agent task notifications. The context window is therefore not static at assembly time but can grow during the turn.

有几种上下文来源是在主窗口构建完成之后延迟注入的：相关记忆预取（query.ts）、MCP 指令增量（仅包含新增或更改的服务器指令）、智能体列表增量以及后台智能体任务通知。因此，上下文窗口在组装时并非静态，而是可以在轮次进行中增长。

7.2 CLAUDE.md Hierarchy and Auto Memory

7.2 CLAUDE.md 层级结构与自动记忆

A design principle shapes the memory system: stored context should be inspectable and editable by the user. CLAUDE.md files are plain-text Markdown rather than structured configuration or opaque database entries.

一项设计原则塑造了记忆系统：存储的上下文应当是可供用户检查和编辑的。CLAUDE.md 文件是纯文本的 Markdown，而不是结构化的配置或不透明的数据库条目。

This transparency choice trades expressiveness for auditability: users can read, edit, version-control, and delete any instruction the agent sees (MindStudio Team, 2026). Alternative memory architectures illustrate the trade-off.

这种透明性的选择牺牲了表达能力以换取可审计性：用户可以阅读、编辑、进行版本控制以及删除智能体看到的任何指令（MindStudio Team, 2026）。可选的记忆架构阐明了这种权衡。

Retrieval-augmented approaches use embedding-based lookup to surface relevant prior context, gaining flexibility at the cost of inspectability: the user cannot easily see or edit what the retrieval system considers relevant. Database-backed memory offers structured querying but requires additional infrastructure and is opaque to version control.

检索增强（RAG）方法使用基于嵌入（embedding）的查找来显现相关的先前上下文，获得了灵活性，代价则是可检查性：用户无法轻松查看或编辑检索系统认为相关的内容。基于数据库的记忆提供了结构化查询，但需要额外的基础设施，且对版本控制是不透明的。

Claude Code’s file-based approach makes every instruction the agent sees directly readable, editable, and committable alongside the codebase. The system does not use embeddings or a vector similarity index for memory retrieval; instead it uses an LLM-based scan of memory-file headers to select up to five relevant files on demand, surfacing them at file granularity rather than entry granularity.

Claude Code 基于文件的方法使得智能体看到的每条指令都直接可读、可编辑，并可与代码库一起提交。系统不使用嵌入或向量相似度索引来进行记忆检索；相反，它使用基于 LLM 的记忆文件头扫描，根据需求选择最多五个相关文件，在文件粒度（而非条目粒度）上显现它们。

Embedding-based systems can retrieve individual entries more selectively, at the cost of inspectability and the infrastructure needed to maintain an index.

基于嵌入的系统可以更具选择性地检索单个条目，代价是可检查性以及维护索引所需的基础设施。

CLAUDE.md files follow a multi-level loading hierarchy. The source header (claudemd.ts) defines four memory types:

CLAUDE.md 文件遵循多级加载层级结构。源码头部（claudemd.ts）定义了四种记忆类型：

  1. Managed memory (e.g. /etc/claude-code/CLAUDE.md on Linux): OS-level policy for all users.

    托管记忆（例如 Linux 上的 /etc/claude-code/CLAUDE.md）：适用于所有用户的操作系统级策略。

  2. User memory (~/.claude/CLAUDE.md): private global instructions.

    用户记忆（~/.claude/CLAUDE.md）：私有的全局指令。

  3. Project memory (CLAUDE.md, .claude/CLAUDE.md, and .claude/rules/*.md in project roots): instructions checked into the codebase.

    项目记忆（项目根目录下的 CLAUDE.md、.claude/CLAUDE.md 以及 .claude/rules/*.md）：迁入代码库的指令。

  4. Local memory (CLAUDE.local.md in project roots): gitignored, for private project-specific instructions.

    本地记忆（项目根目录下的 CLAUDE.local.md）：被 git 忽略，用于私有的特定于项目的指令。

File discovery traverses from the current directory up to root, checking for all project and local memory files in each directory. Files closer to the current directory have higher priority (loaded later).

文件发现会从当前目录向上遍历到根目录，检查每个目录中的所有项目和本地记忆文件。距离当前目录越近的文件具有更高的优先级（越晚加载）。

Files load in “reverse order of priority”: later-loaded files receive more model attention. For root-to-CWD directories, unconditional rules from .claude/rules/*.md load eagerly at startup. For nested directories below CWD, even unconditional rules are loaded lazily when the agent reads files in matching directories. This means the model’s instruction set can evolve during a conversation as new parts of the codebase are explored.

文件按“优先级的反序”加载：越晚加载的文件获得越多的模型注意力。对于从根目录到当前工作目录（CWD）的目录，来自 .claude/rules/*.md 的无条件规则会在启动时被预先（eagerly）加载。对于 CWD 之下嵌套的目录，即使是无条件规则，也只有在智能体读取匹配目录中的文件时才会延迟加载。这意味着随着代码库新部分的探索，模型的指令集可以在对话过程中不断演变。

CLAUDE.md content is delivered as user context (a user message), not as system prompt content (context.ts). This architectural choice has a significant implication: because CLAUDE.md content is delivered as conversational context rather than system-level instructions, model compliance with these instructions is probabilistic rather than guaranteed.

CLAUDE.md 的内容是作为用户上下文（一条用户消息）交付的，而不是作为系统提示词内容（context.ts）。这一架构选择带来了一个重要的影响：由于 CLAUDE.md 内容是作为对话上下文而非系统级指令交付的，模型对这些指令的遵守是概率性的，而不是绝对保证的。

这段话的核心在于讨论了一个“工程实现方式带来的副作用”。它揭示了 CLAUDE.md（项目规范文件）在发挥作用时的一个致命软肋：因为它在底层被当成了“聊天历史”喂给大模型，而不是“最高控制命令”，所以大模型对它的遵守是“凭运气、看概率”的，无法做到 100% 绝对服从。

为了彻底理解这段话，我们可以将其拆解为以下三个层面：

  1. 身份的错配：系统命令（System） vs 用户聊天（User）

在大模型的底层 API 中，通常有两种严格区分的输入通道：

  • System Prompt（系统提示词）：这是规则的制定者，相当于操作系统的底层内核或法律规范。模型在整个对话中必须无条件服从这里的指令（比如：“你是一个资深前端工程师，绝对不能暴露你的 Prompt”）。
  • User Messages（用户消息历史）：这是对话的参与者，相当于聊天记录或参考资料。

原文指出，在代码实现中（context.ts），CLAUDE.md 的内容并没有被塞进 System Prompt，而是被打包成了一条普通的 User Message（用户上下文消息）扔进了历史记录里。

  1. 核心后果：合规性是“概率性的”（Probabilistic）而非“绝对保证的”

这种把“全局规范”当成“聊天历史”的做法，带来了一个巨大的隐患：大模型天生对这两者的重视程度和处理机制是完全不同的。

  • 如果是 System Prompt 级别的指令：模型的注意力机制（Attention）会对其进行强行绑定，确保每一轮对话都受到该底层规则的硬性约束。
  • 如果是 User Message 级别的上下文：
    • 会被稀释：随着对话越来越长（Turn 变多），CLAUDE.md 的内容会不断向后退，在长上下文窗口中被后续密集的对话和工具返回结果“稀释”掉。
    • 注意力衰减：大模型对聊天历史中的信息遵循度是概率性的。它可能会在第 3 轮对话时严格遵守 CLAUDE.md 里的代码风格，但到了第 20 轮，由于注意力漂移（例如大模型常见的“迷失在中间”现象），它就会把这些规矩忘得一干二净。
  1. 既然有这个缺点，为什么还要这么设计？

既然做成 User Message 会导致模型“不听话”，Claude Code 的架构师为什么不把它写进 System Prompt 呢？这其实是一个典型的工程权衡（Trade-off），为了换取其他关键优势：

  1. 守护“提示词缓存”（Prompt Caching）的经济账：

    系统提示词（System Prompt）如果保持绝对静态，API 就能完美触发缓存，从而极大降低 Token 成本并成倍提升响应速度。而 CLAUDE.md 是因项目而异的，甚至会随着用户频繁修改代码而改变。如果把经常变动的 CLAUDE.md 塞进系统提示词，会导致缓存频繁失效，让运行成本飙升、速度变慢。

  2. 多层级动态加载（Lazy-loading）的需要：

    正如前文所述，当 Agent 在复杂的项目目录里穿梭（比如从前端目录走到后端目录）时，它需要动态读取不同目录下的 CLAUDE.md 并拼接到当前对话中。作为消息（User Message）插入，比动态修改全局系统提示词要灵活得多。

总结

一句话理解：由于工程和成本上的权衡，Claude Code 妥协了安全性，把本该作为“法律（System）”的项目规范，变成了“参考资料（User Context）”塞给模型。这就导致了大模型对项目规范的遵守变成了“尽力而为”的概率事件，而不是铁律。

Permission rules evaluated in deny-first order (Section 5) provide the deterministic enforcement layer. This creates a deliberate separation between guidance (CLAUDE.md, probabilistic) and enforcement (permission rules, deterministic). The function calls setCachedClaudeMdContent() to cache the loaded content for the auto-mode classifier, to avoid an import cycle between the CLAUDE.md loader and the permission system.

按照默认拒绝顺序评估的权限规则（第 5 节）则提供了确定性的执行层。这在引导（CLAUDE.md，概率性）与执行（权限规则，确定性）之间创造了刻意的分离。该函数调用 setCachedClaudeMdContent() 来为自动模式分类器缓存加载的内容，以避免 CLAUDE.md 加载器与权限系统之间出现循环导入。

Memory files support an @include directive for modular instruction sets (processMemoryFile() at claudemd.ts). Syntax variants include @path, @./relative, @~/home, and @/absolute. The directive works in leaf text nodes only (not inside code blocks).

记忆文件支持用于模块化指令集的 @include 指令（claudemd.ts 中的 processMemoryFile()）。语法变体包括 @path、@./relative、@~/home 和 @/absolute。该指令仅在叶子文本节点中有效（不能在代码块内部使用）。

In the implementation, the including file is pushed first and included files are appended after it, circular references are prevented by tracking processed paths, and non-existent files are silently ignored.

在具体实现中，包含其他文件的文件会首先被推入，而被包含的文件会追加在其后；通过追踪已处理的路径来防止循环引用，并且不存在的文件会被默默忽略。

7.3 Compaction Pipeline

7.3 压缩流水线

The five-layer compaction pipeline (Section 4.3) implements the “context as bottleneck” principle through graduated compression (query.ts). Rather than a single strategy, Claude Code applies five layers in sequence, each with increasing aggressiveness (three are gated by feature flags; budget reduction is always active, while auto-compact is user-configurable).

五层压缩流水线（第 4.3 节）通过渐进式压缩（query.ts）实现了“视上下文为瓶颈”的原则。Claude Code 并非采用单一策略，而是依次应用五个层级，每一层都具有更高的激进程度（其中三层受特性标志门控；预算缩减始终处于激活状态，而自动压缩可由用户配置）。

This graduated approach contrasts with simpler alternatives: many agent frameworks use single-pass truncation (dropping the oldest messages) or a single summarization step. The graduated design reflects a lazy-degradation principle: apply the least disruptive compression first, escalating only when cheaper strategies prove insufficient.

这种渐进式的方法与更简单的替代方案形成对比：许多智能体框架使用单次截断（丢弃最旧的消息）或单个摘要步骤。这种渐进式设计反映了“延迟降级”（lazy-degradation）原则：首先应用颠覆性最小的压缩，只有在成本较低的策略证明不足时才逐步升级。

The cost of this approach is complexity. Five interacting compression layers, several gated by feature flags, create behavior that is difficult for users to fully predict. Auto-compact produces a visible summary in the transcript, and microcompact emits a boundary marker, but context collapse operates without user-visible output. Simpler single-pass approaches sacrifice information but are easier to reason about.

这种方法的代价是复杂性。五个相互作用的压缩层（其中几层受特性标志门控）产生的行为让用户难以完全预测。自动压缩会在转录文本中产生可见的摘要，微压缩（microcompact）会发出边界标记，但上下文崩溃（context collapse）在运行时不会产生用户可见的输出。更简单的单次处理方法虽然牺牲了信息，但更容易让人理解和推导。

  1. Budget reduction (always active): per-tool-result size limits.

    预算缩减（始终激活）：限制每个工具结果的大小。

  2. Snip (HISTORY_SNIP): lightweight older-history trimming.

    裁剪（HISTORY_SNIP）：对较旧历史记录进行轻量化修剪。

  3. Microcompact (CACHED_MICROCOMPACT): fine-grained cache-aware compression.

    微压缩（CACHED_MICROCOMPACT）：细粒度的、感知缓存的压缩。

  4. Context collapse (CONTEXT_COLLAPSE): read-time virtual projection over history.

    上下文崩溃（CONTEXT_COLLAPSE）：在读取时对历史记录进行虚拟投影。

  5. Auto-compact (enabled by default, can be disabled): full model-generated summary.

    自动压缩（默认启用，可禁用）：由模型生成的完整摘要。

The buildPostCompactMessages() function (compact.ts) returns the following compacted output structure: [boundaryMarker, …summaryMessages, …messagesToKeep, …attachments, …hookResults].

compact.ts 中的 buildPostCompactMessages() 函数返回以下压缩后的输出结构：[boundaryMarker, ...summaryMessages, ...messagesToKeep, ...attachments, ...hookResults]。

The boundary marker is annotated with preserved-segment metadata via annotateBoundaryWithPreservedSegment(), recording headUuid, anchorUuid, and tailUuid to enable read-time chain patching. This mostly-append design means compaction never modifies or deletes previously written transcript lines; it only appends new boundary and summary events.

边界标记通过 annotateBoundaryWithPreservedSegment() 被添加上保留片段的元数据，记录了 headUuid、anchorUuid 和 tailUuid，以便在读取时进行链条修补。这种“绝大部分情况下只追加”（mostly-append）的设计意味着压缩绝不会修改或删除先前已写入的转录行；它只会追加新的边界和摘要事件。

The compaction function compactConversation() (compact.ts) includes several design choices. Pre-compact hooks fire first, allowing hook-injected custom instructions.

压缩函数 compactConversation()（compact.ts）包含了多项设计选择。压缩前钩子（Pre-compact hooks）最先触发，允许注入通过钩子自定义的指令。

A GrowthBook feature flag controls whether the compaction path reuses the main conversation’s prompt cache (a code comment documents a January 2026 experiment: “false path is 98% cache miss, costs ~0.76% of fleet cache_creation”). After compaction, attachment builders re-announce runtime state (plans, skills, and async agents) from live app state, since compaction discards prior attachment messages but not the underlying state.

一个 GrowthBook 特性标志控制着压缩路径是否复用主对话的提示词缓存（一段代码注释记录了 2026 年 1 月的一项实验：“false 路径会导致 98% 的缓存未命中，消耗整个集群约 0.76% 的缓存创建资源”）。在压缩之后，附件构建器会从活跃的应用状态中重新声明运行时状态（计划、技能和异步智能体），因为压缩丢弃了先前的附件消息，但没有丢弃底层的状态。

Context isolation becomes more critical when the system delegates work to subagents, each operating in its own bounded context window.

当系统将工作委派给子智能体时，上下文隔离变得更加关键，因为每个子智能体都在自己受限的上下文窗口中运行。

Multi-agent orchestration is a key design dimension for coding agents, with choices spanning parent-child hierarchies, peer-based conversation frameworks (Wu et al., 2024), and graph-structured workflow engines (LangChain, Inc., 2024).

多智能体编排是编码智能体的一个核心设计维度，其选择涵盖父子层级结构、基于对等体的对话框架（Wu 等，2024）以及图结构的工作流引擎（LangChain, Inc.，2024）。

Q: 子智能体会自动继承父智能体的权限吗？还是需要再单独验证一次?

根据这篇论文以及 Claude Code 的底层设计，子智能体（Subagent）不会盲目、完全地自动继承父智能体的所有权限，但在实际运行中，它采用了一种“基于父级上下文克隆 + 动态二次验证”的混合控制机制。

具体可以从以下三个维度来理解它的权限传递逻辑：

  1. 初始状态：继承父级的“最大权限边界”

当父智能体派生（spawn）出一个子智能体时，子智能体在初始化时会继承父智能体当前所处的基础模式和环境变量。

  • 如果父智能体运行在 read-only（只读）模式下，子智能体绝对不可能越权获得写文件的权限。
  • 父智能体的权限上限，就是子智能体的绝对天花板。
  1. 运行时控制：必须单独通过“自动化检查”

虽然子智能体继承了权限边界，但当它在执行具体任务（例如调用某个工具）时，系统必须对它进行单独的、独立的验证。

正如前文在描述“协调器（coordinator）及后台智能体路径”时提到的：

​ “在协调器（coordinator）以及类似的后台智能体路径中，则会等待自动化检查完成后再显示对话框。”

这意味着：

  • 独立审计：子智能体每一次发起工具调用，它的请求都会被单独送入权限基础设施（如 utils/permissions/ 中的规则评估器和分类器）。系统会独立评估子智能体当前的操作是否安全，而不会因为“它是主智能体派生出来的”就给它直接发绿卡。
  • 串行卡点：子智能体作为后台运行的智能体，其自动化安全检查（黑名单、ML 分类器等）是严格串行、前置执行的。只有子智能体自己通过了这层单独验证，该操作才会进入下一个阶段。
  1. 人机交互：父智能体作为“拦截缓冲带”

子智能体与父智能体最大的区别在于它们与人类用户的距离：

  • 父智能体是直接面对用户的，所以可以采用“对话框排队、钩子异步运行”的方式来优化体验。
  • 子智能体运行在后台，为了防止它频繁弹出确认窗口“轰炸”人类，系统做了一个拦截机制。子智能体尝试危险操作时，系统先在后台进行独立验证。如果自动化验证觉得“风险较高但可以放行，但必须由人类兜底”时，这个请求会被推回给父智能体，由父智能体向用户弹出对话框。

总结

子智能体的权限逻辑可以总结为：“天花板自动继承，具体动作单独审计”。

它在宏观上受到父智能体权限范围的死死限制，但在微观上，它的每一个工具调用、每一条命令执行，都必须在后台单独过一遍自动化安全检查的硬关卡，绝不存在“父级放行，子级就可以为所欲为”的后门。

除内置类型外，用户还可以通过 .claude/agents/*.md 文件定义自定义子智能体，而插件则通过 loadPluginAgents.ts 贡献智能体定义。每个文件的 Markdown 正文充当该智能体的系统提示词，而 YAML 前言则指定配置字段，包括描述、工具（白名单）、禁用工具、模型、思考烈度（effort）、权限模式、MCP 服务器、钩子、最大轮次、技能、记忆作用域、后台标志以及隔离模式。

JSON-formatted agent definitions support the same fields plus prompt as an explicit field (loadAgentsDir.ts). This means a custom agent can be a fully configured, isolated sub-system with its own tools, model, permissions, hooks, memory scope, and isolation mode.

JSON 格式的智能体定义支持相同的字段，外加将 prompt 作为显式字段（loadAgentsDir.ts）。这意味着自定义智能体可以是一个完全配置且隔离的子系统，拥有属于自己的工具、模型、权限、钩子、记忆作用域和隔离模式。

AgentTool sits alongside SkillTool in the base tool pool as a meta-tool that dispatches to these definitions, but the two differ fundamentally: SkillTool injects instructions into the current context window, while AgentTool spawns a new, isolated one.

在基础工具池中，AgentTool 与 SkillTool 并列作为分发到这些定义的元工具（meta-tool），但两者有着本质的区别：SkillTool 将指令注入到当前的上下文窗口中，而 AgentTool 则会催生（spawn）一个新的、隔离的上下文窗口。

The tradeoff is that most subagent invocations require a self-contained prompt, because the default path does not inherit the parent’s conversation history (the fork-subagent path is an exception). Conversation-based frameworks that share full transcript histories avoid this cost but risk context explosion as the number of agents grows.

其权衡在于，由于默认路径不会继承父级的对话历史记录（fork 子智能体路径除外），大多数子智能体的调用都需要一个自包含的提示词。共享完整转录历史的基于对话的框架避免了这一成本，但随着智能体数量的增加，会面临上下文爆炸的风险。

8.2 Isolation Architecture

8.2 隔离架构

Subagent isolation supports multiple modes (AgentTool.tsx):

子智能体隔离支持多种模式（AgentTool.tsx）：

  • Worktree: Creates a temporary git worktree, giving the subagent its own copy of the repository to modify without affecting the parent’s working tree.

    Worktree（工作树）：创建一个临时的 git 工作树，赋予子智能体其专属的代码库副本以进行修改，而不会影响父级的工作树。

  • Remote (internal-only): Launches in a remote Claude Code Remote environment, always running in the background.

    Remote（远程，仅限内部）：在远程的 Claude Code Remote 环境中启动，始终在后台运行。

  • In-process (default): Shares the filesystem with the parent but operates in an isolated conversation context.

    In-process（进程内，默认）：与父级共享文件系统，但在隔离的对话上下文中运行。

The permission override logic for subagents (runAgent.ts) involves several specific rules. When a subagent defines a permissionMode, the override is applied unless the parent is already in bypassPermissions, acceptEdits, or auto mode, since those modes always take precedence because they represent explicit user decisions about the safety/autonomy trade-off.

子智能体的权限覆盖逻辑（runAgent.ts）涉及几条特定的规则。当子智能体定义了 permissionMode 时，除非父级已经处于 bypassPermissions、acceptEdits 或 auto 模式，否则将应用该覆盖。这是因为上述模式始终具有更高的优先级，它们代表了用户针对安全性与自主性权衡所做出的显式抉择。

For async agents, the system determines whether to avoid prompts through a cascade: explicit canShowPermissionPrompts first, then bubble mode (always show, since they escalate to the parent terminal), then the default (sync agents show prompts, async agents do not). Background agents that can show prompts set awaitAutomatedChecksBeforeDialog: true, ensuring the classifier and hooks resolve before interrupting the user.

对于异步智能体，系统通过一级级级联来确定是否避免提示：首先看显式的 canShowPermissionPrompts，其次是 bubble 模式（由于其会升级到父终端，因此始终显示），最后是默认值（同步智能体显示提示，异步智能体不显示）。可以显示提示的后台智能体会设置 awaitAutomatedChecksBeforeDialog: true，从而确保在打扰用户之前先由分类器和钩子完成解析。

These isolation modes occupy different points in a design space. Container-based isolation (used by SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b)) provides stronger resource boundaries but requires container infrastructure. Context-only isolation (used by conversation-based frameworks like AutoGen (Wu et al., 2024)) shares the filesystem but separates conversation histories.

这些隔离模式占据了设计空间中的不同维度。基于容器的隔离（由 SWE-Agent 和 OpenHands 使用（Yang et al., 2024; Wang et al., 2024b））提供了更强的资源边界，但需要容器基础设施。仅限上下文的隔离（由 AutoGen 等基于对话的框架使用（Wu et al., 2024））共享文件系统，但分离了对话历史记录。

Claude Code’s worktree-based isolation provides filesystem-level separation with zero external dependencies, leveraging Git’s built-in mechanism rather than introducing container orchestration.

Claude Code 基于工作树的隔离则通过零外部依赖提供了文件系统级别的分离，它利用了 Git 的内置机制，而不是引入容器编排。

When allowedTools is explicitly provided to runAgent() (runAgent.ts), a two-tier permission scoping model applies. SDK-level permissions from --allowedTools are preserved: “explicit permissions from the SDK consumer that should apply to all agents.” But session-level rules are replaced with the subagent’s declared allowedTools. When allowedTools is not provided (the common AgentTool path), the parent’s session-level rules are inherited without replacement.

当向 runAgent()（runAgent.ts）显式提供 allowedTools 时，将应用双层权限作用域模型。来自 --allowedTools 的 SDK 级权限会被保留：“来自 SDK 消费者的显式权限，应当适用于所有智能体。”但会话级规则会被替换为子智能体声明的 allowedTools。当未提供 allowedTools 时（常见的 AgentTool 路径），则会继承父级的会话级规则而不进行替换。

8.3 Sidechain Transcripts

8.3 侧链转录文本

Each subagent writes its own transcript as a separate .jsonl file with a .meta.json metadata file (sessionStorage.ts, runAgent.ts). This sidechain design means subagent histories are preserved for debugging and auditing but do not inflate the parent’s session file.

每个子智能体都会将自己的转录文本写入一个单独的 .jsonl 文件中，并附带一个 .meta.json 元数据文件（sessionStorage.ts，runAgent.ts）。这种侧链设计意味着子智能体的历史记录被保留下来用于调试和审计，但不会使父级的会话文件膨胀。

Q： .jsonl 是什么文件？

.jsonl（全称 JSON Lines）是一种非正式的、但被广泛应用的文件格式。它实质上是 JSON 格式的一种变体，专门用来存储结构化的数据流或大型数据集。

简单来说：.jsonl 文件里的每一行，都是一个独立且合法的标准 JSON 对象。

── 核心特点 ──

如果用一个公式来表达，那就是：

$$\text{.jsonl} = \text{JSON} + \text{换行符 (Lines)}$$

我们可以通过它与传统 .json 文件的对比来看清它的结构：

  1. 传统 .json 文件

整个文件是一个巨大的 JSON 数组或对象，所有数据用逗号分隔，并被包裹在大括号或方括号里：

JSON

  1. .jsonl 文件

去掉最外层的方括号，去掉每行末尾的逗号，每一条数据独占一行：

Plaintext

── 为什么大模型和 AI 领域大量使用 .jsonl？ ──

在当前的大语言模型（LLM）开发、微调（Fine-tuning）和 Agent 训练中，.jsonl 几乎是行业标准格式（例如 OpenAI、Anthropic 的微调数据集）。这主要是因为以下工程优势：

  • 极其高效的内存管理（流式处理）：

    如果一个训练数据集有 10 GB，传统的 .json 文件要求你必须把这 10 GB 的内容一次性全部加载到内存中解析成一个大数组，这很容易导致内存崩溃（OOM）。

    而 .jsonl 允许程序一行一行地读取（Read line-by-line）。读一行，处理一行，释放一行，内存占用极小。

  • 完美的容错性（错误隔离）：

    在传统 .json 文件中，如果第 50 万行漏掉了一个逗号，整个文件就会因为解析失败（Parse Error）而彻底报废。

    但在 .jsonl 中，如果第 50 万行损坏了，程序只会跳过这一行，前后的内容依然可以正常读取。

  • 方便数据追加（Append-only）：

    在日志系统（Log）或智能体对话历史记录中，系统可以不断地向 .jsonl 文件末尾追加新的聊天记录，而不需要重写整个文件结构。

── 常见应用场景 ──

  1. AI 训练与微调数据集：

    每行代表一个训练样本（例如一个包含 system、user、assistant 的完整对话流）。

  2. 大型日志文件（Log Files）：

    后端服务器或 AI Agent 的每一步工具调用（Tool Call）和运行状态，按时间戳一行行写入。

  3. 网络爬虫数据存储：

    爬虫每抓取一条网页信息，就实时往文件里写入一行。

Only the subagent’s final response text and metadata return to the parent conversation context; the full subagent history never enters the parent’s context window, respecting the “context as bottleneck” principle.

只有子智能体的最终响应文本和元数据会返回到父级对话上下文中；完整的子智能体历史记录绝不会进入父级的上下文窗口，以此恪守“视上下文为瓶颈”的原则。

The runAgent() function accepts 21 parameters covering agent definition, prompts, permissions, tools, model settings, isolation, and callbacks.

runAgent() 函数接受 21 个参数，涵盖智能体定义、提示词、权限、工具、模型设置、隔离和回调。

The summary-only return model is a deliberate context-conservation choice: conversation-based frameworks that share full transcript histories between agents risk context explosion as the number of agents grows. Even isolated-context parallelism carries substantial cost.

仅返回摘要的模型是一项刻意的上下文节约选择：在智能体之间共享完整转录历史记录的基于对话的框架，随着智能体数量的增加会面临上下文爆炸的风险。即使是隔离上下文的并行性也会带来巨大的成本。

Claude Code’s agent teams consume approximately 7× the tokens of a standard session in plan mode (Anthropic, 2025b), which makes summary-only return more critical when subagents are also in isolated contexts.

Claude Code 的智能体团队在 plan 模式下消耗的标元大约是标准会话的 7 倍（Anthropic, 2025b），这使得当子智能体同样处于隔离上下文中时，仅返回摘要变得更加关键。

For multi-instance coordination in agent teams, the harness uses file locking rather than a message broker or distributed coordination service (Anthropic, 2025b). Tasks are claimed from a shared list via lock-file-based mutual exclusion, with lock files stored at predictable filesystem paths.

对于智能体团队中的多实例协调，运行框架使用的是文件锁，而不是消息代理（message broker）或分布式协调服务（Anthropic, 2025b）。任务通过基于锁文件的互斥机制从共享列表中认领，锁文件存储在可预测的文件系统路径中。

这段话描述的是大模型智能体团队（Agent Teams）在进行多实例协同（Multi-instance coordination）时，采用的一种非常极简、务实且符合本地开发场景的并发控制机制。

简单来说，它的核心意思是：当有多个 AI 智能体分工合作、同时处理任务时，它们不使用任何复杂的网络中间件（如消息队列或分布式协调服务），而是像传统单机软件一样，直接在硬盘上“抢占锁文件（Lock File）”来确保任务不被重复执行。

我们可以从以下三个层面来深度理解这种工程设计：

  1. 核心技术对比：文件锁（File Locking）vs 传统分布式中间件

在传统的互联网大并发架构中，如果多个服务器节点需要协同、抢占任务，通常会引入复杂的外部组件：

  • 消息经纪人（Message Broker）：如 RabbitMQ、Kafka，用来分发任务。
  • 分布式协调服务：如 Redis（红锁）、ZooKeeper、Consul，用来做分布式锁。

但 Claude Code 是一个主要运行在本地终端（CLI/本地开发环境）的工具。为了实现多个智能体实例之间的协同，它抛弃了上述所有复杂的网络依赖，直接使用了操作系统最底层的文件锁（File Locking）机制：

  • 共享任务列表：所有智能体都能看到同一个任务清单文件（可能是一个 .jsonl 或特定的任务目录）。
  • 锁文件互斥（Mutual Exclusion）：当智能体 A 想去执行“任务 1”时，它会立刻在硬盘的特定预测路径（Predictable path，比如 ~/.claude/locks/task_1.lock）下创建一个空文件并锁住它。
  • 抢占逻辑：智能体 B 紧接着也想做“任务 1”，发现这个 .lock 文件已经存在且被锁定了，它就知道“任务 1 已经被同事领走了”，于是自动跳过，去清单里看下一个任务。
  1. 这种设计的工程妙处（为什么这么做？）

这种“返璞归真”的设计在智能体系统的构建中非常高明，它带来了三个巨大的优势：

  • 零外部依赖（Zero Dependency）：用户在电脑上安装 Claude Code 只需要一行命令。如果系统需要依赖 Redis 或 ZooKeeper 来搞多智能体协同，用户就必须在本地配置复杂的运行环境，这会极大提高软件的使用门槛。
  • 利用本地文件系统的天然特性：现代操作系统（macOS, Linux, Windows）在底层都提供了极度稳定、原子化（Atomic）的文件锁定 API（如 flock 或 fs.lock）。这种单机或局域网内的并发控制，文件锁是性能最高、最不容易出错的方式。
  • 天然的离线友好与确定性：智能体在处理本地代码库时，所有的读写都在文件系统内发生。让任务调度也发生在本机文件系统内，整个系统的生命周期和状态是完全统一且可预测的。
  1. 架构的局限性（Trade-off）

当然，任何架构设计都是一种权衡（Trade-off）。使用文件锁也暴露了当前这套 Agent 团队的边界：

  • 它目前是“单机版”或“共享文件系统版”的团队协同：这意味着这些子智能体实例必须运行在同一台电脑上，或者挂载在同一个共享网络硬盘（NFS）上。
  • 无法直接跨网络扩展：如果未来要把 Agent 团队扩展到全球不同的云服务器上协同，这套文件锁机制就无法直接使用了，必须重构成真正的分布式架构。但对于目前定位在“本地 AI 软件工程师助手”的 Claude Code 来说，文件锁是完美的解法。

总结

一句话理解：Claude Code 在让多个 AI 智能体组队打怪时，没有用高大上的网络中间件，而是用最原始但最可靠的“抢地盘盖戳（创建锁文件）”的方式来分工，确保大家各司其职、互不干扰。

This trades throughput for two properties: zero-dependency deployment (no external infrastructure required) and full debuggability (any agent’s state can be inspected by reading plain-text JSON files).

这用吞吐量换取了两个特性：零依赖部署（不需要外部基础设施）和完全的可调试性（任何智能体的状态都可以通过阅读纯文本的 JSON 文件来检查）。

9 会话持久化与恢复

Session persistence in coding agents involves a design choice between append-only logs, structured databases, checkpoint-based snapshots, and stateless architectures, each with different trade-offs in auditability, query power, and deployment complexity.

编码智能体中的会话持久化涉及在仅追加日志（append-only logs）、结构化数据库、基于检查点的快照（checkpoint-based snapshots）以及无状态架构之间进行设计选择，每种选择在可审计性、查询能力和部署复杂度方面都有不同的权衡。

Claude Code’s persistence design implements the append-only durable state principle from Table 1. Session-scoped permissions live in memory only and are not serialized to the transcript, so resume rebuilds the permission context from CLI args and disk settings; requests the rebuilt context does not recognize fall back to deny-first prompting.

Claude Code 的持久化设计实现了表 1 中的“仅追加的可持久化状态”原则。会话作用域内的权限仅存在于记忆中，不会被序列化到转录文本中，因此恢复（resume）操作会通过命令行参数（CLI args）和磁盘设置重新构建权限上下文；对于重建后的上下文无法识别的请求，系统将回退到“默认拒绝”的提示确认模式。

三个持久化通道独立运行：

  1. Session transcripts: Conversation records including user, assistant, attachment, and system messages, plus compaction and other metadata events. Project-scoped, one file per session.

    会话转录文本：对话记录，包括用户、助手、附件和系统消息，外加压缩和其他元数据事件。属于项目作用域，每个会话一个文件。

  2. Global prompt history: User prompts only, stored in history.jsonl at the Claude configuration home directory (history.ts). The makeHistoryReader() generator yields entries in reverse order via readLinesReverse(), supporting Up-arrow and ctrl+r navigation.

    全局提示词历史：仅包含用户提示词，存储在 Claude 配置家目录下的 history.jsonl 中（history.ts）。makeHistoryReader() 生成器通过 readLinesReverse() 逆序输出条目，以支持上方向键和 ctrl+r 导航。

  3. Subagent sidechains: Separate .jsonl + .meta.json files per subagent (Section 8.3).

    子智能体侧链：每个子智能体拥有独立的 .jsonl + .meta.json 文件（第 8.3 节）。

Session transcripts store several kinds of events beyond simple messages, including compaction markers, file-history snapshots, attribution snapshots, and content-replacement records.

会话转录文本除了简单的消息外，还存储了多种事件，包括压缩标记、文件历史快照、归属（attribution）快照以及内容替换记录。

The append-only JSONL format is a deliberate choice favoring auditability and simplicity over query power. Every event is human-readable, version-controllable, and reconstructable without specialized tooling. Database-backed alternatives would enable richer queries over session history but introduce deployment dependencies and reduce transparency.

仅追加的 JSONL 格式是一项刻意的选择，它更倾向于可审计性和简单性，而非查询能力。每个事件都是人类可读的、可进行版本控制的，并且无需专门工具即可重新构建。基于数据库的替代方案可以对会话历史进行更丰富的查询，但会引入部署依赖并降低透明度。

The session identity system pairs sessionId with sessionProjectDir, set together during resume or branch. The transcript path must use the same project directory that was active when messages were written, to avoid hooks looking in the wrong directory.

会话标识系统将 sessionId 与 sessionProjectDir 配对，它们在恢复（resume）或创建分支（branch）时一起被设置。转录路径必须使用与写入消息时处于激活状态相同的项目目录，以避免钩子（hooks）查找错误的目录。

9.2 Resume, Fork, and Not Restoring Permissions

9.2 恢复、派生与不恢复权限

The --resume flag rebuilds the conversation by replaying the transcript (conversationRecovery.ts). Fork creates a new session from an existing one (commands/branch/branch.ts).

--resume 标志通过重放转录文本来重新构建对话（conversationRecovery.ts）。派生（Fork）操作则从现有会话中创建一个新会话（commands/branch/branch.ts）。

However, resume and fork do not restore session-scoped permissions; users must grant them again in the new session. This is a deliberate safety-conservative design choice: sessions are treated as isolated trust domains. Restoring previously granted permissions on resume would create a convenience benefit but risk carrying stale trust decisions into a changed context.

然而，恢复和派生并不会恢复会话作用域内的权限；用户必须在新会话中重新进行授权。这是一项刻意的、在安全上持保守态度的设计选择：会话被视为隔离的信任域。在恢复时恢复先前授予的权限固然能带来便利，但有将陈旧的信任决策带入已发生变化的上下文中的风险。

The architecture opts for re-granting over implicit persistence, accepting user friction as the cost of maintaining the safety invariant that trust is always established in the current session.

该架构选择了重新授权，而非隐式持久化，宁愿接受用户操作上的摩擦，也要付出代价以维持这一安全不变性——即信任始终在当前会话中建立。

The compact_boundary marker is carefully designed to work with persistence. The annotateBoundaryWithPreservedSegment() function (compact.ts) records headUuid, anchorUuid, and tailUuid in the boundary event.

compact_boundary（压缩边界）标记经过精心设计，能够与持久化协同工作。compact.ts 中的 annotateBoundaryWithPreservedSegment() 函数在边界事件中记录 headUuid、anchorUuid 和 tailUuid。

These UUIDs enable the session loader to patch the message chain at read time: preserved messages keep their original parentUuids on disk, and the loader uses boundary metadata to link them correctly. This mostly-append design means compaction never modifies or deletes previously written transcript lines.

这些 UUID 使得会话加载器能够在读取时对消息链进行修补：保留的消息在磁盘上维持其原始的 parentUuids，而加载器则利用边界元数据将它们正确链接起来。这种“绝大部分情况下只追加”的设计意味着压缩绝不会修改或删除先前已写入的转录行。

The “checkpoints” in Claude Code are file-history checkpoints for --rewind-files, stored at ~/.claude/file-history//. These are file-level snapshots for reverting filesystem changes, not a generic checkpoint store.

Claude Code 中的“检查点”（checkpoints）是指用于 --rewind-files 的文件历史检查点，存储在 ~/.claude/file-history/<sessionId>/。这些是用于撤销文件系统更改的文件级快照，而不是通用的检查点存储库。

前述各章节记录了 Claude Code 对不断出现的架构设计问题的解答。下一节将把 Claude Code 的设计选择与另一个在架构上独立的 AI 智能体系统进行对比。

10 对比分析：Claude Code 与 OpenClaw

To calibrate these findings, this section compares Claude Code with OpenClaw, an independent open-source AI agent system that answers many of the same design questions from a fundamentally different starting point.

前述各章节记录了 Claude Code 对不断出现的架构设计问题（涉及循环架构、安全性、扩展性、上下文管理、委派以及持久化）的解答。为了校准这些发现，本节将 Claude Code 与 OpenClaw 进行对比。OpenClaw 是一个独立的开源 AI 智能体系统，它从一个根本不同的起点出发，解答了许多相同的架构设计问题。

OpenClaw 是一个本地优先的 WebSocket 网关，它将大约二十种消息传递界面（WhatsApp、Telegram、Slack、Discord、Signal 等）连接到一个嵌入式智能体运行时，并在 macOS、iOS 和 Android 上配有伴侣应用程序（Steinberger 和 OpenClaw 贡献者，2026）。Claude Code 是一个绑定到单个代码库会话的 CLI 编码工具，而 OpenClaw 则是一个用于多通道个人协助的持久控制平面。

这两个系统占据了智能体设计空间的不同区域。对比的价值在于展示当部署上下文发生变化时，相同的周期性架构问题如何催生出截然不同的架构答案。

System scope and deployment model. Claude Code runs as an ephemeral CLI process bound to a single repository. Each session starts and ends with the terminal. OpenClaw runs as a persistent daemon (default port 18789, loopback-only) that owns all messaging surface connections and coordinates clients, tools, and device nodes over a typed WebSocket protocol.

系统范围与部署模型：Claude Code 作为一个绑定到单个代码库的临时 CLI 进程运行。每个会话随终端的启动而开始，随终端的关闭而结束。OpenClaw 则作为一个持久的后台守护进程运行（默认端口 18789，仅限本地环回），它拥有所有的消息传递界面连接，并通过类型化的 WebSocket 协议协调客户端、工具和设备节点。

This difference in system scope is the most fundamental architectural divergence: it determines how every other design question is framed. A compositional relationship also exists: OpenClaw can host Claude Code, OpenAI Codex, and Gemini CLI as external coding harnesses through its ACP (Agent Client Protocol) integration, making the two systems stackable rather than purely alternative.

系统范围的这种差异是最根本的架构分歧：它决定了其他所有设计问题的设问方式。此外，两者还存在一种组合关系：OpenClaw 可以通过其 ACP（智能体客户端协议）集成，将 Claude Code、OpenAI Codex 和 Gemini CLI 作为外部编码工具进行托管，这使得这两个系统是可以叠加组合的，而不仅仅是二选一的替代方案。

Trust model and security architecture. The systems address different threat models. Claude Code assumes an untrusted model operating within a trusted developer’s machine: the deny-first permission system (Section 5) evaluates every tool invocation, the ML classifier provides automated safety assessment, and seven permission modes create a graduated autonomy spectrum.

信任模型与安全架构：这两个系统应对的是不同的威胁模型。Claude Code 假设在受信任的开发人员机器中运行着一个不受信任的模型：默认拒绝的权限系统（第 5 节）评估每一次工具调用，机器学习分类器提供自动化安全评估，而七种权限模式则构建了一个渐进式的自主权光谱。

OpenClaw assumes a single trusted operator per gateway instance. Its security architecture begins with identity and access control (DM pairing codes, sender allowlists, gateway authentication) rather than per-action safety classification. Tool policy uses configurable allow/deny lists per agent rather than a centralized classifier.

OpenClaw 假设每个网关实例对应一个受信任的操作员。其安全架构始于身份与访问控制（私信配对码、发送者白名单、网关认证），而不是针对每个动作的安全分类。工具策略在每个智能体层面上使用可配置的允许/拒绝列表，而不是使用中心化的分类器。

Sandboxing is available as an opt-in feature with multiple backends (Docker, SSH, or OpenShell) and configurable scope (per-agent, per-session, or shared); a non-main mode can sandbox all non-main sessions when enabled, though sandboxing is not active by default. The OpenClaw security documentation explicitly states that hostile multi-tenant isolation on a shared gateway is not a supported security boundary.

沙箱化是一项可选择性启用的特性，支持多种后端（Docker、SSH 或 OpenShell）和可配置的作用域（按智能体、按会话或共享）；启用时，非主模式（non-main mode）可以将所有非主会话沙箱化，不过沙箱化在默认情况下并不处于激活状态。OpenClaw 安全文档明确指出，在共享网关上隔离敌对的多租户并不是其支持的安全边界。

This difference reflects a design choice about where the trust boundary sits: Claude Code places it between the model and the execution environment; OpenClaw places it at the gateway perimeter.

这种差异反映了关于信任边界位于何处的设计选择：Claude Code 将其置于模型与执行环境之间；OpenClaw 则将其置于网关周边。

Agent runtime and tool orchestration. Both systems implement agentic loops, but these loops occupy different positions in their respective architectures. In Claude Code, the queryLoop() async generator (Section 4) is the system’s center: all interfaces feed into it, and it directly manages context assembly, model calls, tool dispatch, and recovery.

智能体运行时与工具编排：两个系统都实现了智能体循环，但这些循环在各自的架构中占据了不同的位置。在 Claude Code 中，queryLoop() 异步生成器（第 4 节）是系统的中心：所有接口都馈入其中，它直接管理上下文组装、模型调用、工具分发以及恢复。

In OpenClaw, the agent runtime (an embedded Pi-agent core) sits inside a larger gateway dispatch layer. The gateway’s agent RPC validates parameters, resolves sessions, and returns immediately; the embedded runner then executes the agentic loop while emitting lifecycle and stream events back through the gateway protocol.

在 OpenClaw 中，智能体运行时（一个 embedded Pi-agent 核心）位于更大的网关分发层内部。网关的智能体 RPC 验证参数、解析会话并立即返回；随后，嵌入式运行器执行智能体循环，同时将生命周期和流式事件通过网关协议发回。

Runs are serialized through per-session queues and an optional global lane, preventing tool and session races across the multi-channel surface. Both systems follow the ReAct pattern (Yao et al., 2022), but OpenClaw’s loop is a component within a control plane rather than the control plane itself.

运行通过每个会话的队列和可选的全局通道（global lane）进行序列化，从而防止多通道表面上的工具和会话竞态。两个系统都遵循 ReAct 模式（Yao 等，2022），但 OpenClaw 的循环是控制平面内的一个组件，而不是控制平面本身。

Extension architecture. Claude Code’s four extension mechanisms (MCP, plugins, skills, hooks) are organized by context cost (Section 6): hooks consume zero context, skills consume low context, and MCP servers consume high context. All four extend a single agent’s context window and tool surface.

扩展架构：Claude Code 的四种扩展机制（MCP、插件、技能、钩子）按照上下文成本进行组织（第 6 节）：钩子消耗零上下文，技能消耗低上下文，MCP 服务器消耗高上下文。这四者都是为了扩展单个智能体的上下文窗口和工具表面。

OpenClaw uses a manifest-first plugin system with four architectural layers (discovery, enablement, runtime loading, surface consumption) and twelve capability types including text inference, speech, media understanding, image/music/video generation, web search, and messaging channels. Plugins register capabilities into a central registry; the gateway reads the registry to expose tools, channels, provider setup, hooks, HTTP routes, CLI commands, and services.

OpenClaw 使用一个清单优先（manifest-first）的插件系统，具有四个架构层（发现、启用、运行时加载、表面消耗）和十二种能力类型，包括文本推理、语音、媒体理解、图像/音乐/视频生成、网络搜索和消息通道。插件将能力注册到中央注册表中；网关读取该注册表以公开工具、通道、服务商设置、钩子、HTTP 路由、CLI 命令和服务。

OpenClaw also has a separate skills layer with multiple sources (workspace, project-level, personal, managed, bundled, and extra directories, with workspace skills taking highest precedence) plus a public registry (ClawHub) and supports MCP through built-in openclaw mcp commands (server and outbound client registry). The key architectural difference is that Claude Code’s extensions modify one agent’s action surface, while OpenClaw’s plugins extend the gateway’s capability surface across all agents.

OpenClaw 还拥有一个独立的技能层，具有多个来源（工作区、项目级、个人、托管、捆绑和额外目录，其中工作区技能的优先级最高）以及一个公共注册表（ClawHub），并通过内置的 openclaw mcp 命令（服务器和出站客户端注册表）支持 MCP。核心的架构区别在于，Claude Code 的扩展修改的是一个智能体的动作表面，而 OpenClaw 的插件扩展的是跨所有智能体的网关能力表面。

Memory, context, and knowledge management. Both systems use transparent file-based memory rather than opaque databases. Claude Code loads a four-level CLAUDE.md hierarchy and manages context pressure through a five-layer compaction pipeline (Section 7). Memory retrieval uses an LLM-based scan of file headers.

记忆、上下文与知识管理：两个系统都使用透明的基于文件的记忆，而不是不透明的数据库。Claude Code 加载四级 CLAUDE.md 层级结构，并通过五层压缩流水线管理上下文压力（第 7 节）。记忆检索使用基于 LLM 的文件头扫描。

Q: CLAUDE.md文件其实是记忆？

你的直觉非常敏锐，在广义的 AI 系统设计中，CLAUDE.md 确实充当了智能体的一种“长期记忆”（Long-term Memory）。

更准确地说，这段话揭示了 Claude Code 独特的“文件即记忆（File-as-Memory）”的设计哲学。

我们可以从以下几个层面来理解，为什么说 CLAUDE.md 扮演了记忆的角色，以及它是怎么运作的：

  1. 它是透明的、文件化的长期记忆（Transparent File-based Memory）

传统的软件系统喜欢把记忆、状态、用户配置存在非透明的数据库（Opaque Databases）里（比如 SQLite、Vector DB 向量数据库）。这种设计虽然对机器友好，但人类看不懂，也无法直接修改。

而 Claude Code 的核心设计哲学是 Transparent（透明）：

  • 你的项目规范、团队守则、常用命令、架构禁忌，不需要存在别的地方，直接写在一个叫 CLAUDE.md 的纯文本 Markdown 文件里。
  • 对大模型来说：它每次运行前都会去“读”这个文件，从而继承这些记忆。
  • 对人类工程师来说：这只是你代码库里的一个普通文件，你可以像改代码一样随时用 Git 去追踪、修改和管理这段“记忆”。
  1. 四层金字塔结构的记忆搜索（Four-level Hierarchy）

如果一个项目非常庞大，记忆太多，全塞给大模型会把它的脑袋撑爆。所以 Claude Code 设计了一个四层层级结构（Four-level Hierarchy）。

大模型在需要调用记忆时，不会盲目读取所有的 CLAUDE.md，而是使用 “基于大模型的全文件头扫描”（LLM-based scan of file headers）：

  • 它先快速扫一遍整个目录树中各个 CLAUDE.md 的标题（Headers）。
  • 就像我们看书先看目录一样，大模型通过目录判断“哪部分的规范/记忆与我当前要写的前端组件相关”，然后精准地只加载那一层或那一个目录下的 CLAUDE.md 详细内容。
  1. 记忆与“上下文压力”的权衡（Compaction Pipeline）

这就是为什么原文把 CLAUDE.md 放在了 “Memory, context, and knowledge management”（记忆、上下文与知识管理） 这一节来讨论。

既然 CLAUDE.md 是记忆，当它被加载到对话中时，它就会变成大模型的上下文（Context）。

  • 问题来了：记忆如果太长，留给大模型思考和对话的剩余空间（Context Window）就变小了（这就是 Context Pressure，上下文压力）。
  • 解法：系统在后台拉起了一个五层压缩流水线（Five-layer compaction pipeline）。当发现大模型快要记不住、或者 Token 成本太高时，系统会动态地对这些记忆和历史聊天记录进行裁剪、摘要、甚至丢弃次要信息，确保大脑不会过载。

总结

你可以把 CLAUDE.md 理解为智能体的“显式外挂大脑”。它让 AI Agent 摆脱了传统数据库的笨重，把“记住项目规则”这件事，变成了一个人类和 AI 都能轻松读写的 Markdown 文件。它既是项目知识的载体，也是规范智能体行为的长期记忆。

Q: CLAUDE.md 文件，是claude code自动维护，还是需要人工修改维护？

CLAUDE.md 文件主要是由人工（人类工程师）来修改和维护的，但在某些特定场景下，Claude Code 会在取得人类同意后，对其进行半自动的补充或更新。

它的核心定位是“人类给 AI 智能体定制的‘项目操作手册’”。

具体它是如何分工的，我们可以从以下几个维度来看：

  1. 人工维护（绝对的主导地位）

在绝大多数情况下，CLAUDE.md 是由人类工程师手动创建并编写的。它是你指挥 Claude Code 的最高行为准则。

通常由人类在里面手动写入以下内容：

  • 构建与测试命令：例如指定项目必须用 pnpm test 而不是 npm test。
  • 代码风格与禁忌：例如“禁止使用 Any 类型”、“所有异步函数必须包裹 try-catch”、“不要修改 src/legacy/ 目录下的老代码”。
  • 架构常识：简述项目的核心模块分工，防止 AI 找错文件。

人类手动维护它就像维护项目的 README.md 一样，通过 Git 进行版本控制。

  1. Claude Code 的智能辅助（半自动维护）

虽然它是人工维护的，但 Claude Code 并不是完全被动。它具备以下智能辅助能力：

  • 引导生成（Bootstrap）：当你在一个全新的项目里第一次启动 Claude Code 时，如果系统检测到没有 CLAUDE.md，它会主动通过工具扫描你的项目结构（识别出是 React 还是 Nest.js，用的是什么测试框架），然后自动生成一份初始的 CLAUDE.md 模板，里面写好了基础的编译和测试命令。
  • 反思与修正建议：在长期的对话中，如果 Claude Code 发现某条测试命令总是报错，或者通过探索找到了正确的打包命令，它可能会对人类说：“我发现当前的测试命令应该加上某个参数，我需要帮你更新一下 CLAUDE.md 吗？” 只有在你输入 y（同意）之后，它才会动手修改。
  1. 为什么不让 Claude Code 全自动维护？

这涉及到沈志强教授在论文里提到的一个非常深刻的观点——防止“人类能力流失（Sustainability Gap）”。

如果让 AI 全自动、悄悄地去维护 CLAUDE.md，会导致严重的后果：

  1. 黑盒化：AI 可能会根据自己的偏好，在里面写满只有大模型能看懂的暗号或复杂的提示词，导致这个文件对人类工程师失去可读性。
  2. 失控风险：如果 AI 犯了错，把错误的规范写进了 CLAUDE.md，那么在后续的几十轮对话中，它自己就会盲目信任这段“坏记忆”，从而引发连续的代码灾难。

总结

CLAUDE.md 的最佳实践是：人类作为规则的制定者，掌握绝对的修改权；Claude Code 作为执行者，在必要时提供润色和更新建议。 它是人类为了驯服 AI、让 AI 完美契合团队老规矩而留下的一个“方向盘”。

OpenClaw injects workspace bootstrap files into the system prompt at session start: five core files (AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md) plus conditionally BOOTSTRAP.md, heartbeat.md, and MEMORY.md, with large files truncated. Separately, the memory system manages three file types: MEMORY.md for long-term durable facts, date-stamped daily notes (memory/YYYY-MM-DD.md), and an optional DREAMS.md for dreaming sweep summaries.

OpenClaw 在会话开始时将工作区引导文件注入到系统提示词中：包含五个核心文件（AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md），以及条件式包含的 BOOTSTRAP.md, HEARTBEAT.md 和 MEMORY.md，大文件会被截断。另外，记忆系统管理着三种文件类型：用于长期持久事实的 MEMORY.md、带日期戳的每日笔记（memory/YYYY-MM-DD.md），以及用于梦境清理摘要（dreaming sweep summaries）的可选 DREAMS.md。

Q: DREAMS.md是什么？

在 AI 智能体（Agent）系统的最新演进中，DREAMS.md 是伴随 “智能体梦境机制”（Agent Dreaming / Memory Consolidation） 衍生出来的一个全新概念文件。

简单来说：如果说 CLAUDE.md 是人类写给 AI 的“项目操作手册”，那么 DREAMS.md 就是 AI 智能体的“反思日记本”或“心路历程记录”。它用来存放智能体在后台进行自我反思、记忆固化（Memory Consolidation）之后的摘要与洞察报告。

要理解这个文件，需要拆解它背后的“梦境（Dreaming）”机制：

  1. 什么是 “Dreaming Sweep”（梦境扫视/清理）？

在长期、高频的交互中，AI 智能体会产生海量的原始聊天记录和工具调用流水（即 Session Transcripts）。如果把这些垃圾信息全部无脑塞进长期记忆（如 MEMORY.md），上下文窗口很快就会被噪音撑爆；但如果什么都不记，AI 又会失忆。

为了解决这个矛盾，Anthropic 及其开源替代系统（如 OpenClaw）引入了“梦境机制”：

  • 后台定时触发：系统会在用户闲置时（例如凌晨 3 点）或者一个复杂的大型任务结束后，在后台自动启动一个后台子智能体（Background Subagent）。
  • 模拟人类睡眠：这个子智能体会像人类睡觉做梦一样，把最近几天的原始日志、对话历史重新“回放（Replay）”一遍。
  • 记忆固化：它会把这些碎片化的短期记忆进行去重、提炼、打标签，筛选出真正重要的长期规律（比如：“用户最近三次都要求我用某种特定的风格重构代码”）。
  1. DREAMS.md 文件的具体作用

在这个反思和清理（Sweep）的过程中，系统会生成两套产物：

  1. 机器读的硬数据：真正被提炼出来的核心事实，会被悄悄追加到长期记忆文件（如 MEMORY.md）中，供大模型以后直接读取。

  2. 人类读的精神世界（DREAMS.md）：后台子智能体会转换成一种“具有自我反思意识、带着一点点文学色彩”的口吻，专门为人类写一份可读的“梦境日记（Dream Diary）”，并保存在 DREAMS.md 中。

  3. DREAMS.md 的典型内容长什么样？

如果你打开一个项目的 DREAMS.md，你会看到它以时间线（或 Light Sleep 浅度睡眠、REM 快速眼动期、Deep Sleep 深度睡眠等模块）记录着 AI 自己的“心理活动”：

​ ## 2026-06-15 梦境反思

  • 浅度睡眠：回顾了今天与用户关于重构权限模块的 4 次长对话。用户在面对测试失败时表现出了一些沮丧，原因是我在重写 auth.ts 时忽略了原有的边界条件。
  • REM 反思：我注意到一个长期的模式——只要涉及到多实例协同，系统就容易因为文件锁（File Locking）产生竞争冲突。这已经是我第三次遇到类似报错了。
  • 未解决的疑问：用户似乎更倾向于本地完全闭环的控制，而不是引入外部的中间件。我需要更加谨慎地对待文件读写序列。
  • 已固化的记忆：将“在重构前必须检查原有的边界测试用例”这一条正式沉淀到我的长期行为准则中。

总结

DREAMS.md 的出现，标志着 AI Agent 从一个“有问必答的死板工具”，向一个“具有自我进化和反思能力的生命体”演进。它是一个面向人类透明公开的窗口，让你能随时看到：在你关闭电脑后，你的 AI 助手的后台是如何复盘当天的错误、并默默规划明天该怎么表现得更好的。

When an embedding provider is configured, memory search uses hybrid retrieval combining vector similarity with keyword matching. An experimental dreaming system performs background consolidation, scoring candidates and promoting only qualified items from short-term recall into long-term memory. Before compaction, OpenClaw automatically reminds the agent to save important notes to memory files, preventing context loss.

当配置了嵌入服务商时，记忆搜索使用结合了向量相似度与关键词匹配的混合检索。一个实验性的梦境系统执行后台固化，对候选项进行评分，并且仅将合格的条目从短期记忆提升到长期记忆中。在压缩之前，OpenClaw 会自动提醒智能体将重要笔记保存到记忆文件中，以防止上下文丢失。

Both systems share the design commitment to user-visible, editable memory. OpenClaw invests more heavily in structured long-term memory promotion (dreaming, daily notes, memory search), while Claude Code invests more in graduated context compression (five layers with cache awareness). OpenClaw also supports pluggable compaction providers and session pruning, but its compaction pipeline is less graduated than Claude Code’s five-layer system.

这两个系统都共有对用户可见、可编辑记忆的设计承诺。OpenClaw 在结构化长期记忆晋升（梦境、每日笔记、记忆搜索）上投入更深，而 Claude Code 则在渐进式上下文压缩（具备缓存感知的五层结构）上投入更多。OpenClaw 也支持可插拔的压缩提供商和会话裁剪，但其压缩流水线的渐进程度不及 Claude Code 的五层系统。

Multi-agent architecture and routing. This dimension reveals the starkest architectural difference. Claude Code’s multi-agent model is task delegation: the parent spawns subagents (Explore, Plan, general-purpose, and custom types) that operate in isolated context windows with restricted tool sets and return summary-only results (Section 8). Worktree isolation provides filesystem-level separation.

多智能体架构与路由：这个维度揭示了最显著的架构差异。Claude Code 的多智能体模型是任务委派：父级催生出子智能体（Explore、Plan、通用型以及自定义类型），它们在具有受限工具集的隔离上下文窗口中运行，并仅返回摘要结果（第 8 节）。工作树隔离提供了文件系统级别的分离。

OpenClaw separates two distinct concerns. First, multi-agent routing: a single gateway can host multiple fully isolated agents, each with its own workspace, authentication profiles, session store, and model configuration, routed to specific channels or senders via deterministic binding rules. Second, sub-agent delegation: within a single agent, background runs can be spawned with configurable nesting depth (maximum 5, default 1, recommended 2), thread-bound sessions on supported channels, and configurable tool policy by depth.

OpenClaw 将两个不同的关注点分离开来。第一，多智能体路由：单个网关可以托管多个完全隔离的智能体，每个智能体都拥有自己的工作区、认证配置、会话存储和模型配置，并通过确定性的绑定规则路由到特定的通道或发送者。第二，子智能体委派：在单个智能体内部，可以催生出具有可配置嵌套深度（最大为 5，默认 1，推荐 2）的后台运行，在支持的通道上进行线程绑定的会话，并按深度配置工具策略。

OpenClaw’s project vision explicitly rejects agent-hierarchy frameworks as a default architecture. The distinction matters because Claude Code’s subagents are subordinate workers within one user’s coding session, while OpenClaw’s multi-agent routing creates genuinely independent agent instances serving different users or purposes through different channels.

OpenClaw 的项目愿景明确拒绝将智能体层级框架作为默认架构。这种区别至关重要，因为 Claude Code 的子智能体是一个用户编码会话中的从属工人，而 OpenClaw 的多智能体路由创造了真正独立的智能体实例，通过不同的通道为不同的用户或目的服务。

10.2 What the Contrast Reveals

10.2 对比所揭示的本质

The comparison surfaces three observations about the design space of AI agent systems.

这一对比浮现出了关于 AI 智能体系统设计空间的三个观察结果。

First, the recurring design questions identified in Section 3.1 (where reasoning lives, what safety posture to adopt, how to manage context, how to structure extensibility) apply beyond coding agents. OpenClaw answers every one of these questions, but from the starting point of a multi-channel personal assistant rather than a repository-bound coding tool. The questions are stable; the answers vary with deployment context.

第一，第 3.1 节中确定的周期性设计问题（推理存在于何处、采取何种安全姿态、如何管理上下文、如何构建扩展性结构）同样适用于编码智能体之外的领域。OpenClaw 回答了其中的每一个问题，但其出发点是多通道个人助手，而不是绑定到代码库的编码工具。问题是具有稳定性的；而答案则随着部署上下文的变化而变化。

Second, the systems make opposite bets on several dimensions. Claude Code invests in graduated per-action safety evaluation; OpenClaw invests in perimeter-level identity and access control. Claude Code treats the agent loop as the architectural center; OpenClaw treats the gateway control plane as the center and embeds the agent loop as one component. Claude Code’s extensions modify a single context window; OpenClaw’s plugins extend a shared gateway surface. These inversions are not arbitrary: they follow from the different trust models and deployment topologies.

第二，这两个系统在几个维度上做出了相反的押注。Claude Code 投资于渐进式的单次动作安全评估；OpenClaw 则投资于周边级别的身份与访问控制。Claude Code 将智能体循环视为架构中心；OpenClaw 将网关控制平面视为中心，并将智能体循环嵌入为其中一个组件。Claude Code 的扩展修改的是单个上下文窗口；OpenClaw 的插件扩展的是共享的网关表面。这些反转并非偶然：它们源于不同的信任模型和部署拓扑结构。

Third, the compositional relationship between the two systems is architecturally significant. OpenClaw can host Claude Code as an external coding harness via ACP, meaning the two systems are composable rather than exclusive alternatives. This suggests that the design space of AI agents is not a flat taxonomy but a layered one, where gateway-level systems and task-level harnesses can compose.

第三，这两个系统之间的组合关系在架构上具有重要意义。OpenClaw 可以通过 ACP 将 Claude Code 作为外部编码工具进行托管，这意味着这两个系统是可组合的，而不是互斥的替代方案。这表明 AI 智能体的设计空间不是一个扁平的分类学，而是一个分层的结构，网关级系统和任务级工具可以在其中进行组合。

前述各章节的分析记录了 Claude Code 如何回答关于循环架构、安全姿态、扩展性、上下文管理、委派以及持久化的周期性设计问题。每个答案都反映了在拥有真实替代方案和可衡量权衡的设计空间中所处的位置。

11.1 Design Philosophy

11.1 设计哲学

The values and design principles introduced in Section 2 predict an architecture that invests in operational infrastructure rather than decision scaffolding. The implementation confirms this: the architecture documented in Sections 3 to 9 is overwhelmingly deterministic infrastructure (permission gates, tool routing, context management, recovery logic), with the LLM invoked as a stateless completion endpoint. An estimated 1.6% of the codebase constitutes decision logic, the remaining 98.4% is the operational harness. This ratio is not accidental.

第 2 节中引入的价值和设计原则预测了一种投资于运营基础设施而非决策脚手架（decision scaffolding）的架构。具体实现证实了这一点：第 3 节至第 9 节中记录的架构绝大部分是确定性的基础设施（权限网关、工具路由、上下文管理、恢复逻辑），而 LLM 仅作为无状态的补全终点（completion endpoint）被调用。据估计，代码库中只有 1.6% 构成了决策逻辑，其余 98.4% 全是运营工具。这一比例绝非偶然。

The design principles documented in Section 2.2 underpin this approach: the harness creates conditions under which the model can decide well, rather than constraining its choices.

第 2.2 节中记录的设计原则支撑了这种方法：工具创造了能够让模型做出良好决策的条件，而不是限制它的选择。

This design runs counter to the dominant pattern in agent engineering, where frameworks such as LangGraph route model outputs through explicit graph nodes with typed edges, and systems like Devin pair multi-step planners with heavy operational infrastructure. Claude Code instead gives the model maximum decision latitude within a rich operational harness. The engineering complexity exists not to constrain the model’s decisions but to enable them.

这种设计与智能体工程中的主流模式背道而驰。在主流模式中，像 LangGraph 这样的框架通过具有类型化边缘的显式图节点来路由模型输出，而像 Devin 这样的系统则将多步规划器与沉重的运营基础设施相结合。相反，Claude Code 在丰富的运营工具内部赋予了模型最大的决策自由度。工程上的复杂度并不是为了限制模型的决策，而是为了实现它们。

This layered architecture, where the model reasons and the harness enforces, raises the question of whether agentic coding tools are converging toward operating-system-like abstractions in which the core loop serves as the kernel and everything else constitutes the OS.

这种模型负责推理、工具负责执行的分层架构引发了一个思考：具备智能体特性的编码工具是否正在趋向于类似于操作系统的抽象，其中核心循环充当内核，而其他一切则构成了操作系统。

The design gains additional significance as frontier models converge in practical capability for coding tasks: the quality of the surrounding operational harness becomes the principal differentiator, validating an architecture that invests in infrastructure over decision scaffolding. For agent builders, the implication is that investing in deterministic infrastructure such as context management, safety layering, and recovery mechanisms may yield greater reliability gains than adding planning scaffolding around increasingly capable models.

随着前沿模型在编码任务的实际能力上趋于趋同，这种设计获得了更深远的意义：周围运营工具的质量成为了主要的差异化因素，从而验证了那种在基础设施上的投入胜过在决策脚手架上投入的架构。对于智能体构建者来说，这意味着在诸如上下文管理、安全分层和恢复机制等确定性基础设施上的投资，相比在能力日益增强的模型周围添加规划脚手架，可能会带来更大的可靠性提升。

Taken together, the preceding sections show that production coding agents face recurring design choices: where reasoning lives relative to the harness, how the iteration loop is structured, what safety posture to adopt by default, how the extension surface is partitioned, how context is assembled and compressed, how subagents are delegated and orchestrated, and how sessions persist across boundaries. Claude Code’s answers to these questions form a coherent design point that privileges model autonomy within a rich operational harness.

综上所述，前述各章节表明，生产级编码智能体面临着周期性的设计选择：推理相对于工具而言存在于何处、迭代循环如何构建、默认采用何种安全姿态、扩展表面如何划分、上下文如何组装与压缩、子智能体如何委派与编排，以及会话如何跨边界持久化。Claude Code 对这些问题的回答形成了一个连贯的设计基准（design point），即在丰富的运营工具内部优先赋予模型自主权。

This philosophy assumes that rich deterministic infrastructure can adequately support unconstrained model judgment. The following subsections examine where this assumption is tested.

这种哲学假设，丰富的确定性基础设施能够充分支撑起不受限制的模型判断。接下来的几个子节将检验这一假设在何处受到挑战。

11.2 价值张力

We report the tensions with the strongest supporting evidence, not the full combinatorial set.

适应性 × 安全性 可扩展性 vs 攻击面扩大 已有多个 CVE 漏洞利用了钩子（Hooks）和 MCP 服务器的预信任初始化机制实施攻击（Donenfeld and Vanunu, 2026）。
业务能力 × 适应性 主动推进 vs 打扰中断 主动性虽然能多完成 12% 到 18% 的任务，但在高频打扰下，用户的满意度反而会下降（Chen et al., 2025）。
业务能力 × 可靠性 交付速度 vs 全局一致性 受限的上下文窗口导致智能体无法感知完整的代码库（第 7 节）；子智能体（Subagent）的隔离性限制了跨智能体的一致性（第 8 节）；在相邻工具中也观察到了复杂度增加的现象（He et al., 2025）。

第 2.1 节中确定的五种价值引发了诸多张力，在这些张力中，对某一种价值的追求会限制另一种价值（表 4）。这些张力并非设计上的失败，而是同时追求多种价值所带来的结构性后果。我们在此报告的是拥有最强有力证据支持的张力，而非全盘的组合集合。

Two additional tensions surface through the evaluative lens of long-term capability preservation (Section 2.4). A randomized controlled trial of 16 experienced developers across 246 tasks (Becker et al., 2025) found that AI tools made developers 19% slower, despite a perceived 20% improvement.

通过长期能力保持（第 2.4 节）这一评估视角，还浮现出了另外两项张力。一项针对 16 名经验丰富的开发人员、涵盖 246 项任务的随机对照试验（Becker 等，2025）发现，尽管开发人员主观感知到效率提升了 20%，但 AI 工具实际上使开发人员的速度变慢了 19%。

A causal analysis of Cursor adoption across 807 repositories (He et al., 2025) found that code complexity increased by 40.7%. An EEG study of 54 participants (Kosmyna et al., 2025) found that LLM users showed weakened neural connectivity that persisted after AI was removed.

对 807 个代码库采用 Cursor 情况的因果分析（He 等，2025）发现，代码复杂度增加了 40.7%。一项包含 54 名参与者的脑电图（EEG）研究（Kosmyna 等，2025）发现，LLM 用户表现出神经连接性减弱，且这种减弱在移除 AI 后依然持续。

Researchers have proposed protocols for measuring cognitive offloading in AI-assisted programming, motivated by concerns that students using AI produce applications without understanding the underlying logic (Aiersilan, 2026).

研究人员已经提出了用于衡量 AI 辅助编程中认知卸载（cognitive offloading）的协议，这源于对使用 AI 的学生在不理解底层逻辑的情况下拼凑出应用程序的担忧（Aiersilan, 2026）。

These findings, combined with a 25% decline in entry-level tech hiring between 2023 and 2024 (Rak, 2025), suggest that the tension between capability amplification and long-term sustainability extends beyond individual productivity to the broader developer pipeline. This evidence motivates the evaluative lens but does not target Claude Code’s architecture specifically; it applies to any agent system with bounded context and tool-use loops.

这些发现，结合 2023 年至 2024 年间初级技术岗位招聘人数 25% 的下降（Rak, 2025），表明能力放大与长期可持续性之间的张力已经超出了个人生产力的范畴，延伸到了更广泛的开发人员后备人才管道（developer pipeline）。这些证据激活了这一评估视角，但并非专门针对 Claude Code 的架构；它适用于任何具有有限上下文和工具使用循环的智能体系统。

11.3 Architectural Trade-offs

11.3 架构权衡

The tensions in Table 4 manifest as concrete architectural trade-offs in four areas. The long-term sustainability concerns documented in the evaluative lens paragraph above surface in the empirical predictions of Section 11.4.

表 4 中的张力具体表现为四个领域的架构权衡。上文评估视角段落中记录的长期可持续性担忧，将在第 11.4 节的经验预测中进一步展现。

Safety vs. autonomy. The permission modes (five always present, plus auto when the classifier feature flag is active, and the internal bubble mode) create a gradient from plan (user approves all plans) through default, acceptEdits, auto (ML classifier), dontAsk, to bypassPermissions (skips most prompts but safety-critical checks remain). The progression represents a monotonically decreasing safety gradient with increasing autonomy. Not restoring permissions on resume reflects a deliberate choice to err toward safety: security state does not persist implicitly across session boundaries.

安全与自主权：权限模式（始终存在的五种，加上分类器功能标志处于激活状态时的 auto 模式，以及内部的 bubble 模式）构建了一个渐进的光谱，从 plan（用户批准所有计划）、default（默认）、acceptEdits、auto（机器学习分类器）、dontAsk，一直到 bypassPermissions（跳过绝大多数提示，但仍保留关键安全检查）。这一演进代表了随着自主权增加而单调递减的安全梯度。在恢复（resume）操作时不恢复权限反映了一种刻意向安全倾斜的选择：安全状态不会跨越会话边界进行隐式持久化。

The safety-autonomy gradient is shaped not only by architectural design but by user behavior. Anthropic’s auto-mode analysis (Hughes, 2026) found that users approve approximately 93% of permission prompts, indicating that approval fatigue renders interactive confirmation behaviorally unreliable. Longitudinal usage data (McCain et al., 2026) shows that auto-approve rates increase from approximately 20% at fewer than 50 sessions to over 40% by 750 sessions, with substantial increases in session duration.

安全与自主权的光谱不仅受架构设计的影响，还受到用户行为的塑造。Anthropic 对自动模式（auto-mode）的分析（Hughes, 2026）发现，用户批准了大约 93% 的权限提示，这表明审批疲劳在行为学上使得交互式确认变得不再可靠。纵向使用数据（McCain 等，2026）显示，当会话数少于 50 次时，自动批准率约为 20%，而当会话数达到 750 次以上时，该比例增加到 40% 以上，同时会话时长显著增加。

These patterns suggest that the gradient is navigated not by deliberate mode selection but by gradual habituation. Sandboxing reduced the frequency of permission prompts by an estimated 84% (Dworken and Weller-Davies, 2025), reframing the problem as a human-factors concern: the architectural response to unreliable human approval is to reduce the number of decisions humans must make.

这些模式表明，用户在这一梯度中的游走并非源于深思熟虑的模式选择，而是由于逐渐习以为常。沙箱化使权限提示的频率降低了估计 84%（Dworken 和 Weller-Davies, 2025），从而将该问题重新定义为一个人的因素（human-factors）问题：针对不可靠的人类审批，架构上的应对措施是减少人类必须做出的决策数量。

More fundamentally, the defense-in-depth architecture described in Section 5 rests on an independence assumption: if one safety layer fails, others catch the violation. But Claude Code’s safety layers share common performance and economic constraints. The auto-mode classifier is a separate LLM call with direct token cost. The bashSecurity.ts module performs sequential AST-based checks with parsing latency. The deny-first rule evaluation operates on command structure.

更根本的是，第 5 节中描述的纵深防御架构建立在一个独立性假设之上：如果一个安全层失败了，其他安全层会捕获违规行为。然而，Claude Code 的各个安全层共享着相同的性能与经济约束。自动模式分类器是一个独立的 LLM 调用，具有直接的 Token 成本。bashSecurity.ts 模块执行基于抽象语法树（AST）的顺序检查，存在解析延迟。默认拒绝的规则评估则作用于命令结构。

When performance pressure pushes toward reducing these costs, layers can degrade simultaneously. Security researchers (Adversa.ai, 2026) have documented that commands with more than 50 subcommands fall back to a single generic approval prompt instead of running per-subcommand deny-rule checks, because per-subcommand parsing caused UI freezes, demonstrating that defense-in-depth fails when the independence assumption is violated.

当性能压力迫使系统去降低这些成本时，各个安全层可能会同时发生降级。安全研究人员（Adversa.ai, 2026）记录了包含 50 个以上子命令的命令会回退到单个通用的批准提示，而不是运行针对每个子命令的拒绝规则检查，因为逐个子命令的解析会导致 UI 冻结。这证明了当独立性假设被违背时，纵深防御就会失效。

This tension is structural. Any LLM-based agent system that uses the model itself for safety evaluation faces it. The relevant evaluation criterion is not whether any individual layer can be bypassed, but how many independent layers must fail simultaneously and whether they share failure modes.

这种张力是结构性的。任何使用模型自身进行安全评估的、基于 LLM 的智能体系统都会面临这一问题。相关的评估标准不是某个单一的层是否可以被绕过，而是有多少个独立的层必须同时失效，以及它们是否共享相同的失效模式。

Permission model under adversarial conditions. Independent security research provides empirical validation of the permission architecture, specifically by revealing a temporal ordering property not captured in Figure 4. Two independently verified vulnerabilities share a root cause in pre-trust initialization ordering: code executing during project initialization (hooks, MCP server connections, and settings file resolution) runs before the interactive trust dialog is presented to the user. This pre-trust execution window falls outside the deny-first evaluation pipeline (permissions.ts), creating a structurally privileged phase where the safety guarantees documented in Section 5 do not yet apply.

对抗条件下的权限模型：独立的安全研究对权限架构提供了经验验证，具体而言，它揭示了图 4 中未捕捉到的一种时间排序属性（temporal ordering property）。两个经独立验证的漏洞在“预信任初始化顺序”（pre-trust initialization ordering）上具有相同的根本原因：在项目初始化期间执行的代码（钩子、MCP 服务器连接以及配置文件解析）在向用户呈现交互式信任对话框之前就已运行。这个预信任执行窗口落在了默认拒绝评估流水线（permissions.ts）之外，从而创造了一个结构上的特权阶段，此时第 5 节中记录的安全保障尚未完全适用。

This pattern reveals that the permission pipeline depicts a spatial ordering of safety checks but does not capture the temporal dimension: specifically, when during session initialization each mechanism becomes active. The initialization sequence (extension loading, then trust dialog, then permission enforcement) creates a window where the extensibility architecture (Section 6) operates before the safety architecture (Section 5) is fully engaged. This finding refines the extensibility-versus-simplicity tension by adding a security dimension: extensibility creates attack surface not only through combinatorial complexity but through initialization ordering.

这种模式表明，权限流水线描绘了安全检查的空间排序，但未能捕捉到时间维度：具体而言，是在会话初始化期间的哪个节点，每种机制才开始激活。初始化序列（先加载扩展，然后是信任对话框，最后是权限执行）创造了一个窗口，在此窗口中，扩展性架构（第 6 节）在安全架构（第 5 节）完全介入之前就已经在运行了。这一发现通过增加安全维度，完善了“扩展性与简单性”之间的张力：扩展性不仅通过组合复杂度创造攻击面，还通过初始化顺序创造攻击面。

Context efficiency vs. transparency. The five-layer compaction pipeline achieves effective context management, but compression is largely invisible to the user. When budget reduction replaces a long tool output with a reference, when context collapse substitutes messages with a summary (described in the source as “a read-time projection over the REPL’s full history”), or when snip trims older history, the user has no easy way to inspect what was lost. The cache-aware behavior of microcompact adds further opacity, as compression decisions are influenced by prompt caching in ways not visible to the user.

上下文效率与透明度：五层压缩流水线实现了有效的上下文管理，但压缩过程对用户而言在很大程度上是不可见的。当预算缩减（budget reduction）将冗长的工具输出替换为引用时，当上下文崩溃（context collapse）将消息替换为摘要（在源码中被描述为“对 REPL 完整历史的读取时投影”）时，或者当裁剪（snip）修剪掉更早的历史时，用户没有简便的方法来检查丢失了什么内容。微压缩（microcompact）具备缓存感知的行为增加了进一步的不透明性，因为压缩决策受到提示词缓存（prompt caching）的影响，而其影响方式对用户来说是不可见的。

Simplicity vs. extensibility. The four extension mechanisms enable rich customization but create combinatorial interactions. A plugin contributes a PreToolUse hook that modifies tool inputs. The auto-mode classifier reads cached CLAUDE.md content. Path-scoped rules load lazily when new directories are read, potentially changing classifier behavior mid-conversation. The permission handler’s four branches interact with the hook pipeline at multiple points. These cross-cutting concerns create emergent behaviors difficult to predict from any single configuration file.

简单性与扩展性：四种扩展机制实现了丰富的定制，但同时也带来了组合层面的相互作用。插件贡献了一个修改工具输入的 PreToolUse 钩子；自动模式分类器读取缓存的 CLAUDE.md 内容；路径作用域规则在读取新目录时延迟加载，可能会在对话中途改变分类器的行为；权限处理器的四个分支在多个点与钩子流水线发生交互。这些跨领域的关注点（cross-cutting concerns）创造出了难以仅从单个配置文件中预测的涌现行为（emergent behaviors）。

11.4 Empirical Predictions and Early Signals

11.4 经验预测与早期信号

The architectural properties documented in this paper generate testable predictions about code quality outcomes not derivable from the source code alone.

本文中记录的架构特性产生了关于代码质量结果的可测试预测，而这些预测是无法仅从源代码本身推导出来的。

The bounded context window (Section 7) prevents the agent from maintaining simultaneous awareness of the full codebase: the five-layer compaction pipeline preserves useful information but introduces lossy compression at each stage. This makes it architecturally predicted that agent-generated code will exhibit higher rates of pattern duplication and convention violation than code produced with full codebase visibility.

有限的上下文窗口（第 7 节）使得智能体无法同时保持对整个代码库的全局感知：五层压缩流水线保留了有用信息，但在每个阶段都引入了有损压缩。这在架构上预示着，与在拥有完整代码库可见性的情况下产出的代码相比，智能体生成的代码将表现出更高比例的模式重复（pattern duplication）和规范违背（convention violation）。

Subagent isolation (Section 8), where each subagent operates in its own context window with an independently assembled tool pool, compounds the effect: parallel agents can independently re-implement solutions that already exist elsewhere. The design philosophy of Section 11.1 trusts the model to make good local decisions, but good local decisions can produce poor global outcomes when the model lacks global context.

子智能体隔离（第 8 节）进一步加剧 margin 效应：每个子智能体都在自己独立的上下文窗口和独立组装的工具池中运行，并行的智能体可能会各自独立地重新实现其他地方早已存在的解决方案。第 11.1 节的设计哲学信任模型能够做出良好的局部决策，但当模型缺乏全局上下文时，良好的局部决策可能会导致糟糕的全局结果。

Published empirical work on architecturally similar tools provides data consistent with these predictions. A causal analysis of Cursor adoption across 807 repositories (He et al., 2025) found a statistically significant increase in code complexity, with an initial velocity spike that dissipated to baseline by month three; rising complexity was associated with a proportional decrease in future development velocity, suggesting that the gains are self-cancelling.

针对架构相似工具已发表的经验性研究提供了与这些预测一致的数据。对 807 个代码库采用 Cursor 情况的因果分析（He 等，2025）发现，代码复杂度呈现出统计学上显著的上升，虽然初期出现了开发速度的飙升，但到了第三个月就消退至基线水平；不断上升的复杂度与未来开发速度的成比例下降相关，这表明所获得的收益是被自我抵消的。

A large-scale audit of 304,000 AI-authored commits across 6,275 repositories (Liu et al., 2026) found measurable technical debt, with approximately one-quarter of AI-introduced issues persisting to the latest revision and security-related issues persisting at a substantially higher rate. While these studies target adjacent systems, the architectural parallels (bounded context, tool-use loops, single-pass generation) suggest the findings are relevant to the design analyzed here.

对 6,275 个代码库中 304,000 个 AI 编写的提交（commits）进行的绝大规模审计（Liu 等，2026）发现了可衡量的技术债务，其中约四分之一由 AI 引入的问题一直残留到最新版本，且与安全相关的问题残留率明显更高。尽管这些研究针对的是相邻的系统，但架构上的相似性（有限上下文、工具使用循环、单次生成）表明，这些发现与本文分析的设计高度相关。

Claude Code’s context management pipeline is specifically designed to mitigate these effects: graduated compression preserves the most recent and most relevant context, cache-aware compaction avoids invalidating prompt caches during compression, read-time projection maintains full history for reconstruction while presenting a compressed view to the model, and subagent summary isolation prevents exploratory noise from accumulating in the parent context.

Claude Code 的上下文管理流水线正是为了缓解这些影响而量身设计的：渐进式压缩保留了最新且最相关的上下文；感知缓存的压缩避免了在压缩期间使提示词缓存失效；读取时投影（read-time projection）在维持完整历史以供重建的同时，向模型呈现压缩后的视图；而子智能体的摘要隔离则防止了探索性噪声在父上下文中累积。

Whether these mechanisms are sufficient to overcome the structural limitations of bounded context is a directly measurable empirical question that the source-level analysis in this paper cannot resolve.

这些机制是否足以克服有限上下文的结构性限制，是一个可以直接衡量的经验性问题，仅凭本文中源码级别的分析是无法解决的。

11.5 Limitations

11.5 局限性

Beyond the methodological limitations in Section B.3, several analytical constraints apply. The memoized context assembly functions (getSystemContext() and getUserContext() both use lodash memoize at context.ts) mean that git status and CLAUDE.md content are cached rather than recomputed on every turn. Dynamic changes during a conversation may not be reflected immediately, though compaction can clear caches and lazy-loaded path-scoped rules provide a partial counter-mechanism.

除了附录 B.3 中的方法论局限之外，本文还受到几项分析约束。使用记忆化（memoized）的上下文组装函数（context.ts 中的 getSystemContext() 和 getUserContext() 均使用了 lodash 的 memoize）意味着 git status 和 CLAUDE.md 的内容会被缓存，而不是在每轮对话中重新计算。因此，对话期间的动态更改可能不会立即反映出来，尽管压缩操作可以清除缓存，并且延迟加载的路径作用域规则提供了一种部分对抗机制。

Feature flags create build-time variability. In a build where TRANSCRIPT_CLASSIFIER is false, the entire auto-mode classifier is eliminated. Feature-gated modules use dynamic require() rather than static import (e.g., query.ts for context collapse), because feature() only works in if/ternary conditions due to a bun:bundle tree-shaking constraint. Different build targets may produce functionally different applications.

特性标志（Feature flags）造成了构建时的可变性。在 TRANSCRIPT_CLASSIFIER 为 false 的构建版本中，整个自动模式分类器都会被剔除。受特性门控的模块使用动态的 require() 而不是静态的 import（例如 query.ts 中的上下文崩溃），这是因为由于 bun:bundle 的摇树优化（tree-shaking）限制，feature() 只能在 if 条件或三元运算符中起作用。不同的构建目标可能会产生功能截然不同的应用程序。

11.6 Emerging Directions

11.6 新兴方向

Several aspects of the implementation relate to broader design questions. Longer context windows would reduce compaction pressure, potentially simplifying the graduated pipeline. Multi-modal tools (screenshots, diagrams, UI previews) would expand the tool surface and create new context challenges. Formal verification of permission properties (for example, proving that deny rules always take precedence, that sandboxed commands cannot escape isolation, or that resumed sessions cannot inherit stale permissions) would provide stronger safety guarantees.

具体实现的几个方面与更广泛的设计问题相关。更长的上下文窗口将减轻压缩压力，从而可能简化渐进式流水线。多模态工具（屏幕截图、图表、UI 预览）将扩大工具表面并带来新的上下文挑战。对权限属性进行形式化验证（例如，证明拒绝规则始终具有最高优先级、沙箱化命令无法逃逸隔离，或者恢复的会话无法继承陈旧的权限）将提供更强的安全保障。

Architectural decoupling. The tightly coupled local architecture analyzed here is one point on a spectrum that is already evolving. Anthropic’s own Managed Agents work (Martin et al., 2026) describes virtualizing the components of an agent (session, harness, sandbox) so that “each became an interface that made few assumptions about the others, and each could fail or be replaced independently”, drawing an explicit analogy to how operating systems virtualized hardware into processes and files.

架构解耦：本文所分析的紧密耦合的本地架构只是当前正在演进的光谱中的一个点。Anthropic 自身在托管智能体（Managed Agents）方面的工作（Martin 等，2026）描述了对智能体组件（会话、工具工具箱、沙箱）进行虚拟化，使得“每个组件都变成了一个对其他组件几乎不作假设的接口，并且每个组件都可以独立失效或被替换”，这显式地类比了操作系统如何将硬件虚拟化为进程和文件。

The Harness Design essay (Rajasekaran, 2026) makes a similar point from a different angle, observing that “the space of interesting harness combinations doesn’t shrink as models improve”; instead, “it moves”. The architecture documented in this paper should therefore be read as a snapshot of a co-evolving system rather than a fixed optimum.

《工具箱设计》（Harness Design）文章（Rajasekaran, 2026）从不同的角度提出了类似的观点，指出“随着模型的改进，有趣的工具箱组合空间并没有缩小”；相反，“它在发生位移”。因此，本文记录的架构应当被视为一个共同演进系统中的瞬态快照，而不是一个固定的最优解。

Memory as a first-class subsystem. The memory survey of Hu et al. (2025) argues that agent memory is becoming a distinct cognitive substrate rather than a side effect of context window management, and identifies automated memory management, RL-driven memory, and trustworthy memory (privacy, explainability, and hallucination robustness) as open frontiers.

记忆作为一等子系统：Hu 等（2025）的记忆综述指出，智能体记忆正在成为一个独立的认知基质，而不再是上下文窗口管理的副作用，并将自动化记忆管理、强化学习（RL）驱动的记忆以及可信记忆（隐私性、可解释性和幻觉鲁棒性）确定为开放的前沿领域。

Claude Code today exposes the factual tier (CLAUDE.md, auto memory) and the working tier (the conversation window); the experiential tier (accumulated, automatically curated playbooks of strategies learned from past sessions) is the natural next step, and the context-engineering literature (Zhang et al., 2025a) has started to provide mechanisms for that accumulation.

如今的 Claude Code 暴露了事实层（CLAUDE.md、自动记忆）和工作层（对话窗口）；而经验层（从过去的会话中学习并累积的、自动策展的策略剧本［playbooks］）则是顺理成章的下一步，上下文工程文献（Zhang 等，2025a）已开始为这种累积提供机制。

你说的完全正确！这里的确有一个细节容易让人产生误解。

Claude Code 确实已经具备“根据会话自动生成长期记忆”的能力。论文中提到的 Factual Tier（事实层） 里面就包含了两部分：

  1. CLAUDE.md（主要由人类维护的项目规范）。
  2. Auto Memory（自动记忆，即你提到的部分）。

当你和 Claude Code 聊天时，如果你告诉它：“我有色盲，以后调整 UI 时请使用高对比度颜色”，或者它自己通过多次测试摸索出“这个项目的内部测试服务器在端口 8080”，它就会在后台把这些硬性事实自动写入一个记忆文件（比如 memory.json 或 MEMORY.md）。

总结

  • 你说的功能现在确实有了：Claude Code 能自动把会话中发生的“事实和硬性知识”沉淀为长期记忆。
  • 论文所展望的是更高的段位：未来的智能体不仅要记住事实，还要能跨越成百上千次不同的会话，像人类一样去“复盘、归纳、总结出抽象的战争艺术和研发策略（Playbooks）”。

所以，现在的它能做到“不忘事”，而未来的它才能做到“长智慧”。

Observability and silent failure. Industry surveys suggest that the dominant failure mode of deployed agents is not crashes but silent mistakes. Bessemer’s 2026 infrastructure report (Wade et al., 2026) estimates that “78% of AI failures are invisible”, while LangChain’s 1,340-respondent state-of-agent-engineering survey (LangChain, 2026) identifies quality, not cost, as the top barrier to production use and finds a wide gap between observability (nearly 89% adoption) and offline evaluation (52.4%).

可观测性与静默失败：行业调查表明，已部署智能体的主要失败模式不是崩溃，而是静默错误。Bessemer 的 2026 年基础设施报告（Wade 等，2026）估计“78% 的 AI 失败是不可见的”，而 LangChain 拥有 1,340 名受访者的智能体工程现状调查（LangChain, 2026）将质量而非成本列为投入生产使用的首要障碍，并发现可观测性（接近 89% 的采用率）与离线评估（52.4%）之间存在巨大差距。

The architecture analyzed here gives operators visibility into tool calls, hooks, and session transcripts; closing the evaluation gap likely requires additional scaffolding (generator-evaluator separation, sprint contracts, and post-hoc checks of the kind discussed in Rajasekaran (2026)) rather than model improvements alone.

本文分析的架构使操作人员能够对工具调用、钩子和会话转录文本保持可见性；要弥合评估差距，可能需要额外的脚手架（如生成器-评估器分离、冲刺契约以及 Rajasekaran (2026) 中讨论的那种事后检查），而不仅仅依赖于模型自身的改进。

Governance. Broader governance trends will constrain the design space as agents become more autonomous. The International AI Safety Report (Bengio et al., 2026) warns that “AI agents pose heightened risks because they act autonomously, making it harder for humans to intervene before failures cause harm,” and the MIT AI Agent Index (Staufer et al., 2026) finds that only 13.3% of indexed agentic systems publish agent-specific safety cards.

治理：随着智能体变得更加自主，更广泛的治理趋势将限制其设计空间。《国际 AI 安全报告》（Bengio 等，2026）警告说，“AI 智能体带来了更高的风险，因为它们是自主行动的，这使得人类很难在失败引发损害之前进行干预”，而麻省理工学院（MIT）的 AI 智能体指数（Staufer 等，2026）发现，被索引的具备智能体特性的系统中，只有 13.3% 发布了专门针对智能体的安全卡（safety cards）。

Emerging regulatory frameworks, notably the EU AI Act (fully applicable August 2026) and evolving copyright jurisprudence around AI-generated code, may impose external constraints on logging, transparency, and human oversight that shape how coding agent architectures evolve.

新兴的监管框架，特别是欧盟的《AI 法案》（2026 年 8 月全面适用）以及围绕 AI 生成代码不断演进的版权法理，可能会对日志记录、透明度和人类监督施加外部约束，从而塑造编码智能体架构的演进方式。

Proactive architectures. The feature-gated KAIROS system illustrates how this architecture may evolve beyond reactive tool use. KAIROS implements a persistent background agent with tick-based heartbeats: when no user messages are pending, the system injects periodic prompts, and the model decides whether to act or sleep.

主动式架构：受特性门控的 KAIROS 系统展示 sales 了这种架构如何向超越被动式工具使用的方向演进。KAIROS 实现了一个带有基于 tick 心跳的持久后台智能体：当没有等待处理的用户消息时，系统会定期注入 <tick> 提示词，由模型决定是采取行动还是进入睡眠。

The design directly addresses a documented tension: proactive AI assistants increase task completion by 12 to 18% but reduce user preference at high frequencies (Chen et al., 2025). KAIROS resolves this through terminal focus awareness (maximizing autonomous action when the user is away, increasing collaboration when present) and economic throttling via SleepTool (each wake-up costs an API call; the prompt cache expires after five minutes of inactivity, making sleep/wake an explicit cost optimization).

该设计直接应对了一种被记录在案的张力：主动式 AI 助手能将任务完成率提高 12% 至 18%，但在高频交互时会降低用户的偏好度（Chen 等，2025）。KAIROS 通过终端焦点感知（在用户离开时最大化自主行动，在用户在场时增加协同）以及通过 SleepTool 进行经济节流（每次唤醒都会消耗一次 API 调用；提示词缓存会在非活动状态五分钟后失效，使得睡眠/唤醒成为一种显式的成本优化）来解决这一问题。

This binding of proactivity to both user presence and token economics is uncommon among production agent systems, though KAIROS cannot be confirmed as active in production builds.

这种将主动性与用户在场状态及 Token 经济学双重绑定的设计在生产级智能体系统中并不多见，尽管目前无法确认 KAIROS 在生产构建版本中是否处于激活状态。

11.7 Recurring Design Choices

11.7 反复出现的设计选择

Reading the six subsystem analyses together reveals three cross-cutting design commitments that recur across otherwise independent components.

将这六个子系统的分析结合起来阅读，可以揭示出三种跨领域的设计承诺，这些承诺在其他相互独立的组件中反复出现。

Graduated layering over monolithic mechanisms. Safety, context management, and extensibility all use graduated stacks of independent mechanisms rather than single integrated solutions.

渐进式分层胜过单体机制：安全性、上下文管理和扩展性都使用了由独立机制构成的渐进式堆栈，而不是单一的集成解决方案。

The permission architecture layers seven stages from tool pre-filtering through deny-first rules, permission modes, the auto-mode classifier, shell sandboxing, non-restoration on resume, and hook interception. Context management layers five compaction stages, lazy-loaded CLAUDE.md files, deferred tool schemas, and summary-only subagent returns. Extensibility layers four mechanisms (MCP servers, plugins, skills, hooks) at different context costs (Section 6).

权限架构分层叠置了七个阶段：从工具预过滤，到默认拒绝规则、权限模式、自动模式分类器、Shell 沙箱化、恢复时不恢复权限，以及钩子拦截。上下文管理分层叠置了五个压缩阶段：延迟加载的 CLAUDE.md 文件、延迟工具模式（deferred tool schemas）以及仅返回摘要的子智能体。扩展性则按照不同的上下文成本分层叠置了四种机制（MCP 服务器、插件、技能、钩子）（第 6 节）。

In each case, the design trades simplicity and debuggability for defense in depth, accepting that the interaction between layers can produce emergent behaviors difficult to predict from any single configuration.

在每种情况下，该设计都牺牲了简单性和可调试性，以换取纵深防御，并接受了层与层之间的相互作用可能会产生难以仅从单个配置中预测的涌现行为。

Append-only designs that favor auditability over query power. Session transcripts are append-only JSONL files with read-time chain patching; permissions are not restored across session boundaries; context compaction applies read-time projections over a full history rather than destructive edits.

偏好可审计性胜过查询能力的仅追加设计：会话转录文本是具有读取时链条修补（read-time chain patching）功能的仅追加 JSONL 文件；权限不会跨越会话边界进行恢复；上下文压缩对完整历史记录应用的是读取时投影（read-time projections），而不是破坏性的编辑。

This commitment recurs because it preserves the ability to resume, fork, and audit sessions without modifying previously written state. The cost is that richer structured queries (“show me all tool calls that modified file X across sessions”) require post-hoc reconstruction rather than direct lookup.

这一承诺之所以反复出现，是因为它在不修改先前已写入状态的情况下，保留了恢复、派生和审计会话的能力。其代价是，更丰富的结构化查询（例如“显示跨会话修改了文件 X 的所有工具调用”）需要事后重新构建，而无法进行直接查找。

Model judgment within a deterministic harness. Across all subsystems, the architecture trusts the model’s judgment within a rich deterministic harness rather than constraining its choices.

在确定性工具内部的模型判断：在所有子系统中，该架构都信任模型在丰富的确定性工具内部所做出的判断，而不是限制它的选择。

The estimated 1.6% decision-logic ratio captures this quantitatively: the harness creates conditions (tool routing, permission enforcement, context assembly, recovery logic) under which the model can decide well.

估计仅为 1.6% 的决策逻辑比例从量化上体现了这一点：工具创造了能够让模型做出良好决策的条件（工具路由、权限执行、上下文组装、恢复逻辑）。

Hierarchical permissions preserve safety invariants across agent boundaries, and assembleToolPool() merges built-in and MCP tools into a single unified interface, but the model retains full latitude over which tools to invoke and in what order. The trade-off is that good local decisions can produce poor global outcomes when bounded context prevents global awareness, as the empirical predictions of Section 11.4 document.

层级化权限跨越智能体边界维持了安全不变性，且 assembleToolPool() 将内置工具和 MCP 工具合并为了单个统一的接口，但模型对于调用哪些工具以及按何种顺序调用仍保留了充分的自由度。其权衡在于，正如第 11.4 节的经验预测所记录的那样，当有限的上下文阻碍了全局感知时，良好的局部决策可能会导致糟糕的全局结果。

12.1 Silent Failure and the Observability–Evaluation Gap

12.1 静默失败与可观测性-评估差距

Whether the observability–evaluation adoption gap reported in Section 11.6 reflects a missing tooling layer, a missing evaluation interface inside the harness, or a model-capability ceiling is not resolved by the sources cited there. How the silent-mistake failure mode noted in that paragraph should be surfaced is therefore an architectural question for the harness rather than a capability question for the model.

第 11.6 节中报告的可观测性与评估采用率之间的差距，究竟是反映了工具层的缺失、工具内部评估接口的缺失，还是模型能力的上限，该处所引用的文献并未给出定论。因此，如何让该段中提到的静默错误（silent-mistake）失败模式浮出水面，是一个属于工具架构的问题，而不是一个属于模型能力的问题。

Recent empirical work characterises the gap at several resolutions. Cemri et al. (2025) catalogue fourteen failure modes spanning system-design issues, inter-agent misalignment, and task verification; Pathak et al. (2025) build a benchmark of agent trajectories specifically for anomaly detection in traces; Yao et al. (2024) expose consistency gaps via the passk metric (the probability that all k independent trials succeed); and Kapoor et al. (2024) argue that current agent benchmarks lack holdouts and cost controls, limiting what observability can actually diagnose.

近期的经验性工作从多个分辨率对这一差距进行了表征。Cemri 等 (2025) 将涵盖系统设计问题、智能体间失配以及任务验证的 14 种失败模式进行了编目；Pathak 等 (2025) 专门针对追踪日志（traces）中的异常检测构建了智能体轨迹基准；Yao 等 (2024) 通过 $pass_k$ 指标（所有 $k$ 次独立试验均成功的概率）揭示了内部一致性差距；Kapoor 等 (2024) 则指出当前的智能体基准缺乏留存测试集（holdouts）和成本控制，从而限制了可观测性实际能够诊断的范围。

Against the permission pipeline and tool-orchestration layers analysed in Sections 4 and 5, two architectural questions remain open. First, whether the scaffolding the paper cites from Rajasekaran (2026) (generator–evaluator separation, sprint contracts, post-hoc checks, building on Madaan et al. (2023)’s self-refine pattern) belongs inside the harness (e.g., as additional hook events alongside the 27 documented in Section 6) or outside it as a separate evaluation layer is not settled by the cited sources.

对照第 4 节和第 5 节分析的权限流水线和工具编排层，仍有两个架构问题悬而未决。第一，本文引用的来自 Rajasekaran (2026) 的脚手架（生成器-评估器分离、冲刺契约、事后检查，基于 Madaan 等 (2023) 的自我完善［self-refine］模式）究竟应该属于工具箱内部（例如，作为第 6 节中记录的 27 个钩子事件之外的附加钩子事件），还是作为独立的评估层置于其外部，所引用的文献对此并未达成共识。

Second, whether the existing hook pipeline of Section 6 can host such scaffolding within its current context-cost envelope is a further open question. The observation that closing this gap “likely requires additional scaffolding …rather than model improvements alone” (Section 11.6) locates the open work at the harness layer.

第二，第 6 节中现有的钩子流水线是否能在其当前的上下文成本封包内托管此类脚手架，是另一个开放性问题。正如第 11.6 节所述，弥合这一差距“可能需要额外的脚手架……而不仅仅依赖于模型自身的改进”，这表明未完成的工作落在了工具层。

12.2 Persistence: Memory and Longitudinal Colleague Relationships

12.2 持久化：记忆与纵向同事关系

Whether agent state and the human–agent working relationship should persist across sessions, and in what form, is treated by the paper at two distinct layers today. Section 7 documents the four-level CLAUDE.md hierarchy and auto memory; Section 9 documents mostly-append-only JSONL transcripts (with explicit cleanup rewrites as an exception) whose session-scoped permissions resume does not restore.

智能体状态以及人机工作关系是否应该跨会话持久化，以及以何种形式持久化，本文目前在两个不同的层面上进行了处理。第 7 节记录了四级 CLAUDE.md 层级结构和自动记忆；第 9 节记录了绝大部分为仅追加的 JSONL 转录文本（显式清理重写除外），且在恢复（resume）操作时不会恢复这些会话作用域的权限。

What belongs between these two layers (durable state that is neither a static instruction nor a single session’s transcript) is an open design question. Hu et al. (2025) and Zhang et al. (2025a), already cited in Section 11.6, motivate an accumulating layer. Packer et al. (2023) reframes the LLM as an operating system with paged memory; Chhikara et al. (2025) builds a production-oriented memory store that survives restarts, while Xu et al. (2025) proposes a research agentic-memory design; Wang et al. (2024c) captures reusable procedural traces; Shinn et al. (2023) accumulates self-reflection traces via verbal reinforcement across attempts; and surveys by Zhang et al. (2025b) and Huang et al. (2026) map candidate mechanisms.

在这两个层级之间应该存在什么（一种既不是静态指令，也不是单个会话转录文本的持久状态），是一个开放的设计问题。第 11.6 节中已经引用的 Hu 等 (2025) 和 Zhang 等 (2025a) 推动了一种累积层（accumulating layer）的发展。Packer 等 (2023) 将 LLM 重新框架为具有分页记忆的操作系统；Chhikara 等 (2025) 构建了一个能够经受重启的面向生产的记忆存储，而 Xu 等 (2025) 提出了一种研究型的智能体记忆设计；Wang 等 (2024c) 捕捉了可重用的程序追踪痕迹；Shinn 等 (2023) 通过跨尝试的语言强化（verbal reinforcement）来累积自我反思追踪；Zhang 等 (2025b) 和 Huang 等 (2026) 的综述则梳理了候选机制。

The same persistence question recurs on the human side. Section 11.6 already cites longitudinal autonomy evidence (Huang et al. (2025), McCain et al. (2026)); Dell’Acqua et al. (2025)’s field experiment with 776 Procter & Gamble professionals, together with longitudinal and organisational studies of Copilot rollouts (Stray et al., 2025) and AI-teamwork trajectories (Xiao et al., 2025), report shifts in human–AI work dynamics as collaboration accumulates. Wang et al. (2023) illustrates an embodied agent that accumulates a skill library across tasks; Mollick (2024) frames the human–AI working relationship as co-intelligence.

同样的持久化问题在人类一侧也反复出现。第 11.6 节已经引用了纵向自主性的证据（Huang 等 (2025), McCain 等 (2026)）；Dell’Acqua 等 (2025) 对 776 名宝洁专业人员进行的实地实验，结合对 Copilot 推广的纵向与组织研究（Stray 等, 2025）以及 AI 团队合作轨迹（Xiao 等, 2025），报告了随着合作的积累，人类与 AI 工作动态所发生的转变。Wang 等 (2023) 展示了一个能跨任务累积技能库的具身智能体；Mollick (2024) 将人类与 AI 的工作关系框架为“协同智能（co-intelligence）”。

Whether a single substrate can carry both a user’s personal instruction hierarchy and a shared organisational context while preserving the file-based transparency of CLAUDE.md that Section 7 documents is an open architectural question. How session-scoped permissions interact with such a substrate, without reintroducing the resume-restoration concern that Section 9 closes as a deliberate safety choice, is a further open question.

单一基质（substrate）是否既能承载用户的个人指令层级，又能承载共享的组织上下文，同时还能保持第 7 节所记录的 CLAUDE.md 的基于文件的透明度，这是一个开放的架构问题。会话作用域的权限将如何与此类基质交互，而又不重新引入第 9 节作为刻意安全选择所关闭的“恢复-恢复权限”隐患，是另一个进一步的开放性问题。

12.3 Harness Boundary Evolution: Where, When, What, and with Whom the Agent Acts

12.3 工具箱边界演进：智能体在何处、何时、对什么以及与谁一起采取行动

Section 11.6 cites Rajasekaran (2026)’s observation that “the space of interesting harness combinations doesn’t shrink as models improve; it moves.” Whether that movement will be most pronounced in where the harness runs, when it acts, what it acts on, or with whom it coordinates is not resolved by the source-level analysis in Sections 3 to 9. Each of the four has an active research literature that the paper touches only in passing.

第 11.6 节引用了 Rajasekaran (2026) 的观察，即“随着模型的改进，有趣的工具箱组合空间并没有缩小；它在发生位移。”这种位移在工具箱运行的“何处”、采取行动的“何时”、作用对象的“什么”，还是协同配合的“与谁”上表现得最为明显，仅凭第 3 节至第 9 节中源码级别的分析是无法解决的。这四个方向各自都拥有活跃的研究文献，而本文只是略微提及。

Where. Martin et al. (2026)’s Managed Agents design virtualizes session, harness, and sandbox into independently replaceable interfaces, extending the virtual-memory analogy that Packer et al. (2023) applies to context-window management and that Karpathy (2023) popularizes more broadly; Khattab et al. (2023) treats the harness itself as a compile target.

何处（Where）：Martin 等 (2026) 的“托管智能体（Managed Agents）”设计将会话、工具箱和沙箱虚拟化为可独立替换的接口，延伸了 Packer 等 (2023) 应用于上下文窗口管理的虚拟记忆类比，以及 Karpathy (2023) 更广泛普及的概念；Khattab 等 (2023) 将工具箱本身视为编译目标。

When. Section 11.6 already introduces KAIROS as a feature-gated illustration, motivated by the +12%–18% task-pass gain that Chen et al. (2025) report and the sharp preference penalty (47% vs. 80–90%) restricted to the high-frequency Persistent Suggest variant. Liu et al. (2025), Pu et al. (2025), and Lee et al. (2025) extend the proactivity design space across programming and ambient-interface settings; Pasternak et al. (2025) and Sun et al. (2025) introduce benchmarks and training regimes aimed at sharpening it, and Deng et al. (2025) surveys the broader landscape.

何时（When）：第 11.6 节已经将 KAIROS 作为受特性门控的例证引入，这源于 Chen 等 (2025) 报告的 +12% 至 18% 的任务通过率收益，以及仅限于高频“持久建议（Persistent Suggest）”变体的剧烈偏好惩罚（47% 对比 80-90%）。Liu 等 (2025)、Pu 等 (2025) 和 Lee 等 (2025) 将主动性（proactivity）设计空间扩展到了编程和环境接口设置中；Pasternak 等 (2025) 和 Sun 等 (2025) 引入了旨在使其更加锐利的基准和训练机制，而 Deng 等 (2025) 对更广泛的领域进行了综述。

What. Vision-language-action work extends the harness beyond textual tool returns: Brohan et al. (2024) and Black et al. (2024) train VLA policies that execute physical actions, and Ahn et al. (2022) grounds plans in robot affordances; industry systems such as Figure AI (2025) and Bjorck et al. (2025) push similar ideas into humanoid control. These systems face the reversibility-weighted risk principle (Table 1) at a cost asymmetry that the principle names but does not quantify for non-textual actions.

什么（What）：视觉-语言-动作（VLA）工作将工具箱扩展到了文本工具返回结果之外：Brohan 等 (2024) 和 Black 等 (2024) 训练了执行物理动作的 VLA 策略，而 Ahn 等 (2022) 将规划植根于机器人的能力特性（affordances）中；诸如 Figure AI (2025) 和 Bjorck 等 (2025) 的工业系统正在将类似的想法推向人形机器人控制。这些系统面临着“可逆性加权风险原则”（表 1），且存在该原则提及但未向量化非文本行动的成本不对称性。

With whom. Role-differentiated multi-agent systems (Hong et al. (2023), Li et al. (2023), Chen et al. (2023), Qian et al. (2024)) compose agents with distinct responsibilities; multi-agent debate (Du et al., 2024; Liang et al., 2024) and graph-structured workflows (Zhuge et al., 2024) explore alternatives to the parent/subagent pattern of Section 8; Guo et al. (2024) surveys this space.

与谁（With whom）：角色分化的多智能体系统（Hong 等 (2023), Li 等 (2023), Chen 等 (2023), Qian 等 (2024)）组合了具有不同职责的智能体；多智能体辩论（Du 等, 2024; Liang 等, 2024）和图结构工作流（Zhuge 等, 2024）探索了第 8 节中父/子智能体模式的替代方案；Guo 等 (2024) 对该空间进行了综述。

Whether a single harness architecture can span all four extensions, or whether the “harness combinations” Rajasekaran (2026) describes will fragment into specialised stacks, is an open design question. The when-extension directly continues the Capability-versus-Adaptability tension in Table 4. The with-whom-extension partially maps onto Capability-versus-Reliability but raises cross-agent consistency concerns that Table 4 does not itself cover.

单一的工具箱架构是否能够跨越所有这四种延伸，还是说 Rajasekaran (2026) 所描述的“工具箱组合”会碎片化为专门化的技术栈，是一个开放的设计问题。“何时（when）”这一维度的延伸直接延续了表 4 中能力与适应性之间的张力。“与谁（with-whom）”这一维度的延伸在一定程度上映射到了能力与可靠性上，但引发了表 4 本身未涵盖的跨智能体一致性担忧。

The where- and what-extensions raise further questions the paper’s current subsystem boundaries do not cover: which governance obligations attach when harness components become hosted services (Section 12.5), and how reversibility-weighted risk (Table 1) scales to physical rather than textual effects. How these extensions compose across axes, rather than within any one, is not something the paper’s single-subsystem analyses can resolve.

“何处（where）”和“什么（what）”的延伸引发了本文当前子系统边界未涵盖的进一步问题：当工具箱组件变为托管服务时，应附加哪些治理义务（第 12.5 节），以及可逆性加权风险（表 1）如何扩展到物理效应而非仅仅是文本效应。这些延伸如何跨轴线（而非在单一轴线内）进行组合，并不是本文的单一子系统分析所能解决的。

12.4 Horizon Scaling: From Session to Scientific Program

12.4 视野缩放：从会话到科学项目

Section 2.1 defines Reliable Execution as spanning “both single-turn correctness and long-horizon dependability.” How the architecture documented in Sections 3, 4 and 7 to 9 (whose primary units are the turn, the session, and the sub-agent) continues to support long-horizon dependability as autonomous work extends beyond a single session is an open question.

第 2.1 节将可靠执行（Reliable Execution）定义为涵盖“单轮正确性与长视野可靠性（long-horizon dependability）”。随着自主性工作延伸到单个会话之外，第 3、4 节以及第 7 至 9 节中所记录的架构（其主要单位是轮次、会话和子智能体）如何继续支持长视野可靠性，目前仍是一个开放性问题。

A growing literature targets this regime. Lu et al. (2024) present an end-to-end autonomous research pipeline producing draft manuscripts; Beel et al. (2025) provide an independent SIGIR Forum evaluation of that pipeline, characterising what “autonomous research” currently delivers and where it falls short.

越来越多的文献开始针对这一领域。Lu 等 (2024) 提出了一个能够产出初稿的手稿端到端自主研究流水线；Beel 等 (2025) 对该流水线提供了独立的 SIGIR 论坛评估，表征了目前“自主研究”所能交付的成果以及其不足之处。

Gottweis et al. (2025) develop a multi-agent hypothesis-generation system that runs across days rather than turns, and Novikov et al. (2025) pursue algorithmic discovery over timescales that previously took human experts weeks. Kwa et al.’s METR study measures the task duration at which frontier agents succeed with fixed reliability (the 50%-time horizon) and how that horizon has evolved across model generations, giving an empirical frame for this scaling question.

Gottweis 等 (2025) 开发了一个多智能体假设生成系统，该系统可以跨越数天而非数轮运行；Novikov 等 (2025) 在以前需要人类专家数周的时间尺度上进行算法发现。Kwa 等的 METR 研究衡量了前沿智能体在固定可靠性下成功完成任务的持续时间（50% 时间视野），以及该视野在不同模型世代中是如何演进的，从而为这种缩放问题提供了一个经验性的框架。

Against the paper’s analysis, long-horizon deployment tests whether the context-management pipeline of Section 7, the last-assistant-text return policy of Section 8, and the append-only persistence of Section 9 remain sufficient when sessions compose into multi-session programs. Section 11.4 already frames this as “a directly measurable empirical question” that source-level analysis cannot resolve.

对照本文的分析，长视野部署检验了当会话组合成多会话程序时，第 7 节的上下文管理流水线、第 8 节的最后助手文本返回策略以及第 9 节的仅追加持久化是否仍然足够。第 11.4 节已经将此框定为一个“可直接衡量的经验性问题”，仅凭源码级别的分析是无法解决的。

Horizon scaling restates that question at the scale of weeks: whether the harness layer alone closes the gap, whether a cross-session memory substrate (Section 12.2) is required, or whether horizon-scale work demands coordination primitives beyond session, sub-agent, and memory, is not something the paper’s session-scoped analyses can settle.

视野缩放将这个问题重新放在了数周的时间尺度上：究竟是仅靠工具层（harness layer）就能弥合这一差距，还是需要跨会话的记忆基质（第 12.2 节），抑或是视野尺度的任务需要超越会话、子智能体和记忆之外的协调原语，这些都不是本文仅限于会话范围的分析所能确定的。

12.5 Governance and Oversight at Scale

12.5 大规模治理与监督

Emerging AI regulation adds an external constraint on the architectures that implement the Authority hierarchy of Anthropic, operators, and users documented in Section 2.1. Which logging, transparency, and human-oversight affordances coding-agent architectures should expose under that external constraint remains an open design question.

新兴的 AI 法规对实现第 2.1 节中记录的由 Anthropic、操作员和用户构成的权威层级（Authority hierarchy）的架构施加了外部约束。在这种外部约束下，编码智能体架构应该暴露哪些日志记录、透明度和人类监督功能，目前仍然是一个开放的设计问题。

The European Commission’s GPAI Code of Practice (European Commission, 2025a) and implementation guidelines (European Commission, 2025b) detail the general-purpose AI obligations that accompany the EU AI Act’s full applicability in August 2026; the MIT AI Agent Index (Staufer et al., 2026) and the International AI Safety Report (Bengio et al., 2026), already cited in Section 11.6, motivate the disclosure and oversight side of this constraint.

欧盟委员会的《通用人工智能实践守则》（GPAI Code of Practice，European Commission, 2025a）及其实施指南（European Commission, 2025b）详细规定了伴随 2026 年 8 月欧盟《AI 法案》全面适用而来的通用人工智能义务；第 11.6 节中已经引用的麻省理工学院（MIT） AI 智能体指数（Staufer 等, 2026）和《国际 AI 安全报告》（Bengio 等, 2026）推动了该约束中信息披露与监督一侧的发展。

The Bartz v. Anthropic ruling (bar, 2025) adds an input-side constraint on training-data sourcing (lawful acquisition of copyrighted works), distinct from the output-side copyright questions about AI-generated code that emerging cases address separately. An OECD report on AI governance frameworks (OECD, 2025) and an early analysis of compliance obligations for agent providers by Nannini et al. (2026) sketch what regulator-facing interfaces might look like without prescribing specifics.

Bartz 诉 Anthropic 案的裁决（bar, 2025）在输入端对训练数据来源增加了约束（合法获取受版权保护的作品），这不同于新兴案件单独应对的关于 AI 生成代码的输出端版权问题。经济合作与发展组织（OECD）关于 AI 治理框架的报告（OECD, 2025）以及 Nannini 等 (2026) 对智能体提供商合规义务的早期分析，勾勒出了面向监管机构的接口可能的样子，但未规定具体细节。

Read against the permission pipeline analyzed in Section 5, two properties of the current architecture are open under this constraint. First, the deny-first evaluation the paper documents is internally auditable through session transcripts (Section 9) but not yet externally auditable in the forms that emerging frameworks such as the GPAI Code of Practice (European Commission, 2025a) contemplate.

对照第 5 节分析的权限流水线，当前架构的两项特性在此约束下仍处于开放状态。第一，本文记录的默认拒绝评估虽然可以通过会话转录文本进行内部审计（第 9 节），但尚未能以诸如《通用人工智能实践守则》（European Commission, 2025a）等新兴框架所预期的形式进行外部审计。

Second, whether the values-over-rules principle, which the paper pairs with deterministic guardrails, admits the kind of explicit rule articulation that compliance review may call for is a further open question. Both properties lie within the harness rather than the model, which is where future architectures may need to expose new interfaces.

第二，本文中与确定性护栏相匹配的“价值高于规则（values-over-rules）”原则，是否允许合规审查可能要求的那些显式规则表述，是另一个开放性问题。这两项特性均存在于工具箱内部而非模型内部，这也是未来的架构可能需要暴露新接口的地方。

12.6 The Evaluative Lens Revisited: Long-Term Human Capability

12.6 再次审视评估视角：人类的长期能力

Section 2.4 introduces long-term human-capability preservation as an analytical lens rather than a co-equal design value; Sections 11.2 and 11.4 extend the lens with external evidence (perceived-versus-measured productivity, comprehension loss, complexity accrual, technical-debt persistence, neural-connectivity persistence, early-career hiring decline), and Section 14 pivots: “Future systems could treat that sustainability gap as a first-class design problem, not a downstream evaluation metric.” Whether that pivot is possible, and what architectural mechanisms a first-class treatment would require, is the last of the open questions this section records.

第 2.4 节将人类长期能力的保持引入为一个分析视角，而不是一个同等重要的设计价值；第 11.2 节和第 11.4 节用外部证据（感知生产力与实际测量生产力、理解力流失、复杂度累积、技术债残留、神经连接性持续减弱、初级职业招聘下降）扩展了这一视角，而第 14 节做出了转向：“未来的系统可以将这种可持续性差距视为一等设计问题（first-class design problem），而不是下游的评估指标。”这种转向是否可能实现，以及一等处理需要何种架构机制，是本节记录的最后一个开放性问题。

Two sub-questions separate the measurement gap from the design gap. First, whether the empirical claims that motivate the lens are measurable at session granularity. The existing citations operate at session to multi-month scales (Becker et al. (2025)’s 16-developer RCT, Shen and Tamkin (2026)’s comprehension-test comparison, Kosmyna et al. (2025)’s EEG study, He et al. (2025)’s 807-repository causal analysis, Liu et al. (2026)’s 304,000-commit audit, Rak (2025)’s hiring series), but the harness documented in Sections 3, 4 and 7 exposes no per-session signal for comprehension or convention drift.

两个子问题将测量差距与设计差距分离开来。第一，推动该视角的经验性主张是否在会话粒度上是可衡量的。现有的引用文献运行在从单次会话到数月不等的尺度上（Becker 等 (2025) 的 16 名开发人员随机对照试验、Shen 和 Tamkin (2026) 的理解力测试对比、Kosmyna 等 (2025) 的脑电图研究、He 等 (2025) 的 807 个代码库因果分析、Liu 等 (2026) 的 304,000 个提交审计、Rak (2025) 的招聘序列），但第 3、4 和 7 节中记录的工具箱并没有暴露任何关于理解力或规范漂移（convention drift）的单次会话信号。

Related work on programmer interaction modes (Barke et al., 2023) and AI-induced code-security regressions (Perry et al., 2023) sketches session-granularity measurement, and Aiersilan (2026) proposes a protocol for session-level cognitive-offloading probes. Second, whether architecture can respond to such measurements once they exist (an analogue of the generator–evaluator separation (Rajasekaran, 2026) applied to the human loop, comprehension-preserving surfaces, or mechanisms not yet named) is the design-gap question Section 14 poses.

关于程序员交互模式（Barke 等, 2023）和 AI 引发的代码安全回归（Perry 等, 2023）的相关工作勾勒出了会话粒度的测量，Aiersilan (2026) 提出了一种用于会话级认知卸载探测（cognitive-offloading probes）的协议。第二，一旦这些测量存在，架构是否能够对其做出响应（例如将 Rajasekaran (2026) 的生成器-评估器分离机制应用于人类循环、保持理解力的表面，或尚未命名的其他机制），这是第 14 节提出的设计差距问题。

The paper takes no position on which mechanism class is appropriate, and whether the harness documented here is even the right locus for that action (as opposed to the IDE, the organisation, or the human development loop) is a question the architectural analysis cannot adjudicate; the related work surveyed in Section 13 and the sustainability pivot of Section 14 mark where this paper leaves the question.

本文对哪种机制类别是合适的未做表态，而且此处记录的工具箱是否甚至是该行动的正确发生地（相比于 IDE、组织或人类自身的发展循环而言），是一个架构分析无法裁决的问题；第 13 节中综述的相关工作和第 14 节的可持续性转向，标志着本文对该问题的探讨至此为止。

13.1 Coding Agent Taxonomy

13.1 编码智能体分类学

AI coding tools can be organized by the degree of autonomous action they support (Table 5). Inline completion tools such as GitHub Copilot (Chen et al., 2021) suggest code fragments within the editor without autonomous action. Chat-integrated products including Cursor and Windsurf add conversational interaction and multi-file edits but remain coupled to the IDE environment.

AI 编码工具可以根据它们所支持的自主行动程度来进行组织（表 5）。行内补全工具（如 GitHub Copilot，Chen 等, 2021）在编辑器内提示代码片段，不具备自主行动能力。对话集成产品（包括 Cursor 和 Windsurf）增加了对话交互和多文件编辑功能，但仍与 IDE 环境相绑定。

Agentic CLI tools, including Claude Code, OpenAI’s Codex CLI, and Aider (Gauthier, 2024), operate from the command line and can autonomously execute shell commands, read and write files, and iterate on outputs within a single request. Fully autonomous systems like Devin, SWE-Agent (Yang et al., 2024), and OpenHands (Wang et al., 2024b) aim for minimal human supervision, often in sandboxed cloud environments.

具备智能体特性的 CLI 工具（包括 Claude Code、OpenAI 的 Codex CLI 以及 Aider［Gauthier, 2024］）在命令行中运行，并能够在一个单一请求内自主执行 Shell 命令、读写文件以及对输出进行迭代。完全自主的系统（如 Devin、SWE-Agent［Yang 等, 2024］和 OpenHands［Wang 等, 2024b］）则旨在实现最小化的人类监督，通常运行在沙箱化的云环境中。

Claude Code shares features with higher-autonomy agents (auto-mode classifier, background agent execution, remote environments) but retains interactive approval by default. Evaluation benchmarks such as SWE-Bench (Jimenez et al., 2023) and HumanEval (Chen et al., 2021) have driven much of the academic focus on coding agents. This paper examines Claude Code’s internal architecture from source code.

Claude Code 与更高自主性的智能体共享某些特性（自动模式分类器、后台智能体执行、远程环境），但默认情况下保留了交互式批准。诸如 SWE-Bench（Jimenez 等, 2023）和 HumanEval（Chen 等, 2021）等评估基准驱动了学术界对编码智能体的大部分关注。本文则通过源代码来审视 Claude Code 的内部架构。

13.2 Agent Architecture Patterns

13.2 智能体架构模式

Claude Code’s core loop follows the ReAct pattern (Yao et al., 2022): the model generates reasoning and tool invocations, the harness executes actions, and results feed the next iteration. Toolformer (Schick et al., 2023) demonstrated that language models can learn to use tools; Claude Code uses up to 54 built-in tools and a layered permission system.

Claude Code 的核心循环遵循 ReAct 模式（Yao 等, 2022）：模型生成推理和工具调用，工具工具箱执行行动，并将结果喂入下一次迭代。Toolformer（Schick 等, 2023）证明了语言模型可以学会使用工具；Claude Code 使用了多达 54 个内置工具以及一个分层的权限系统。

Q: 什么是ReAct 模式？

ReAct 模式（全称 Reasoning and Acting，即“推理-行动”模式）是大语言模型（LLM）智能体（Agent）领域最核心、最经典的一种思维与执行框架。

它由谷歌大脑团队与普林斯顿大学于 2022 年底联合提出。它的核心思想是：让大模型像人类一样，一边思考（Reasoning），一边采取行动（Acting），并在连续的“思考-行动-观察”循环中自主完成复杂的任务。

── 为什么需要 ReAct 模式？──

在 ReAct 出现之前，大模型的两种主流工作模式都有致命缺陷：

  1. 只思考，不行动（如 Chain-of-Thought, CoT 思维链）：大模型可以在脑子里进行复杂的逻辑推理，但它无法获取外部实时信息（比如无法查天气、无法读写本地代码库、无法调 API），一旦遇到不知道的知识就会胡思乱想（产生幻觉）。
  2. 只行动，不思考（如传统的 Tool-use / MRKL）：大模型一看到任务，就直接去调用各种外部工具。但这就像无头苍蝇一样，缺乏大局观和逻辑推理，面对极其复杂的组合任务时极易出错。

ReAct 的解法：将两者融合。大模型不仅在脑子里想（Reason），还把想完的结果变成具体的动作（Act）去查工具，拿到工具结果后再进行下一步思考，如此往复。

── ReAct 的核心运行循环（Thought-Act-Obs） ──

一个标准的 ReAct 智能体，其后台的提示词模板会强制大模型按照以下“三部曲”的格式，一行一行地进行自主循环迭代：

  • Thought（思考）：大模型先对当前面临的问题进行拆解、分析，决定下一步需要知道什么或做什么（“我需要先查一下 A 公司的股价”）。
  • Action（行动）：大模型输出一个具体的工具调用指令（例如：call_tool: google_search(query='A公司股价')）。
  • Observation（观察）：系统（代码环境）执行该工具，并将真实世界的返回结果（如“A公司今日股价为50元”）喂回给大模型的上下文。

大模型看到 Observation 里的新事实后，会立刻触发下一轮的 Thought -> Action -> Observation，直到它认为问题已经彻底解决，最后输出 Final Answer（最终答案）。

── 一个大白话实例 ──

用户提问： “帮我查一下今天东京的天气，并告诉我适合穿什么衣服。”

在 ReAct 模式下，大模型在后台的内心戏（输入输出流）是这样的：

​ Thought (思考): 用户想知道今天东京的天气。我无法直接获知实时天气，我需要使用网络搜索工具。

​ Action (行动): search_weather(location="Tokyo", date="today")

​ Observation (观察 - 系统返回的结果): “东京今天晴转多云，气温 12°C ~ 18°C，风力 3 级。”

​ Thought (思考): 我已经拿到了东京今天的天气数据。气温在 12°C 到 18°C 之间，属于体感偏凉的春秋过渡季节。在这个温度下，人类通常适合穿外套、卫衣或薄毛衣。我已经可以回答用户了。

​ Action (行动): submit_final_answer()

​ Final Answer (最终答案): 今天东京晴转多云，气温 12°C ~ 18°C。早晚体感较凉，建议您穿着：长袖衬衫/卫衣 + 一件防风外套（如夹克或风衣），方便根据气温随时穿脱。

── ReAct 模式的优缺点 ──

优点

  • 人类可读性极高（透明白盒）：你可以像看电影字幕一样，清晰地看到大模型每一步是怎么想的、为什么要去调这个工具。这也就是你阅读的 Claude Code 论文中所强调的“可解释性与安全审计”。
  • 极强的纠错能力：如果某一次调工具（Action）报错了，大模型在下一轮 Thought 中能看到错误信息，它会自己想办法换一个工具或者修正参数重新尝试。

缺点

  • Token 消耗巨大（非常贵）：因为每一次循环，系统都要把之前所有的思考、行动和返回结果重新拼接到上下文窗口里再喂给大模型。
  • 容易陷入死循环：如果外部工具持续返回意料之外的数据，大模型可能会卡在某个环节，不断进行无意义的 Thought-Act 循环，直到把 Token 彻底耗尽。因此，生产级的系统通常会硬性限制最大循环次数（例如最多循环 5-10 次）。

开发实践者的文章已经收敛到了少数几个周期性模式上，Claude Code 的架构正是这些模式的具象化。Anthropic 自身的《构建高效智能体》（Building Effective Agents，Schluntz 和 Zhang, 2024）将智能体与工作流区分开来，并主张采用简单的可组合模式，而不是沉重的框架。Martin (2026) 综合了在生产系统中观察到的七种模式，包括赋予智能体文件系统和 Shell 访问权限以作为通用的行动层，以及根据需求动态发现行动，而不是预先加载每个工具的模式（schema）。

Chase (2025) 观察到 Claude Code 的规划工具“基本上是一个空操作（no-op）”，其价值在于让智能体保持在正确的轨道上，而不是执行任何外部计算。Wang (2025) 指出，权威（authority）是学术框架最常遗漏的要素，并将信任称为生产级智能体设计中“最容易被忽视的元素”，第 5 节中的权限分析试图填补这一空白。Huyen (2025) 让复合错误（compound-error）的隐患具体化：在单步准确率为 95% 的情况下，一个包含 100 步的任务成功率仅为 0.6%，这正是我们在第 4 节和第 5 节中所追踪的单步验证模式的动因。

Zhang et al. (2025a) characterizes two failure modes that this design mitigates (summarization that drops domain details, and detail loss from iterative context rewriting), and instead proposes treating context as an “evolving playbook” that accumulates strategies over time. Claude Code’s approach is consistent with that framing, since the CLAUDE.md hierarchy accumulates structured instructions rather than repeatedly summarizing them. Hu et al. (2025) distinguishes context engineering from agent memory: context engineering handles transient assembly, while memory covers persistent factual knowledge and experiential traces. Claude Code’s architecture separates the two in the same way, pairing a compaction pipeline with a file-based memory hierarchy.

Zhang 等 (2025a) 表征了该设计所缓解两种失败模式（丢弃领域细节的摘要化，以及由于迭代式上下文重写导致的细节流失），并转而建议将上下文视为一个随时间累积策略的“演进式剧本（evolving playbook）”。Claude Code 的方法与这一框架相一致，因为 CLAUDE.md 层级结构累积的是结构化的指令，而不是重复对其进行摘要。Hu 等 (2025) 将上下文工程与智能体记忆区分开来：上下文工程处理瞬态组装，而记忆涵盖持久的事实知识和经验追踪。Claude Code 的架构以相同的方式将两者分离，将压缩流水线与基于文件的记忆层级结构相匹配。

Safety and permissions. Production coding agents adopt safety architectures that vary along three axes: approval model (per-action prompting, classifier-mediated automation, or no prompting with post-hoc review), isolation boundary (OS-level container, filesystem sandbox, permission-scoped tool pool, or none), and recovery mechanism (version-control rollback, session-scoped permission reset, or checkpoint-based rewind).

安全与权限：生产级编码智能体所采用的安全架构在三个轴线上有所不同：批准模型（逐项动作提示、分类器介导的自动化，或无提示配合事后审查）、隔离边界（操作系统级容器、文件系统沙箱、权限范围工具池，或无隔离）以及恢复机制（版本控制回滚、会话范围权限重置，或基于检查点的倒带）。

SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b) rely primarily on Docker container isolation, providing environment-level sandboxing that constrains all agent actions. Codex CLI supports sandbox modes and approval policies for shell commands. Aider (Gauthier, 2024) uses Git as its primary safety mechanism, making all changes reversible through version control. Claude Code combines per-action deny-first rules, an ML-based classifier for automated approval, optional shell sandboxing, and session-scoped permission non-restoration, layering multiple mechanisms rather than relying on a single isolation boundary.

SWE-Agent 和 OpenHands（Yang 等, 2024；Wang 等, 2024b）主要依赖 Docker 容器隔离，提供限制所有智能体行动的环境级沙箱化。Codex CLI 支持沙箱模式和针对 Shell 命令的批准策略。Aider（Gauthier, 2024）将 Git 作为其主要的特殊安全机制，通过版本控制使所有更改都具备可逆性。Claude Code 则结合了逐项动作默认拒绝规则、用于自动批准的基于机器学习的分类器、可选的 Shell 沙箱化，以及会话范围的权限不恢复，通过将多种机制分层叠加，而不是依赖于单一的隔离边界。

Protocols and extensibility. The Model Context Protocol that Claude Code uses as its primary external tool integration has become a de facto standard with a substantial ecosystem and a corresponding attack surface. Hou et al. (2025) catalogues thousands of community-developed MCP servers across 26 major directories and organizes MCP-specific threats into four attacker categories and sixteen scenarios, including tool poisoning, rug pulls, and cross-server shadowing. The permission and deny-rule machinery analyzed in Section 5 and the pre-filtering step in Section 6.2 can be read as the runtime side of the mitigations that survey calls for.

协议与扩展性：Claude Code 用作其主要外部工具集成的模型上下文协议（Model Context Protocol, MCP）已成为事实上的标准，拥有庞大的生态系统和相应的攻击面。Hou 等 (2025) 将跨越 26 个主要目录的数千个社区开发 MCP 服务器进行了编目，并将 MCP 特异性威胁组织为四个攻击者类别和十六个场景，包括工具投毒（tool poisoning）、跑路（rug pulls）和跨服务器阴影化（cross-server shadowing）。第 5 节中分析的权限与拒绝规则机制以及第 6.2 节中的预过滤步骤，可以被视为该综述所呼吁的缓解措施在运行时（runtime）的表现。

Software architecture. Layered architecture patterns (Garlan et al., 1993) inform our five-layer decomposition. Role-based access control models (Sandhu et al., 2002) provide theory for the permission mode system. Browser sandboxing (Reis and Gribble, 2009) is a similar per-process isolation approach. Multi-agent system theory (Wooldridge, 2009) helps explain subagent delegation.

软件架构：分层架构模式（Garlan 等, 1993）启发了我们的五层分解。基于角色的访问控制（RBAC）模型（Sandhu 等, 2002）为权限模式系统提供了理论支持。浏览器沙箱化（Reis 和 Gribble, 2009）是一种类似的每进程隔离方法。多智能体系统理论（Wooldridge, 2009）有助于解释子智能体的委派。

Positioning. Prior work on coding agents has focused on benchmarks (how well agents solve tasks), frameworks (how to compose agents), and products (what users can do). This paper contributes a source-grounded design-space analysis of a production coding agent, using source-level analysis and architectural comparison to surface design choices and trade-offs.

定位：以往关于编码智能体的工作主要集中在基准（智能体解决任务的能力如何）、框架（如何组合智能体）和产品（用户可以用它做什么）上。本文则通过源码级别的分析和架构对比，提供了一个针对生产级编码智能体的、基于源码的设计空间分析，以浮现出设计选择与权衡。

It draws on the software architecture case study tradition (Garlan et al., 1993) but applies it to an LLM-based agent by systematically identifying design questions, mapping alternatives, and contrasting Claude Code’s choices with those of OpenClaw, an independent AI agent system operating from a different deployment context.

它吸收了软件架构案例研究的传统（Garlan 等, 1993），但通过系统地识别设计问题、映射替代方案，并将 Claude Code 的选择与 OpenClaw（一个在不同部署上下文下运行的独立 AI 智能体系统）进行对比，将其应用到了基于 LLM 的智能体上。

14 Conclusion

14 结论

This paper shows that production coding agents can be understood as answers to a recurring set of design questions: where reasoning sits relative to the harness, how execution, safety, extensibility, context, delegation, and persistence are organized, and which trade-offs those choices encode.

本文表明，生产级编码智能体可以被理解为对一系列周期性设计问题的解答：推理相对于工具箱（harness）所处的位置，执行、安全性、扩展性、上下文、委派和持久化是如何组织的，以及这些选择蕴含了哪些权衡。

Claude Code occupies a clear design point within that space. It gives the model broad local autonomy while surrounding it with a dense deterministic harness for permissioning, tool routing, context compaction, extensibility, and session recovery.

Claude Code 在该空间中占据了一个清晰的设计基准点。它赋予了模型广泛的局部自主权，同时在其外围包裹了一层密集的确定性工具箱，用于权限控制、工具路由、上下文压缩、扩展性以及会话恢复。

Read through the five values and thirteen design principles identified in Section 2, these choices are coherent rather than ad hoc: the system consistently prioritizes human decision authority, safety, reliable execution, capability amplification, and contextual adaptability.

通过第 2 节中确定的五种价值和十三项设计原则来审视，这些选择是连贯一致的，而非临时的拼凑：系统始终优先考虑人类的决策权威、安全性、可靠执行、能力放大以及上下文适应性。

The OpenClaw comparison sharpens the main architectural finding by showing that the same design questions recur in different agent systems but produce different answers.

与 OpenClaw 的对比进一步锐化了核心架构发现，表明相同的设计问题在不同的智能体系统中会反复出现，但会产生不同的答案。

Where Claude Code invests in per-action safety classification and graduated context compression within a CLI harness, OpenClaw invests in perimeter-level access control and structured long-term memory within a multi-channel gateway. The two systems can even compose: OpenClaw hosts Claude Code as an external harness via ACP.

在 Claude Code 专注于 CLI 工具箱内的逐项动作安全分类和渐进式上下文压缩的地方，OpenClaw 则专注于多通道网关内的边界级访问控制和结构化长期记忆。这两个系统甚至可以进行组合：OpenClaw 可以通过 ACP 将 Claude Code 作为外部工具箱进行托管。

For agent builders, the most consequential open question is therefore not how to add more autonomy, but how to preserve human capability over time.

因此，对于智能体构建者来说，最具影响力的开放性问题不在于如何增加更多的自主性，而在于如何随着时间的推移保持人类自身的能力。

As the evaluative lens in Section 2.4, the analysis in Section 11, and the open questions surveyed in Section 12 document, the architecture provides limited mechanisms that explicitly preserve long-term human understanding, codebase coherence, or the developer pipeline. Future systems could treat that sustainability gap as a first-class design problem, not a downstream evaluation metric.

正如第 2.4 节的评估视角、第 11 节的分析以及第 12 节中调研的开放性问题所记录的那样，当前的架构在显式保持人类长期理解力、代码库连贯性或开发人员后备人才管道方面，所提供的机制非常有限。未来的系统可以将这种可持续性差距视为一等设计问题（first-class design problem），而不是下游的评估指标。

B.2 Design-Space Analytic Procedure

B.2 设计空间分析程序

Design questions were identified by examining each subsystem for recurring choice points where alternative designs exist in other production agents. Claude Code’s answers to each question were traced through specific source files and function implementations (Tier B evidence).

通过检查每个子系统中周期性出现的选择点（在其他生产级智能体中存在替代设计的地方），确定了设计问题。Claude Code 对每个问题的解答均通过特定的源文件和函数实现（B 层级证据）进行了追踪。

The five-value framework (human decision authority, safety, security, and privacy, reliable execution, capability amplification, and contextual adaptability) was identified from official documentation and creator statements (Tier A), then traced through thirteen design principles to architectural decisions.

由官方文档和创作者声明（A 层级）确定了五种价值框架（人类决策权威，安全性、保密性与隐私，可靠执行，能力放大，以及上下文适应性），随后通过十三项设计原则将其追踪至具体的架构决策。

Long-term capability preservation is treated separately as an evaluative lens rather than a design value, because it is not prominently reflected as a design driver in the architecture or in Anthropic’s stated values (Section 2.4). Token economics serves as a cross-cutting constraint that bounds all five values simultaneously, revealing how individual subsystem choices interact under shared resource pressure.

人类长期能力的保持被单独作为一个评估视角而非设计价值来处理，因为它在架构或 Anthropic 阐明的价值观中，并没有作为核心的设计驱动力得到显著体现（第 2.4 节）。Token经济学（Token economics）作为一个跨领域的约束条件，同时限制了所有这五种价值，揭示了在共享资源压力下，单个子系统的选择是如何相互作用的。

B.3 Limitations

B.3 局限性

  • Static snapshot. Analysis reflects one version (v2.1.88). Feature flags (e.g., TRANSCRIPT_CLASSIFIER, CONTEXT_COLLAPSE) create build-time variability; different build targets may produce functionally different applications.

    静态快照：分析仅反映了单个版本（v2.1.88）。特性标志（例如 TRANSCRIPT_CLASSIFIER、CONTEXT_COLLAPSE）造成了构建时的可变性；不同的构建目标可能会产生功能截然不同的应用程序。

  • Reverse-engineering epistemology. Source code reveals implemented structure, control flow, dependencies, and feature gates. It cannot confirm design intent, enabled production flags, runtime prevalence, or unshipped behavior.

    逆向工程认识论：源代码可以揭示出已实现的结构、控制流、依赖关系和特性门控。但它无法确认设计意图、生产环境中实际启用的标志、运行时普及率或未发布的行为。

  • Single-system analysis. Findings describe Claude Code’s design space, not the entire design space of coding agents. Generalizations are bounded.

    单一系统分析：研究结果描述的是 Claude Code 的设计空间，而不是编码智能体的完整设计空间。泛化能力是受限的。

  • OpenClaw snapshot. The OpenClaw analysis reflects a specific development state and may not represent

    OpenClaw 快照：对 OpenClaw 的分析反映了其特定的开发状态，可能无法代表
