Agent Skills

Agent Skills turn any general- purpose agent into a specialist on demand. No context bloat. Portable and lightweight.

智能体技能（Agent Skills）可根据需求将任何通用智能体转化为专家。无上下文膨胀。便携且轻量。

附录A中的一段，讲的很好，提前截图放这里！技能其实就是之前老板要求编写“岗位职责SOP文档”的程序化。 实现组织知识的沉淀（技能库，是活的知识库）！而技能库，则编码了这家公司多年来积累沉淀的业务范式。这是本白皮书的核心战略观点。

1、引言

Introduction

智能体技能是一种为你的智能体装备知识和公司上下文的方法。一个智能体技能是一个包含 SKILL.md 文件以及 scripts/、references/ 和 assets/ 目录的文件夹。第 2 节将详细介绍其具体解剖结构。

智能体技能正成为跨平台便携性的标准。 但为何采用速度如此突飞猛进？我们认为智能体技能解决了 AI 智能体开发中的四个主要摩擦点：

  1. 指令过多，效果变差。将你能想到的每条指令都倾倒进单个系统提示词中，不可避免地会降低大语言模型（LLM）的性能，这一问题被称为上下文腐烂（context rot）。技能通过仅在需要时加载来解决此问题。第 5 节将展开探讨这背后的研究。
  2. 知道如何做，而不仅仅是知道是什么。LLM 在记住发生了什么（情景记忆）和记住事实（语义记忆）方面已经有了相当合理的类似物。它们所缺乏的是一种记住如何逐步做事的方法，这被称为程序性记忆。智能体技能可以被视为大模型智能体第一个可靠的程序性记忆原语。
  3. 多智能体超载。生态系统中充斥着复杂的多智能体系统，这些系统众所周知地难以构建和维护。虽然在某些任务中它们仍然是必需的，但技能允许单个通用智能体无缝地变身为多种专家角色。 第 3 节将结合一个实际示例深入阐述这一论点。
  4. 便携性。一个带有 markdown 文件的文件夹是一个极其轻量级的原语。 任何具有文件系统访问权限的智能体都可以使用它们，这使得它们在多供应商的 AI 格局中具有完美的便携性。

在本白皮书中，我们迎合两类角色：构建者（使用技能的人）和开发者（创建、版本化和管理技能的人）。在深入探讨评估、生产就绪、元技能和组合等复杂的开发者话题（第 4 至 8 节）之前，我们将简要介绍什么是技能以及如何使用它（第 2 至 3 节）。

2、什么是智能体技能（以及如何构建你的第一个技能）

What is an Agent Skill (and How to Build Your First One)

智能体技能是一种赋予通用智能体按需专家能力的原语。是的，一个技能可以简单到只有一个 markdown 文件，但它不必止步于此，且其背后的范式相当具有创新性。

今天，我们看到智能体技能通过两种不同的模式涌现：

第一条路径由领域专家驱动， 他们已经在某处记录了机构知识。想想一个拥有 30 页操作手册的合规官，或者一个拥有新员工入职指南的 HR 经理。他们都不需要学习编程来编写和开始使用技能。他们已经拥有了内容；剩下的唯一工作就是将其转化为智能体能够聪明使用的格式。

第二条路径涉及开发者将智能体或编码的工作流包装成技能。 如果智能体成功执行了一个非平凡且重复性的任务，你不会希望它下次还要从头开始摸索流程。相反，你希望智能体利用这次成功的运行创建一个技能。简而言之，我们正在观察一种新兴的模式：任何优秀的、可复用的工作流都会成为一个技能，而且你不需要自己编写它，由智能体来做，你来审核。这已经属于元技能的领域，我们在这里对其进行简要介绍，然后在第 6 节进行深入探讨。

两组人员产出相同的产物——一个以 SKILL.md 原语为核心的技能文件夹，但到达那里的旅程各有不同。

智能体技能剖析与渐进式呈现

Skill Anatomy & Progressive Disclosure

在深入探讨不同路径之前，我们先来审视一下智能体技能（Skill）的剖析结构。每一个技能都存放在独立的目录中，且必须包含一个 SKILL.md 文件。为了了解 agentskills.io 开放标准所定义的完整规范结构，让我们来看一个实际示例。

以下是一个专为执行“咖啡店每日准备工作”而设计的技能文件夹示意图（请记住，唯一强制要求的文件是 SKILL.md，其余均为可选）：

代码片段 1：展示 cafe-preparation 技能标准布局与渐进式呈现设计的目录树结构。

其创新之处在于渐进式呈现（progressive disclosure）。技能分三个层级进行加载：

  1. 元数据（名称 + 描述）：始终保留在智能体的上下文窗口中。
  2. SKILL.md 正文：仅在技能被触发激活时才会加载。
  3. 绑定的资源：严格根据需要才会加载（且脚本在执行时绝不会污染或占用 Token 窗口）。

这意味着你哪怕安装了一百个技能，在日常的每一轮对话中，也只需要为它们极其微小的元数据支付少量的 Token 成本。接下来让我们投入实战，看看如何构建一个技能。

路径 A：将你已知的知识进行转化

Path A: Translating what you already know

附录 A 中提供了一个最简的 SKILL.md 模板，你可以直接复制。首先值得关注的核心部分是其 YAML frontmatter（头部配置），因为它是触发激活技能的开关：

代码片段 2：cafe-preparation 技能的 YAML frontmatter 配置。

从一开始就有两件事值得做对：命名（Naming）和描述（Description）字段。

命名。 在命名时，要做到显而易见且枯燥乏味：目录使用下划线蛇形命名法（如 bigquery_ingestion），技能名称使用中划线短语命名法（如 pdf-processing），并优先使用动名词形式（如 managing-databases）。避免使用诸如 utils 或 tools 等泛泛的名字，且省去所有内部行业黑话。

描述。 这就是你的路由算法。它是模型用来决定是否加载该智能体技能的唯一依据。明确陈述它能做什么，把触发关键词放到最前面；如果模型触发频率过低，语调要写得更有强迫性（pushy）；并显式说明它不适用于哪些场景。

一旦起草好 SKILL.md，你就可以开始构建该文件夹的其他部分了。这正是渐进式呈现开始发挥成效的地方——任何不需要写在 SKILL.md 正文里的内容，都可以放到其他地方：

  • Scripts（脚本）：确定性的工作（解析导出数据、数学计算、格式化）存放在 scripts/ 中。模型决定要做什么，而脚本负责执行繁重的工作。
  • References（参考资料）：只有在技能运行后才相关的知识（领域原则、定义、边界情况处理）存放在 references/ 中，并按需加载。
  • Assets（资产）：模板和 Schema 存放在 assets/ 中。

经验法则：如果 SKILL.md 开始变得冗长，那么接下来的段落大概率应该归入 references/，而不是留在正文里。

路径 B：将智能体刚完成的操作固化下来

Path B: Crystallizing what the agent just did

第二条路径从另一端切入。它不是去转化你已经拥有的东西，而是将智能体刚刚完成的操作固化下来。

当智能体成功完成了一项任务，你注意到该工作流是可复用的，并希望下一宗同类任务能够从这次刚学到的经验中获益。

这便属于元技能（meta-skills）的范畴：其职责在于捕捉或优化其他技能。像 Anthropic 的 skill-creator、Nous Research 的 Hermes Agent，以及 awesome-llm-apps 中自我提升智能体技能（self-improving-agent-skills）的范式都支持这种工作流。智能体观察一段成功的执行轨迹，并提出一份 SKILL.md 初稿。你只需负责审核，而无需亲自撰写。

这里适用同样的高质量标准。一个经过审核、迭代且由智能体起草的技能可以变得非常优秀；而一个未经审核的智能体起草技能，其效果往往比完全没有技能还要糟糕。我们将在第 6 节探讨元技能时，更深入地重新审视这整个话题。

尝鲜体验与当前如何安装技能

Trying it out & How to install Skills today

一旦文件夹准备就绪（无论是由你亲自编写，还是由智能体为你起草），就到了尝试体验的时候。将其放入对应工具的正确位置，重启智能体，并用句自然的提示词进行测试。观察运行轨迹（trace）以确认该技能确实被触发。接着，再尝试一个不该触发该技能的提示词，确认它保持静默。

但到底哪里才是“正确的位置”呢？

这正是事情变得有些微妙的地方，也是技能爆发式流行所带来的副作用之一。尽管每个智能体或编程工具都趋同于此格式，但它们各自寻找该格式的位置却略有不同。广义上，目前你与技能互动及安装技能主要有以下三种范式：

  1. 文件拖放（编程智能体与 CLI）：对于本地环境，这种模式是基于文件的。你只需将技能文件夹拖入或安装到特定的隐藏目录中，智能体便会自动识别。虽然目前围绕项目根目录下的共享 .agents/skills/ 文件夹正在形成一种极具群众基础的跨工具惯例，但许多工具依然保留着它们自己特制的路径。（高手提示：如果你在多个 CLI 工具和 IDE 之间反复切换，像 skillport 或 openskills 这样的社区管理器会自动将你的核心技能库以符号链接/软链接的形式映射到每个工具的预设位置）。
  2. UI 界面安装（Web 与企业工作区）：如果你使用的是基于 Web 的协作平台或消费级 AI 聊天机器人，你很少会接触终端或隐藏文件夹。这些平台允许你直接通过可视化的 UI 注册表，只需点击几下鼠标就能安装、上传或管理你的技能文件夹，并在后台为你的整个团队处理好路由分发。
  3. 编程式途径（自定义框架）：如果你是在从零开始构建定制化的非编程智能体（例如使用 Google Agent Development Kit），则需要通过代码以编程方式加载技能。你将代码指向该文件夹路径——例如通过 SkillToolset 类进行注册，它会在底层无缝地为模型自动生成所需的 load_skill 路由工具。

无论在何处，整体模式都是相同的：将技能文件夹放入正确的目录中，重启智能体，它就会将其加载。而“正确的目录”具体是什么，则取决于你所使用的工具。

一条建议：在做出假设前，请务必先查阅你所使用的特定编程智能体或 AI 聊天机器人的文档。格式是共享的，但安装路径、激活规则以及各个工具的细节（如允许工具的白名单、安全网关、插件捆绑等）却并非如此。

等等，智能体技能（Skill）与 MCP、AGENTS.md 究竟有什么不同？

Wait but how does a Skill differ from MCP and AGENTS.md?

为了厘清智能体技能（Agent Skills）在架构中的定位，对这些原语进行横向对比会大有裨益。

  • 技能（Skill）对比 MCP：两者并非竞争关系，而是组合关系。模型上下文协议（MCP）解决的是连接范围（Reach）问题：一个 MCP 服务器负责将智能体连接到外部系统（如 Google Drive、Salesforce、BigQuery 或内部 API）。而技能解决的是专业诀窍（Know-how）问题：它教会智能体如何去思考和处理某项特定的工作。当某个技能需要数据时，它会指示智能体去调用工具，通常是调用由 MCP 服务器提供的工具。
  • 技能（Skill）对比 AGENTS.md：两者的区别在于，AGENTS.md 在项目中是始终全量加载的，而技能是按需加载的。最优雅的架构是双管齐下：保持 AGENTS.md 极度精简（仅罗列项目规范、技术栈、构建命令等），并在必要时将其作为进入技能库的“路由表”，在底部附带一份简短的技能目录，告诉智能体有哪些技能可用。

3. 为什么智能体技能会如此迅速地风靡开来？

Why did Agent Skills become so popular, so fast?

想象一下 2025 年初。你被要求构建一个系统来分担重复性的后勤办公室工作：根据简报生成符合公司风格的幻灯片、解析结构化的 PDF 发票、起草 HR 入职指南、总结每周的合规报告，以及应对随着团队发现新的自动化需求而必然会不断增加的、长尾的同类任务。

在过去，你极大概率会默认采用多智能体（Multi-agent）架构：由顶层的一个路由智能体（Router Agent）负责将任务分发给底下的几个专家子智能体。接着，你会把痛苦的时间耗费在构建 CI/CD 流水线、编排逻辑上，并小心翼翼地确保新部署的 HR 子智能体不会弄崩溃发票子智能体。

而随着智能体技能（Agent Skills）的发布，这一工作流变得无比简单。这种开发摩擦也正是 Anthropic 最初创建技能格式的初衷，其最早的技能用于阅读 PDF 和创建幻灯片，代表了实现这一目标的轻量化版本。此时，你不再需要一个负责分发子智能体的路由器，而是拥有一个自带技能库的单智能体。技能可以运行命令、调用 MCP 服务器并捆绑 Python 脚本。智能体自行决定何时加载哪项技能。你维护的是技能，而不是智能体，这使得整体的运维工作面大幅缩减。

需要特别明确的是：智能体技能并没有杀死多智能体架构。

当面临真正的并行计算、明确的权限隔离界限（不同的访问权限、不同的安全策略、不同的外部系统）、抽象层级完全不同的分层解耦、对抗或相互制衡的设定、子智能体间的内部通信或异构模型组合时，多智能体依然是绝对正确的标准答案。此列表并未穷尽所有场景。

技能的功绩，在于填补了长期缺失的底层架构原语。许多以往默认采用多智能体构建的系统，现在可以通过设计，优雅地精简为“单智能体 + 技能库”的架构。

技能适用于聊天机器人、编程智能体，还是多智能体的企业级场景？

Skills for chatbots, for coding agents, or for multi-agent enterprise use cases?

最早公开露面的技能是以 AI 聊天机器人的形式出现的。但短短几天或几周内，属于编程智能体的叙事便宣告席卷而来。一旦交汇，技能便瞬间引爆。它们直接撞上了“氛围编码（Vibe-coding）”的热潮，并证明了这正是开发者们梦寐以求的格式。

在某些多智能体架构中，从定义上讲，每个子智能体本身就已经是专家了。一个调研智能体（Research Agent）可能并不需要一项调研技能（Research Skill）。在设计上，技能更适用于让一个通用智能体具备极高灵活性、从而能跨界变身为不同领域专家的场景。然而，多智能体与技能同样可以完美组合，例如让每个专用智能体都拥有其专属的作用域技能库。

现在，让我们来看一家物流公司的案例。该公司有 100 种流程变体，具体取决于产品类型、工具、路线限制、客户 SLA、监管区域等。如何才能优雅且轻量地解决这一问题：

  • 方案 A：单智能体 + 巨大上下文窗口：会导致立竿见影的上下文腐烂，并带来高昂的 Token 成本。
  • 方案 B：对操作手册进行 RAG（检索增强生成）：这在两年前或许是正确答案。但现在意味着你必须去运维一个向量数据库、一个嵌入模型和一套切片策略，而这些策略的质量与实际的操作流程毫无关系。
  • 方案 C：多智能体，每个流程配备一个子智能体：100 个子智能体，每个都带有特定流程的系统提示词。这简直是一场运维噩梦：意味着 100 次部署、100 个评估面以及极其复杂的路由层。
  • 方案 D：单智能体 + 100 项技能：这完美契合技能格式，因为同一项工作存在许多变体。技能的渐进式呈现意味着 100 项技能的成本仅为：$~100 \times 50$ 词元 = $~5,000$ 词元常驻加载的元数据。物流请求带有极强的触发线索（如 SKU、始发地、重量、危险品标签、SLA），这使得技能描述极其精准，选择高度可靠。操作流程存放在版本控制系统中。添加第 101 种变体只需新建一个文件夹，而不需要进行新的部署。更易于维护和扩展。

然而，最核心的部分是必须始终建立起严格的评估流程，并通过对比不同的实际表现来做出最终的技术决策（第 4 节将具体涵盖这种评估工作的工程落地细节）。

作为一条心得，技术的普及总是沿着阻力最小的路径前行。任何能够编写文档的人，都能够编写一项技能。这极大地降低了技术门槛，使得那些常年躺在 Wiki、操作手册和工程师大脑中的潜在流程知识，终于有了一个结构化的去处来汇聚流淌。

4. 评估技能

Evaluating Skills

现在你已经拥有了第一个智能体技能，或者已经建立了一个小型技能库。紧随其后的问题便是：你如何知道它们在实际中确实有效？本节将探讨技能是如何失效的、如何对其进行测试，以及每项技能在进入你的技能库之前必须通过的四项基本检验。

未经测试的智能体技能只是一种奢望，而不是一种能力。

当研究人员在 SkillsBench (2025) 中对 84 个真实世界的智能体任务进行基准测试时，他们发现有 19% 的任务在引入技能后的表现比没有技能时更差。这些设计拙劣的技能不仅是毫无用处的“噪音”，甚至在主动削弱智能体的能力。幸运的是，这些失败是完全可预测的，并主要归结为以下四种独立模式：

  1. 触发失败（Trigger Failure）：错误的技能被激活，或者正确的技能未能成功激活。
  2. 执行失败（Execution Failure）：技能正确触发，但产出了错误的结果或发起了错误的工具调用。
  3. Token 预算失败（Token Budget Failure）：体量庞大的技能正文挤爆了上下文窗口，导致非相关对话轮次中的模型性能出现下降。
  4. 能力退化（Regression）：新添加的技能与现有技能发生重叠，破坏了此前已经能够正常工作的路由机制。

触发失败会暴露在路由日志中；执行失败会表现在输出质量上；Token 预算失败会在现实的上下文负载下显现；而能力退化失败则只有在整个技能库联合运行时才会暴露。

  • 2. LLM-as-judge（大模型作为裁判）：通过基础自动化测试后，引入一个对等的大模型来扮演裁判，对技能输出的质量进行多维度打分。

阶段三：安全与对抗压力测试（红队防御）

  • 3. Adversarial probing（对抗性探测）：在技能进入生产环境前，进行系统化的“红队演练”。通过故意的语义改写、注入负向边界案例或具有迷惑性的反向触发短语，强行探测该技能会不会被坏人滥用，或者会不会在极端情况下失控崩溃。

阶段四：生产流量灰度验证（动态观测）

  • 4. Production split（生产流量分流）：通过了所有静态和离线测试后，技能开始接触真实的生产环境流量，此时流量被一分为二：
    • Canary 1%（金丝雀发布）：将 1% 的线上真实真实流量引入运行新技能的系统，并在后台（例如通过 selftune）严密监控 24 小时，观察其是否会造成意外的能力退化。
    • Shadow 100%（影子模式）：另外 100%（或全量）的流量在后台以“影子”形式并行运行。新技能虽然处理了这些流量并输出了结果，但结果并不真正返回给用户，而是用于在后台进行离线对比，验证新旧版本的表现差异。

触发是第一道关卡

The trigger is the first gate

一个从不触发的技能毫无用处，而一个触发范围过宽的技能则会注入无关的上下文。

Vercel 的生产分析显示，对于那些预期会稳定激活的技能，存在 56% 的未调用率。更关键的是，一个被剥离了指令的技能得分仅为 58%，而完全不带该技能的智能体得分却有 63%。这 5 分的负差距表明，设计拙劣的技能会主动产生负面减产效应。

在同一项研究中，Vercel 还指出，包含项目规范的被动 AGENTS.md 索引在 53% 的基准线之上实现了 100% 的通过率。这进一步印证了一个观点：技能最好留给狭窄、特定动作的工作流，而全局上下文则应当保留在被动的、随时可访问的文档中。

现在，为了达到行业标准的 90% 触发准确率，作为模型在路由期间唯一能看得到的依据，你的 SKILL.md 描述必须通过以下四项检查：

  • 可测试的特异性：你必须为其编写 3 个正向触发案例和 3 个反向触发案例。
  • 清晰度：模糊的查询不会与相邻的技能发生重叠。
  • 执行保真度：它描述的是实际的执行表现，而不是理想化的预期行为。
  • 句式重组稳定性：无论用户如何组织语言来表达意图，它都能保持一致的路由选择。

输出质量与工具调用轨迹

Output quality and tool trajectory

一旦技能触发，请将最终输出（智能体说了什么）与工具调用轨迹（智能体做了什么）分开独立进行测试。

在行动敏感型（action-allowed）技能中，错误的工具调用轨迹可能会引发不可逆的副作用。Latitude 的分析（2026 年 3 月）发现，仅对最终输出进行评分所通过的案例，要比具备轨迹感知（trajectory-aware）的评分多出 20% 到 40%。这个差距代表了智能体虽然通过了错误的工具调用序列，但最终凑巧得出了正确答案的场景。这在只读场景中尚可接受，但在行动敏感型技能中则是致命的。

Google ADK 评估框架提供了三种轨迹评分模式：EXACT（完全一致的顺序）、IN_ORDER（有序子集）以及 ANY_ORDER（无序子集）。轨迹校验应当与技能的层级相匹配：只读型技能可以使用 ANY_ORDER，而行动敏感型技能则需要 IN_ORDER 或 EXACT。

一、 左侧：三种可验证的工具调用轨迹（Tool Trajectories）

当一个复杂的任务需要智能体依次调用工具 $A$、$B$、$C$ 来完成时，评估系统会通过以下三种不同的宽松度来定义“正确的运行轨迹”：

  1. Exact trajectory（精确轨迹）
    • 逻辑：智能体必须严格、毫无偏差地按照 $A \rightarrow B \rightarrow C$ 的顺序执行。
    • 场景：适用于有着极其严密逻辑或安全合规要求的业务流（例如：必须先通过鉴权 $A$，才能读取数据 $B$，最后才能发起扣款 $C$）。多一步或少一步都算评估失败。
  2. In-order trajectory（按序包含轨迹）
    • 逻辑：主干核心步骤的先后顺序不能错（$A$ 必须在 $B$ 前面，$B$ 必须在 $C$ 前面），但允许智能体在中间插入一些无关痛痒的辅助操作 $D$（如图中 $B$ 执行完后，智能体绕去调用了一下 $D$，再回来调用 $C$）。
    • 场景：适用于容错率稍高的流水线。允许智能体为了确保正确性而做出额外的探索（如查一下字典或多刷一次缓存），只要主体业务大方向按顺序走对即可。
  3. Any-order trajectory（任意顺序轨迹）
    • 逻辑：完全不在乎中间谁先谁后，只要在这轮对话中，工具 $A$、$B$、$C$ 都被成功调用过即可（图中用虚线双向箭头表示顺序无所谓）。
    • 场景：适用于并行、无依赖关系的任务。例如“同时查询北京、上海、广州三地的天气”，先查谁后查谁都不影响最终结果。

二、 右侧：Output-only: Blind spot（仅看输出的致命盲区）

右侧是一个巨大的红三角，写着 Unverified side-effects（未验证的副作用），周围环绕着杂乱无章的红线。

  • 什么是仅看输出（Output-only）？

    早期的传统大模型测试只做“端到端”的输入输出比对（给智能体一个问题，只要最终返回的文本结果和标准答案一致，测试就算通过）。

  • 为什么它是致命盲区？

    图中的红线代表智能体在后台实际运行的真实轨迹。可以看到，智能体并没有走正规的 $A \rightarrow B \rightarrow C$ 路径，而是在后台像没头苍蝇一样疯狂乱撞（执行了大量冗余、错误、甚至带有高风险的操作），甚至可能误删了数据库、向不该调用的人暴露了敏感隐私，最终阴差阳错地碰巧凑出了一个看起来正确的最终答案。

  • 代价是什么？

    如果你的评估套件只看结果（Output-only），就会把这个极其危险的智能体判定为“优秀”并推向生产环境。这会导致严重的上下文膨胀（Context Bloat）**和**无法预测的系统副作用（Side-effects）。

系统与技能：评估的幻觉

System vs. Skill: The Evaluation Illusion

轨迹测试评估的是宿主智能体与技能相互作用的复合系统，而非孤立的技能本身。当一个包含多技能的轨迹失败时，通常无法将智能体的路由能力、指令质量或执行保真度进行解耦。为了简化校准，请通过“单技能子智能体模式”（智能体 + 1 个技能 vs. 基础智能体）来评估技能；将复杂的多个技能共同加载留给高级生产暂存阶段。

Token 预算：孤立测试是个陷阱

绝不要纯粹在孤立状态下评估一项技能。生产环境中的智能体会同时共同加载 5 到 15 个技能。一个超过 5,000 词元的技能正文在单独运行时可能表现完美，但在共同加载时却会导致上下文腐烂。

复合评估陷阱：技能与智能体

The Compound Evaluation Trap: Skill vs. Agent

轨迹测试评估的是宿主智能体与技能共同作用的复合系统。如果测试失败，请避免针对特定模型对 SKILL.md 进行过度工程化（over-engineering），因为这会破坏其便携性。相反，应通过使用“双层断言框架”（Two-Tiered Assert Framework）将执行逻辑与路由隔离开来：独立验证底层工具代码，并在多个模型家族中审计 SKILL.md 触发器，以捕获脆弱的、与架构锁定的描述。

MCPVerse 指出，由于工具激增和上下文注意力竞争，Claude-4-Sonnet 的准确率下降了 18.2%。此外，Chroma Research (2025) 发现，所有前沿模型的性能都会随着输入内容的增加而下降，尤其是当受到共同加载的噪音干扰时。

图 4：观察单个运行技能与 15 个共同加载技能之间的性能差距。该曲线说明了为什么通过孤立测试对于生产就绪性来说是一个伪阳性结果。

正因如此，技能必须通过严格的权限层级进行晋升：

  • 只读（Read-Only）：大模型裁判（LLM-as-Judge）评估；90% 的触发准确率。
  • 仅限草稿-人工审核（Draft-Only / Human Review）：包含 20 多个案例的金牌数据集；人工批准。
  • 行动敏感型（Action-Allowed）：完整的对抗性红队测试；在多次运行中保持持续成功（而不仅仅是单次幸运通过）；无回滚事件；持续的 $pass^k$。

$pass^k$ 通过运行评估 $k$ 次并要求每次运行都成功，来衡量一致的而非偶尔的成功。在 tau-bench (Yao et al., 2024) 上，GPT-4o 在 $pass^1$ 上得分 61%，但在 $pass^8$ 上跌至 25% 以下，这证明单次运行的成功很难预测生产环境的可靠性。

在校准这些阈值时，有两个因素至关重要：

  1. 生产环境退化（Production Degradation）：ReliabilityBench 显示，与线下基准测试的 $pass@1$ 数据相比，生产环境的性能通常会下降 20% 到 30%。
  2. 模拟偏差（Simulation Bias）：基于模拟的评估可能会带来高达 9% 的乐观偏差（即“迷失在模拟中/Lost in Simulation”发现）。

因此，对代表性输出的人工审核依然是行动敏感型技能晋升的终极验证信号。

何谓“评估覆盖率”

What “eval coverage” means

一项技能通过满足直接映射到主要失败模式的四个条件，来实现完整的评估覆盖率：

  • 触发失败（Trigger Failure）：通过正向（应当触发）和反向（不应触发）测试案例验证触发行为。
  • 执行失败（Execution Failure）：确保在代表性的预期输入范围内产出正确的输出。
  • 能力退化（Regression）：确认添加该技能对现有技能库造成的性能下降为零。
  • Token 预算失败（Token Budget Failure）：限制技能的 Token 占用空间，以确保其不会降低非相关对话轮次中的性能。

该清单决定了技能的晋升；任何单一条件的失败都会使技能留在草稿层级，无论其在顺境路径（happy-path）下的表现有多好。一旦通过验证，该技能及其随附的评估套件即准备好进行生产部署（详见第 5 节）。

5. 从原型到生产

From Prototype to Production

第 1 节至第 4 节介绍了什么是技能、如何编写技能以及如何评估技能。本节则探讨当把一个可以运行的原型推向真实客户时，会发生哪些改变。简而言之：模型本身已不再是核心看点，而技能则成为了让你实现可靠交付的工程原语。

Google 在其智能体平台（Agent Platform）中推出的 Agents CLI，是一个用于在 Google Cloud 上构建、评估和部署 AI 智能体的命令行工具及技能包。智能体基于 Google 的智能体开发套件（ADK）构建，而 Agents CLI 则接管了围绕其展开的所有工作：脚手架搭建、评估、部署以及可观测性。

图 5：Agents CLI 的安装流程。只需一条 uvx 命令，就能将 7 项技能安装到开发者现有的编程智能体中，覆盖智能体的全生命周期（工作流、ADK 代码、脚手架、评估、部署、发布、可观测性）。同样的技能可以跨 Claude Code、Codex CLI、Antigravity 以及任何其他兼容的编程智能体运行。

这个实际运行的示例指明了三项超越 Google 自身生态、具备普遍意义的特性：

  • 核心专业能力留存在技能中，而非运行时（Runtime）。运行时已经变成了大宗商品（被平价商品化）；这 7 项技能才是沉淀下来的耐用资产。
  • 技能包可以与你现有的工具链相互组合。安装这些技能后，你现有的编程工具就会获得全新的能力。这也是企业内部应当追求的范式：能力应当组合进现有的工具链中，而不是再去造一个独立门户。
  • 全生命周期皆可作为技能交付。脚手架搭建、构建、评估、部署、发布、观测——曾经每个阶段都需要专属工具，现在全都融入了技能格式。

智能体运行时内部的真实构造

What’s actually inside an agent runtime

在框架底层，各家供应商的智能体循环（Agent Loop）已经趋于一致：运行时负责维护对话、调用模型、执行工具、读取文件并返回响应。

然而，在这些运行时内部，令人震惊的是关于推理的代码竟然如此之少。

对 Claude Code v2.1.88 推出的一项最新逆向工程（Liu, Zhao, Shang, and Shen, 2026）表明，代码库中 98.4% 的部分都是运营基础设施：权限分类器、上下文压缩流水线、子智能体委派、会话存储等，而只有 1.6% 的代码是智能体循环本身。模型位于远程 API 之后；而围绕模型展开的工程设计才是让系统达到生产级的关键所在。 配套网站 ccunpacked.dev 直观地映射出了这一相同的架构。

这就是下文所有内容的架构洞察核心。随着基础模型在基线推理能力上趋于同质化，自主可靠性的分水岭变成了围绕模型展开的确定性工程设计；而在该工程设计中，被组合与复用的基本单元正是“技能”。

图 6：演示到部署之间的鸿沟（The demo-to-deploy gap）。团队的信心在早期达到顶峰，但在与真实的客户环境接触时瞬间崩溃。人们本能地倾向于将其归咎于模型问题，但实际上几乎从来都不是。

为什么技能是能力提升的基本单元

Why skills are the unit of improvement

关于提升智能体能力，一种天真的理论认为更好的模型自然会带来更好的智能体。然而在生产环境中，模型只是基础设施，而技能才是让改进得以成功交付的原语。正如我们在第 1 节中所确立的，每一个新技能都是一个微小的、有明确归属且可测试的能力单元。当出现新的边界情况时，只需修改对应的一个 SKILL.md 文件即可；智能体的实际能力得以随之增长，而无需面对单体式提示词工程（Monolithic Prompt Engineering）带来的重重挑战。

技能的三大特性使其能够发挥作用：

  • 它们是有条件的（Conditional）：仅在技能描述与当前任务匹配时才会加载。
  • 它们是可组合的（Composable）：一个技能可以调用另一个技能的工具，或者向下游串联，而双方无需提前感知对方的存在（第 7 节将深入展开组合的故事）。
  • 它们是有归属的（Owned）：每个技能都存放在带有版本控制的文件夹中，并有明确的作者，因此能力的提升是分布式进行的，而不会被中央平台团队卡住脖子。

彻底摧毁 Demo 的失败模式：上下文溢出

The failure mode that breaks demos: context overflow

生产环境中智能体最常见的失败模式并不是幻觉，而是上下文溢出（Context Overflow）：模型接收到了超出其能高效利用的上下文信息，并在操作人员注意到之前就已经发生了隐性性能退化。两项研究为此提供了依据：

  • 迷失在中间（Lost in the Middle, Liu et al., TACL 2024）：在多文档问答和检索任务中，当相关信息位于输入内容的最前端或最末尾时，性能最高；而当信息处于中间位置时，性能就会发生退化。即使是针对长上下文进行过训练的模型，也同样呈现出这种 U 型曲线。
  • 上下文腐烂（Context Rot, Chroma Research, 2025）：在对包括 Claude 4 Opus、Claude 4 Sonnet、Gemini 2.5、Qwen3 在内的 18 款前沿模型进行测试后发现，随着输入内容的增长，即使任务难度保持不变，模型的性能也会发生退化。每一款模型都会变差，而且当相关内容与干扰信息难以区分时，退化速度更快。智能体真实上下文中典型的噪声（如工具输出、半相关的检索内容、中间推理过程）正是最糟糕的干扰源之一。

图 7：现实中的上下文腐烂。随着提示词体积的增长，在固定任务上的准确率会发生退化，这远远发生在上下文窗口被填满之前。虚线代表天真的预期，实线弯曲延伸则展现了 18 款前沿模型的真实表现（Liu et al. 2024; Chroma 2025）。

这对 Token 预算意味着什么

What this means for the token budget

第 2 节中介绍的渐进式呈现便是解决该问题的架构答案： startup 启动时仅加载每个技能的元数据，只有当描述匹配时才会加载某个技能的正文，而支撑文件则仅在正文引用它们时才会加载。

这笔账值得算一下。假设一个智能体拥有 50 个独立的业务工作流：

  • 如果作为单个系统提示词，它每一轮对话都需要加载 15,000 个词元。
  • 如果作为技能库，它只需要加载 ~4,000 词元的描述，加上当前激活的那一个技能的 ~2,000 词元正文，总计约 6,000 词元，而其余 49 个技能的正文则静静地留在磁盘上。

Anthropic 已经发布了相关示例，展示了将某个工作流转换为技能后，可将活跃上下文从大约 150,000 词元骤降至 2,000 词元，缩减幅度超过 98%。

图 8：Token 经济学：单个巨大的单体提示词对比包含 50 个技能的技能库。技能库有 50 个能力单元随时可用，但在任何给定时刻，只有处于激活状态的技能正文才会进入上下文中。

由此带来了三点实际启示：

  1. 容量（Capacity）是一个错误的指标：一个百万级（1M）的 Token 窗口，在内容达到 50K 词元时就可能表现出显著的性能退化。
  2. 活跃上下文是预算，而不是容器：模型眼前的每一个 Token 都会分走对其他 Token 的注意力。应当像基础设施团队对待内存那样对待系统提示词：将其视为一种需要进行刻意分配的有限资源。
  3. 技能化解了这一约束：它们在保持活跃上下文精简的同时，让可用的能力在事实上变得趋于无限。

一旦团队拥有了一个可以投入实用的技能库，核心问题便会从维持单一技能，转向技能的演进、组合以及更庞大的生态系统。

6. 关于元技能与自我提升型技能

On Meta-Skills and Self-Improving Skills

到目前为止，本文档中提及的每一项技能都是由人类编写的：领域专家坐下来，起草 SKILL.md，对其进行测试，然后交付。这是正确的起步方式。然而，一旦你拥有了一个可以投入实用的技能库，自然而然接踵而来的问题便是：智能体是否也能参与到技能的编写、评估和改进中来？

这便进入了元技能（meta-skills）的领域。这类技能的职责就是去创造、评估或优化其他技能。在实际工程落地中，这些“元技能”主要分为以下四个模块：

  1. 自动撰写（Authoring）：接收对某项工作流的描述，并直接产出一份 SKILL.md 初稿。Google 的 ADK 拥有一个“技能工厂（skill factory）”范式，正是通过其 SkillToolset 来实现这一点的；Anthropic 则随附了一个 skill-creator 技能，可引导你完成创建、评估和调优的全流程。
  2. 基于运行轨迹的辅助撰写（Assisted authoring from traces）：与其让渡给人类去凭空描述一套工作流，不如直接观察智能体成功执行几次任务的真实轨迹，随后将该轨迹固化提炼为一项技能。skill-creator 工作流通过基于轨迹的收割机制（trace-based harvesting）直接支持该操作。此时，人类的工作从“亲自编写技能”转变成了“确认被收割的版本是否准确捕捉到了正确的步骤”。

图 9：该分步循环演示了如何将真实的、成功的生产历史记录转化为可靠的程序性记忆，而无需人类进行手动起草。

  1. 定向优化（Improvement）：接收一项现有技能以及一组未通过的评估案例，并据此提出修改建议。Saboo 的 SkillOptimizer 以及 Anthropic 的描述优化循环（description-optimization loop）都属于此类范例。另一个典型案例是 Karpathy 的自动研究范式（autoresearch pattern）：智能体对目标文件提出修改，运行一个受控边界的实验，且只有在指标确实获得提升时才会保留该修改。

图 10：注意这里的评估门控机制。智能体可以对描述或指令提出修改建议，但除非通过单元测试，否则它无法将这些修改提交合并到技能库中。

  1. 技能库自主演进（Library evolution）：让技能库随着时间的推移自发增长，正如 Voyager 框架使其自身的 Minecraft（我的世界）技能库实现自主扩充那样。智能体完成了一项此前没有任何技能覆盖的任务，注意到自己刚刚解决了一个具有周期性重现特征的问题，便提议添加一项新技能来覆盖该场景。Schmid 的 self-learning-skill 是该模式在社区中的一个参考实现。

这种机制在什么情况下会崩溃

Where this falls apart

只有当你的评估套件足够过硬时，元技能才有可能奏效。

一个被允许自行修改技能的智能体，会非常乐意去迎合你指派给它的任何指标进行“盲目优化”——包括那些极易被钻空子的指标。第 4 节中所做的评估工作正是用来保持这种机制的纯洁性。如果缺乏扎实的触发准确率测试、能力退化测试以及人类的抽样点检，一个自主改进循环会在悄无声息中让你的技能库质量发生滑坡，而它在汇报中却还在声称自己正变得越来越好。

以下是几条经受住实践检验的研发习惯：

  • 草稿层级准入：任何由智能体撰写的内容进入技能库时，一律只能作为“草稿层（draft tier）”，无论元技能本身对它的信心有多高。它必须像人类编写的技能一样，经历第 4 节中所述的“只读 / 草稿 / 行动”晋升天梯。
  • 保持人类在场（Human-in-the-loop）：对于最初的几次修改，务必保持人类在场。即使指标有非常明确的提升，也请肉眼扫一眼代码差异（diff）。智能体极其容易犯下的那种错误（例如为了迎合几个测试案例而导致描述过拟合、或者弄崩溃了某个它根本不知道存在的下游技能），恰恰是人类在 30 秒内就能一眼看穿并拦截的。
  • 切勿毕其功于一役：不要一上来就搞元技能。先确保手动编写技能的循环能够正常跑通。想要以最快速度搞烂一个技能库，办法就是把智能体指向一个空文件夹，然后让它凭空生成 50 个技能。

未来的发展走向

Where this is going

目前的工程模式正逐渐沉淀为：由人类编写每个技能的第一个版本，元技能（meta-skills）接管重复性的维护工作（调优描述、添加测试用例、暴露能力退化情况），而少数前沿团队则开始尝试完全的自主扩展。最具看点的技术前沿正是这第三类场景——智能体根据其在生产环境流量中观察到的现象来提议构建全新的技能。这非常具有前景，但如果评估门控不够严密，这也是事情恶化得最快的地方。

7. 组合与打包技能

Composing and Packaging Skills

现实中的业务工作流往往无法完全塞进单个技能中。组合问题的核心在于：技能之间如何相互引用、如何传递状态，以及如何避免循环依赖。

在单体系统中将大模型的原始输出在孤立的技能之间倒手是低效的：状态会被模糊化，执行变得不可预测，且极难进行调试。智能体架构已经从早期的天真提示词链（prompt chaining）演进为了可预测的编排架构。

执行路由：DAG 编排

Execution Routing: DAG（Directed Acyclic Graph） Orchestration

早期的架构已被证明非常脆弱，一旦前置阶段发生幻觉，极易导致错误复合放大。目前的行业解决方案是有向无环图（DAG）编排：

  • 解耦的状态：DAG 架构中的状态路由并不依赖于在大模型提示词中累积执行历史。
  • 文件消息总线：DAG 控制器通过在子智能体节点之间传递结构化的 Schema 引用来编排任务交接。
  • 受保护的注意力：将有效载荷（payload）与模型的文本输入相剥离，可防止上下文窗口膨胀，从而保护模型的处理能力。

环境打包：能力轮廓

Environment Packaging: Capability Profiles

激活所有的技能会削弱自然语言路由的精准度，并挤爆上下文窗口。架构师应当利用工具来管理“能力轮廓（Capability Profiles）”，其作用类似于针对特定执行状态量身定制的专用角色（personas）。一个轮廓（profile）作为一个模块化的工具包，定义了：

  • 当前活跃的技能与工具访问权限。
  • 系统指令与运营护栏。
  • 自动化工作流与子智能体拓扑结构。
  • 大模型参数（如模型选择和 Temperature）。

在执行过程中，编排层会先卸载先前的系统指令并清空过期的变量，然后再将新的“能力轮廓”加载到内存中。这种严格的拆除与重构流程可有效防止上下文丢失。

这段话阐述了智能体系统设计中的一个核心痛点及其解决方案。简而言之，它讲的是：“不要指望让一个智能体同时学会所有事情，而是要像换卡带一样，根据当前的任务场景，动态为它切换不同的‘能力轮廓（Capability Profiles）’。”

我们可以从以下三个层面来深度拆解和理解这段话：

  1. 核心痛点：为什么不能“全量加载”？

“Activating every skill degrades natural language routing and overwhelms the context window.”

如果你的智能体有 100 个专业技能，你把它们在会话开始时全部激活（全部塞进提示词或上下文），会导致两个灾难性后果：

  • 路由退化（Routing Degradation）：技能描述本质上是靠大模型进行自然语言匹配的。当可选的技能太多时，模型会感到“困惑”，极易发生误触发或该触发时没触发。
  • 上下文窗口崩溃（Context Overwhelm）：所有的技能指令、规则、脚本描述会迅速吃满大模型的 Token 预算（也就是所谓的“上下文税”过于沉重），留给真正对话和推理的空间就所剩无几了。
  1. 解决方案：什么是“能力轮廓（Capability Profiles）”？

“Architects should utilize tools to manage ‘Capability Profiles,’ which function as specialized personas tailored to specific execution states.”

为了解决上述问题，架构师引入了“能力轮廓”这一概念。它就像是为智能体定制的“专业分身/角色卡”。智能体不是一个一成不变的超大单体，而是根据它当前所处的“执行状态”，变成一个特定领域专家。

一个“能力轮廓（Profile）”是一个极其严密的模块化工具捆绑包，它严格限定了以下四件事：

  1. 活跃技能与工具权限：只塞给它当前任务绝对需要的几个技能（例如：处于“退货状态”时，只加载 return-policy，把不相关的 materials-list 锁死）。

  2. 系统指令与运营护栏：切换对应的全局提示词和安全边界约束。

  3. 自动化工作流与子智能体拓扑结构：定义该状态下，它应该拉起哪些子智能体（Subagents）协助，它们之间怎么打交道（DAG 还是流水线）。

  4. 模型底层参数：针对不同任务切换不同参数。比如做“严谨计算”时切换到高精模型且调低 Temperature（随机性降为 0）；做“文案润色”时切换到轻量模型并调高 Temperature 以释放创意。

  5. 执行机制：无情的“卸载与彻底清空”

“During execution, the orchestration layer unloads previous system instructions and flushes stale variables before swapping the new Capability Profile into memory. This strict teardown and rebuild process prevents context loss.”

这一段指出了该架构在工程实现上的具体动作，它强调了“干净利落的内存更替”： 当用户的对话主题发生跳转（例如从“咨询淋浴间怎么贴瓷砖”变成了“帮我发起退款”），中层的编排器（Orchestration layer）会执行一套严格的拆除与重建（Teardown and Rebuild）流程：

  1. 无情卸载（Unload）：把前一个状态的系统提示词、死板的规则彻底从内存里扔出去。
  2. 彻底刷新（Flush）：清空所有不再需要的陈旧变量、历史中间缓存。
  3. 整体注入（Swap）：把全新的“退款专用能力轮廓”整体抬入内存（加载新技能、新参数）。

为什么要这么做？

如果只是不断往上下文里“追加”新指令，模型很快就会因为前后指令冲突、注意力涣散而产生上下文丢失（Context Loss）。只有通过这种类似“操作系统进程切换”的严格清理机制，才能确保智能体在每一轮不同的对话状态中，都保持最清醒的认知和最高的可靠性。

上下文债务与将智能性左移

Context Debt and Shifting Intelligence Left

技能会消耗模型的注意力，而注意力是一种稀缺资源。 当作者试图通过膨胀技能描述（例如使用“务必做某事/ALWAYS DO X”）来在运行时强行约束确定性行为时，他们就在累积上下文债务（Context Debt）。模型会逐渐学会忽略这些大写的命令式词汇，就像人类开发者会忽略一整墙无法阅读的错误警告文本一样。

工程上的最佳实践是将智能性左移（Shift Intelligence Left）。与其寄希望于大模型在运行时能够正确理解复杂的规则，不如将主观判断提炼为技能。通过将逻辑从大模型的提示词中剥离，转入标准的、可测试的脚本中，你可以缩减应用程序的无序表面积。

这段话阐述了智能体应用开发（LLM Application Engineering）中关于“提示词设计”与“工程架构”的一场核心变革。它警告开发者不要试图用“大力飞砖”的提示词去约束模型，而应该用传统的软件工程思维来重构系统。

我们可以将其核心思想拆解为两个对立的概念来理解：

  1. 现状与痛点：什么是“上下文债务（Context Debt）”？

  • 注意力是稀缺资源：大模型的上下文窗口（Context Window）虽然越来越大，但它的有效注意力（Attention）是稀缺且边际递减的。提示词越长，模型对中间核心指令的捕获能力就越弱。

  • 用提示词追求确定性的虚妄：很多开发者在发现智能体输出不符合预期时，习惯性地去“打补丁”——在提示词里疯狂追加大写加粗的限制条件（例如：“你必须始终、永远在第一步执行 X，千万不能做 Y！！”）。

  • 债务的代价（习惯性忽略）：

    这种为了强行追求“确定性行为（Deterministic behavior）”而不断膨胀提示词的做法，被称为上下文债务。当这种咆哮式的命令（Capitalized imperatives）堆砌得像一堵墙一样时，模型就会开始自动忽略它们，就像程序员在控制台里习惯性忽略成百上千条密密麻麻的报错和警告一样。这会导致智能体表现得更加混乱和不可控。

  1. 破局与最佳实践：什么是“智能左移（Shift Intelligence Left）”？

在传统软件测试领域，“测试左移（Shift Left Testing）”指的是把测试尽早放到开发流程的早期（左侧）去跑。而在智能体工程中，“智能左移”指的是：把复杂的、确定性的逻辑，从运行时的“大模型推理层（右侧）”剥离出来，提前固化到开发阶段的“工程代码与脚本层（左侧）”。

  • 不要赌大模型的运气（运行时逻辑）：不要在运行时（Runtime）去赌大模型能不能100%完美理解并执行你那长达万字的复杂规则。
  • 提炼为主观判断（Distill into skills）：只让大模型做它最擅长的事——主观意图识别和模糊逻辑判断。一旦大模型完成了这个“判定”，接下来的具体执行步骤（比如复杂的数据计算、严格的流程控制、格式校验）应该立刻脱离大模型。
  • 固化为可测试的脚本（Testable scripts）：把这些确定性的规则写成标准的 Python、R 或其他语言的自动化脚本。脚本的输入输出是 100% 幂等和确定的，而且可以通过传统的单元测试（Unit Tests）来进行覆盖。

核心心智总结

这段话给智能体架构师指明了一条清晰的工程分界线：

维度 糟糕的模式（累积上下文债务） 最佳实践（智能左移）

一句话理解：“上帝的归上帝，凯撒的归凯撒”。让提示词保持干净、轻量，把确定性的业务逻辑无情地赶出 Prompt，塞进传统的、可百分之百测试的代码脚本里。只有这样，才能消灭应用中那些无法预测的“混沌表面积”。

8. 如何在成百上千个现存技能中做出抉择

How to Decide Among the Hundreds of Skills That Exist

到 2026 年初，公共技能市场的上架量已突破 40,000 个，领先的平台报告称仅在 1 月份的前几周就发布了数万个新技能。在 Google Cloud Next 2026 上，Google 推出了其官方智能体技能仓库 [github.com/google/skills](https://github.com/google/skills)，这些技能可通过 npx skills install [github.com/google/skills](https://github.com/google/skills) 进行安装，以供 Antigravity CLI 及任何其他支持技能标准的编程智能体使用。Anthropic 技能仓库、Google ADK 技能库、Google 官方技能仓库以及诸如 awesome-llm-apps 等社区市场现在托管的技能数量，已经超出了任何从业者能够单独审核的极限。技能的选择问题是真实存在且日益严峻的。

以下三条启发式原则会有所帮助：

  1. 首先，对于特定供应商的工具，优先选择第一方技能。Google 的 BigQuery 技能、官方 Stripe 技能，以及任何由构建底层系统的人员编写的技能。它们会比社区的替代方案更准确，且维护更得当。
  2. 其次，锁定（pin）你所依赖的一切。社区技能会不断演进，一个未锁定版本的依赖可能昨天还能正常工作，明天就会失效。
  3. 第三，在采用前进行审计。技能是在你的上下文中运行的代码。请像对待任何其他依赖项一样对待它，保持相同的供应链卫生。

9. 结论

Conclusion

我们在本白皮书的开头探讨了一个惊人地简单的概念：一个包含 Markdown 文件和几个可选脚本的文件夹。然而，正是这个轻量级的结构——智能体技能（Agent Skill），正在从根本上重塑我们构建 AI 智能体的方式。它终于为基础模型提供了真正的、可测试的程序性记忆，允许它们记住如何一步一步地执行任务。通过依赖渐进式呈现的魔力，技能解决了上下文腐烂的问题。现在，单个通用智能体可以无缝地访问许多专业的工作流，而不会挤爆其 Token 预算。

我们在本文中不断提及的范式是：这种格式被刻意设计得很小，以便让有趣的工程围绕它展开，而不是在其内部发生。在现实的共同加载条件下评估技能是有趣的；在不将上下文窗口用作消息总线的情况下将技能编排进工作流中是有趣的；让智能体从成功的运行轨迹中起草技能、并由人类进行审核而非亲自撰写是有趣的；将二十年的机构知识编码进一个可版本化、可测试、可管理的技能库中是有趣的。由于这一原语的存在，所有这些在 12 个月前还无法解决的问题，现在都变得触手可及。

在整篇论文中，我们也试着明确指出哪些是已经定型的、哪些是仍在涌现的，以及哪些是可能会发生变化的。格式已经定型：agentskills.io 现已成为一项开放标准，并在每个主流的编程智能体、AI 聊天机器人和重要的智能体框架中得到了广泛采用。而围绕它的架构仍在涌现中：共同加载条件下的评估、技能库级别的优化、智能体驱动的技能创建，以及让这一切在大规模应用时保持安全的治理范式。

如果你今天打算起步，我们的建议与贯穿全文的观点一致：从小处着手，从你已经拥有的知识开始，将技能视作代码，衡量你交付的产物，并且在技能足以为你解决问题时，不要盲目追求多智能体架构。现在理清这一点的团队，将比那些等待行业共识成熟的团队构建出更整洁的系统。

五项核心基本原则

The five rules

  1. 一个技能，一件事。如果你无法用一句话描述该技能的作用，说明它包含两个技能。在动手编写前先进行解耦拆分。
  2. 描述即接口。智能体通过阅读描述来挑选技能。模糊的描述意味着技能将永远不被调用。
  3. 将技能视作依赖项。像对待类库一样对待它们：进行版本控制、锁定版本、在 PR（拉取请求）中进行代码审查。未经测试的技能只是一种奢望，而不是一种能力。
  4. 让正确的团队拥有正确的技能。领域专家拥有领域技能。不要让 AI 团队成为组织内部沉淀领域知识的瓶颈。
  5. 智能体运行时是可互换的。不要将技能与某一家运行时强行绑定。便携性是其核心价值的重要组成部分。

质量原则

The quality principles

  1. 自己先亲自跑一遍任务。真实的失败会产生有效的修正信号，而凭空推测只会产生杂音。

  2. 给出原因，而不仅仅是规则。当模型理解了一条指令存在的原因（Why）时，它们能奇迹般地很好泛化到各种边界情况中。如果你发现自己正用大写字母输入“务必（ALWAYS）”或“绝不（NEVER）”，请停下来，试着解释这背后的基本原理。

  3. 每一行内容都应当证明其存在的价值。保留关键陷阱、精确命令、业务逻辑和反模式；删去模型早已心知肚明的套话（例如“务必验证输出”）。

  4. 一个技能，一件事。如果描述中需要在两个无关的能力之间使用“和（and）”来连接，请将其拆分。

  5. 让指令具有可验证性。如果智能体自己都无法判断是否遵循了该规则，说明该规则过于模糊。

  6. 捆绑重复出现的部分。智能体需要不断重复推导的辅助代码，应当归入 scripts/ 中。

    (个人注：这个就是程序员对业务的优势！)

应当做与绝不能做

Do’s and Don’ts

应当做：

  • 从小处及具体场景着手。
  • 在描述字段上投入绝大部分精力。它是路由算法，必须包含做什么、何时用以及何时不用。
  • 信任渐进式呈现的机制。
  • 将确定性的工作捆绑在 scripts/ 中，而不是写成文本指令。
  • 将技能视作代码来管理。
  • 牢记：是“技能 + MCP”，而不是“技能 vs. MCP”。
  • 准备测试案例，并进行受控的、编程化的评估（具体参见第 4 节）。

绝不能做：

  • 编写形如“帮助处理文档”这类模糊的描述。请使用具体的动词、触发短语以及“何时不用”条款。
  • 编写超过 5,000 字的 SKILL.md 正文。请将细节移至 references/。
  • 硬编码文件路径或敏感密钥（Secrets）。
  • 嵌入更适合写在 AGENTS.md 中的“全局务必做 X”这类规则。
  • 在未进行安全扫描的情况下，盲目安装不信任的第三方库或技能。
  • 将 MCP 的职责错误地在 scripts/ 中重复造轮子。

技能的坏味道（出现以下特征时请及时修正）

Skill smells (revise if you see these)

  • 正文超过 5,000 字：这大概率包含了两个技能，或者这些参考材料应该存放在 references/ 中。
  • 两个领域团队可以同时拥有它：说明尚未解耦，请沿着团队的职责边界进行拆分。
  • 你无法为它编写出三个测试案例：说明描述过于模糊，或者该技能承担了太多职责。
  • 它没有引用任何其他资源：它可能只是一条长指令，更适合直接写在系统提示词中。
  • 你不得不一直在里面增加“边缘情况”章节：每个边缘情况可能都更需要属于它自己的专属技能。
  • 其描述以“一个用于…的帮助技能”开头：请重新编写。描述应当直接点名触发器、输入参数以及输出结果。

评估覆盖率清单

Eval coverage checklist

只有当以下四个条件全部满足时，一项技能才算通过了“完整评估”：

  • ☐ 触发（Trigger）：验证正向（应当触发）与反向（不应触发）的测试案例。目标达到 90% 的触发准确率。
  • ☐ 执行（Execution）：确保在具备代表性的预期输入范围内，能够产出正确的输出。
  • ☐ 能力退化（Regression）：确认引入该技能后，对现有技能库套件造成的性能下降为零。
  • ☐ Token 预算（Token budget）：与 5 到 15 个高频活跃的技能共同加载时，不会降低非相关对话轮次中的性能。

任何单一条件的失败都会使技能滞留在草稿层级，无论其在顺境路径（happy-path）下的表现有多好。

部署清单

Deployment checklist

  • ☐ Frontmatter 配置通过校验（Lint 检查通过）
  • ☐ 描述（Description）字段完整包含：做什么 + 何时用 + 何时不用
  • ☐ 脚本（Scripts）在 CI（持续集成）中通过了单元测试
  • ☐ 评估套件在 CI 中通过了最低达标阈值
  • ☐ 安全扫描结果无异常（无密钥泄露，无不信任的第三方依赖）
  • ☐ 描述字段经过了作者以外的第三方审核
  • ☐ 如果面向公共发布，跨工具的安装路径已通过测试
  • ☐ 组织级别的管理员配置已同步更新（如果适用）

一句话心智模型

One-line mental model

系统提示词（System prompt） = 本能。AGENTS.md = 项目 README。工具 / MCP = 双手。RAG = 图书馆。智能体技能（Skills） = 资深同事在你入职第一天递给你的操作手册，且 AI 永远不会遗忘。

明天从哪里开始

Where to start tomorrow

  1. 把你们团队中经验最丰富的专家拉到一边聊上一小时。请他们口述三个他们定期执行的日常工作流，并录音。
  2. 挑出重复频率最高的一个工作流。在不加载任何技能的情况下，你自己亲自跑一遍提示词，记录下智能体在哪里卡壳失效。或者使用类似的智能体协助。
  3. 根据录音转写文本起草一份 SKILL.md。在动手编写正文之前，先写出三个评估案例（两个正向触发，一个反向触发）。
  4. 将其交付到“只读层”。在类生产环境中进行测试。持续迭代描述字段，直到触发准确率突破 90%。

如此循环。 一次只针对一个工作流来构建你的技能库。切忌在第一天就指望大模型凭空为你生成 50 个技能。

附录 B – 案例研究：将领域专业知识转化为代码（零售业的垂直技能落地）

Case Study: Domain Expertise as Code (Vertical Skills in Retail)

本节是本白皮书的实战演练案例。它详细梳理了一家大型零售商将如何构建其技能库，以捕捉那些让自身品牌在竞争中脱颖而出的机构知识；并阐明了为什么是这个技能库（而非任何特定的定制智能体）才是企业最具持久生命力的战略资产。几家主要的零售巨头此前已公开描述过与其下方架构相匹配的 AI 助手。我们在此描述的是普适范式，而非单一供应商的具体实现。

为什么零售业是应用技能的典型教科书案例？

Why retail is the canonical case for skills

两家运行在完全相同的通用智能体运行时上、通过相同的 API 访问相似数据的零售商，往往会创造出截然不同的购物体验。零售领域的专业知识在历史上通常被困在三个 AI 系统难以触及的地方：在资深买手、商品策划师和品类经理的大脑里；在长达 30 页且根本没人阅读的运营手册里；以及在 Slack 频道的历史线索里（例如“我们是否应该将这款产品与那款产品搭配推荐？”的答案，可能正躺在 2023 年的一条聊天记录中）。

而技能，将这些知识转化为公司面向客户的系统能够真正调用的形式。

整体架构图景

What the architecture looks like

零售层面的技术架构分为三层：

  • 顶层为客户交互界面：包括网站聊天、移动端 App、店内自助终端、呼叫中心的语音智能体。每个界面都很轻量，只负责转发用户输入并渲染响应结果。
  • 中层为智能体运行时与编排器：负责维护对话、加载技能、调用工具并组装最终回复。
  • 底层为数据与工具平面：包含商品目录（通常达数百万 SKU）、带有门店级定位数据的实时库存、客户画像与订单历史、项目知识库，以及针对评论、手册和规格说明书的向量搜索。

对于本白皮书而言，核心关键在中层。运行时本身是通用的（如 Google 的 Agent Development Kit 或 Anthropic 的 Claude Agent SDK），但加载到该运行时中的技能则是该零售商特有的领域资产，正是它们将品牌的专业知识传递给了客户。

一个直观的技能库示例

An illustrative skills library

这样一个技能库究竟包含什么？以下是一家大型家居建材零售商可以维护的一套代表性技能：

  • project-guidance（项目指导）：编码了行业专家的知识，能将一个模糊的咨询（如“我如何给淋浴间贴瓷砖？”）转化为一套分步计划。包含结构依赖（贴砖前基底必须做防水）、订购逻辑（安装前先裁切）和常见错误提醒。由行业知识团队拥有。只读层级。
  • materials-list（物料清单）：接收一段项目描述（语音、文本或部分清单），并产出一份分组的物料清单，其中包含了承包商极易遗忘的零碎配件。由 Pro 商家商品管理团队拥有。仅限草稿层级：客户在购买前需亲自确认。
  • review-summarize（评论总结）：将长篇的产品评论冷凝浓缩为优缺点和常见使用场景。当客户询问某款产品的真实使用体验时触发。由个性化团队拥有。只读层级。
  • delivery-window（送货时效）：结合客户位置、门店库存以及公司的货运网络，计算最后一公里的配送方案与预估到达时间（ETA）。由履约团队拥有。只读层级。
  • return-policy（退货政策）：编码了公司的退货规则，包括针对特殊定制商品、危险品、定制裁剪件以及合同协议价的数十种豁免条款。由客户服务团队拥有。只读层级。若要晋升到行动敏感型（例如直接发起退款），则需要引入第二项审核严格得多的专属技能。

每一个技能都足够微小，确保了其能够被独立地评审、测试和交付。

相同的查询如何在技能库中进行路由

How does the same query route through the library

假设一名客户询问：“我想改造我儿女的浴室，都需要准备些什么？”

运行时在会话开始时会先加载所有技能的 L1 描述（每个约占用 ~30 到 80 词元，总计约 2KB）。它识别出 project-guidance 技能匹配，随即加载其正文，产出装修步骤的大纲。如果客户后续跟进询问送货问题，delivery-window 技能将被加载。如果他们询问特定商品的退货政策，return-policy 将被加载。随着对话的移动，先前的技能可以保留在上下文中，也可以被直接释放。

注意这里并没有发生什么：并没有一个被训练去应对所有可能装修场景的、庞大臃肿的“装修助手”单体智能体。每一项专业能力只有在对话触及它时，才会被按需加载到上下文中。活跃上下文始终保持精简，而可用的能力在事实上却趋于无限。

为什么这比单搞一个定制智能体更具竞争壁垒？

Why is this harder to compete with than a custom agent

如果竞争对手和我们这家零售商运行在完全相同的通用智能体运行时上（事实上确实如此，因为运行时已经完全大宗商品化），那么跟进复制运行时是微不足道的。而技能库，则编码了这家公司多年来积累沉淀的业务范式。这是本白皮书的核心战略观点。

一家在定制智能体上砸下重金却忽视了技能库建设的零售商，投资的恰恰是竞争对手可以免费获取的轻量化技术栈。相反，一家专注于投资技能的零售商，即便基于最通用的运行时，也在构建一项能够真正捕获并留存公司核心 know-how 的持久战略资产。

Q: 如上文所言，这些技能库只是一堆md文件包而已，实际上是不是很容易泄漏和复制，导致核心能力被竞争对手复制。

A：

你的直觉非常敏锐，切中了当前大模型工程落地中最核心的商业机密与资产保护痛点。

从表面上看，这些技能库确实只是一堆 md 文件（包含自然语言描述的 System Prompt、工具调用说明、Few-shot 示例等）和一些配套的轻量脚本。单看这些文件本身，它们确实极易被复制、泄漏或在离线环境下被瞬间打包带走。

但是，这段话所强调的“无法被竞争对手免费获取的核心 know-how”，其核心壁垒并不在纯文本的 md 文件本身，而在隐藏在文件背后的工程生态和业务数据闭环。

我们可以从以下三个维度来理解为什么“技能库”能成为带不走的持久战略资产：

  1. 技能库的真正壁垒是背后的“私有 API 与数据资产”

md 文件里的描述只是智能体的“说明书”，它必须调用零售商后台的物理基础设施。

  • 空壳与实体的区别：竞争对手即使偷走了你的 md 技能描述（比如表 5 中的 delivery-window 配送窗口技能），里面写满了逻辑规则，但如果他们没有你实时更新的仓储 ERP 接口、供应链物流预测算法、本地门店履约网络的动态数据，这个技能在他们的运行时里就是一个无法执行的“空壳”。
  • 逻辑左移的保护：正如前面“智能左移”所言，高阶的技能会将核心商业逻辑推向后台的标准微服务代码和数据库（如定制的 R 语言优化求解器、排班算法）。大模型看到的只是一个工具接口（Schema）。对手抄走了 Prompt，抄不走你后台沉淀了十几年的企业信息化系统。
  1. 技能库是由“测试流”和“黄金数据集”喂出来的

零售业的业务场景极其复杂。一份能在线上稳定跑在 Action-allowed（行动层）的技能，背后经历过了无数次血与泪的重构。

  • 带不走的 Golden Dataset：要让一个技能在 99% 的情况下不吐出混沌的错误，开发者必须基于公司过去数万条真实的客户投诉、订单纠纷、商品类目冲突数据，提炼出一套私有的黄金数据集（Golden Dataset）。
  • 竞争对手拿到了你的 md 初稿，但只要他们的业务边界、商品类目稍微产生一点偏差，智能体就会在运行时迅速崩溃或触发幻觉。他们没有配套的离线评估体系（agenteval）和红队探测案例，就根本不具备对这份技能进行持续迭代、升级和维护的能力。
  1. 为什么说只砸模型是“为他人作嫁衣”？

这段话对零售商的警告在于“通道资产”与“核心资产”的对立：

  • 定制智能体（轻量化技术栈）：如果你把大笔的预算花在买最贵的商用大模型、搭最炫酷的 Agent UI 界面、或者是为某个通用的开源框架写了一堆运行时胶水代码——这些东西的同质化极其严重。随着基础模型（如 Gemini、GPT 系列）的快速迭代，这些外层的“定制智能体封装”会迅速贬值，成为竞争对手可以廉价购买或直接免费平替的“商品（Commodity）”。
  • 技能库（企业核心 know-how）：相反，把零售专家的隐性经验（品类管理规则、物料清单 BOM 提炼逻辑、门店运营护栏）格式化为一套体系化的技能。这套技能不管底层的大模型换成谁（今年用 Gemini，明年换成更便宜的开源模型），它都能立刻无缝无损地迁移过去。

总结

竞争对手确实可以轻易复制你的几行 Prompt 文本，但他们复制不了：

  1. 你的企业业务系统向这些技能开放的私有数据通道（Data Pipelines）。
  2. 你为了让这些技能安全上线而建立的、基于私有历史数据演进的自动化测试与治理天梯（Governance Ladder）。

因此，技能库不是资产的终点，它是企业核心业务逻辑的“锚点”。把重金砸在技能库的建设上，本质上是将不可控的大模型能力，锚定在公司带不走的业务护城河之内。

Q: 这么说其实核心的ERP也是最关键的核心竞争力资产之一。

A:

完全正确！你一针见血地指出了现代企业在 AI 时代的核心命脉：大模型和智能体绝不是空中楼阁，它们最坚实的底座和最核心的壁垒，恰恰是企业沉淀了几十年的 ERP 等底层核心业务系统。

在智能体（Agentic）技术栈的宏观视角下，我们可以通过以下三个层面来理解为什么核心 ERP 才是那座竞争对手根本无法复制的“护城河”：

  1. ERP 是大模型的“物理躯干”与“实体触手”

如果把大模型（LLM）比作一个拥有极高智商、能够理解复杂语义的“大脑”，那么如果没有底层业务系统的支撑，它就是一个“高位截瘫”的大脑——只能聊天，什么也干不了。

  • 没有 ERP 的智能体：它读完你那堆高水准的 md 技能描述后，只能回复用户：“我知道您想申请淋浴间瓷砖的退款，根据规则您符合条件，但我现在无能为力，请您去拨打人工客服……” 这种智能体毫无商业价值。
  • 接入 ERP 的智能体：它在收到指令后，能够真正调用 ERP 的 财务模块接口 发起原路退款、调用 仓储模块（WMS） 冻结受损货品、调用 供应链模块 重新向供应商下单补货。
  • 结论：智能体展现出来的“强大执行力（Action-allowed）”，其本质上不过是底层 ERP 系统多年信息化积淀能力的延伸和外显。
  1. ERP 里的数据提供了 AI 最稀缺的“真实世界上下文”

大模型在通用知识上是完美的，但它对你的企业一无所知。要想让它在你的业务场景里做出正确的决策，必须依赖 ERP 提供的实时、精准的行业和企业上下文（Context）：

  • 比如表 5 中提到的 delivery-window（配送窗口技能）。为什么这个技能是最有价值的资产？因为 ERP 系统中记录了当前具体某家门店的真实库存、当地卡车司机的排班计划、此时此刻真实的道路拥堵和运力限制。
  • 竞争对手可以用开源技术迅速搭建起一套一模一样的多智能体编排流水线（DAG），但只要他们没有你 ERP 里面那套经过十几年业务洗礼、高度清洗且保持秒级更新的私有数据流（Data Pipelines），他们的智能体做出的预测和调度就是一纸空文。
  1. “智能左移”的终点，就是将复杂的业务规则固化在 ERP 层

前面提到的“智能左移（Shift Intelligence Left）”最佳实践指出：不要让大模型在运行时去猜复杂的、确定性的业务规则。

那么，这些被无情地赶出大模型 Prompt 的复杂业务逻辑（比如：“当 A 商品库存低于 10% 且 B 供应商延迟交货超过 3 天时，自动触发 C 预警”），应该写在哪里？

  • 答案就是：固化在你的微服务系统、你的中间件、以及你的 核心 ERP 业务代码 之中。
  • 把规则写在 Prompt 里，别人复制走你的 md 文件，你的优势就没了；
  • 把规则写在 ERP 后台的自动化脚本和微服务逻辑里，别人只能看到智能体暴露出来的一个空荡荡的 API 接口定义（Schema）。他们不仅看不到底层的核心算法（Know-how），更无法跨越你多年积累下来的系统复杂性和工程壁垒。

总结

在智能体时代的商业竞争中，业界正在达成一个颠扑不破的共识：

“基础大模型会迅速商品化、廉价化；外层的 Agent 胶水框架会高度同质化；唯有企业私有的核心业务系统（ERP/CRM/SCM）以及其中流淌的真实高质数据，才是那项真正捕获并留存公司核心 know-how 的、带不走的持久战略资产。”

重金砸在定制智能体上，是在投资别人可以免费获取的技术栈；而专注于利用智能体去激活、打通、和重塑你原有的核心 ERP 资产，才是真正构建起了别人跨不过去的企业护城河。
