关于误删除数据的案例分析

发表于 2017-12-13 | 分类于 技术相关

公司案例

背景

两个系统的接口，在同步插入数据的时候需要删除对应项目的历史数据。

场景

新接口通过测试上线之后，发现所有项目的数据都没有了。

处理过程

1、怀疑是新的接口导致。但是开发工程师声称删除的语句是加了where条件的，只会删除对应项目的历史数据；
2、找一台测试机，用Sql server Profiler重现接口的过程，发现有很多delete语句误删了其它项目的数据；

分析

原来where语句里面太多条件，有一个or条件没有括号，导致变成所有的项目了。

反思

1、为什么通过测试的工具还会出问题？

在测试的过程当中，测试人员同步一个项目，然后检查这个项目的数据同步成功了，就测试通过，没有考虑过这个工具有删除别的项目数据的风险；

2、如何避免这个问题的重现

1）测试过程对于有delete update的语句，需要单独测试这一部分（单元测试的概念），而不只是看结果；
2）开发的过程中，要养成好的开发习惯，例如

在sql中删除数据库中记录我们会使用到delete命令，这样如果不小心给删除了很难恢复了，总结一些删除数据但是不在数据库删除的方法。
方法一
我常用的做法，就是在数据库中加一个删除标识字段，
如: isdel=1 这样就代码删除的字段了
方法二
直接限制mysql删除
启动MySQL的时候加上参数 -U
-U, —safe-updates Only allow UPDATE and DELETE that uses keys.
防止执行delete的时候没带有条件语句，如果没加上where则语句不执行。

经验之谈

偶尔要做 DB 维护，这是一点点经验。
不要走神。不听歌，不聊天，不吃东西，不喝饮料，头脑清醒，知道自己在做什么，要做什么。
写出步骤。连接到哪个数据库，ip是什么，运行什么命令，先做什么，后做什么，出了问题怎么roll back，我知道你都懂，但要写出来，不要相信自己的记忆。
在测试环境验证。拿来写好的步骤，在测试环境中跑一遍，一半以上的可能会发现问题，然后再修改步骤，不要直接在产品环境中跑。
delete 和 update 前，先查询，用同样的 where 语句 select，至少知道有多少记录会被影响到。
drop 和 truncate 之前，检查三遍，连接的是不是正确的数据库。
一次只连接一个DB，不要开几个窗口，有的连测试，有的连产品，或早或晚，你会出错。
